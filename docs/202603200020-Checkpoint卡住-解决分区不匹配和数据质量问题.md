# Checkpoint 卡住问题 - 解决分区不匹配和数据质量问题

**时间**: 2026-03-20 00:20  
**问题**: Flink DWD SQL 作业在 Checkpoint 时卡住，数据无法写入 Doris  
**状态**: 已解决

---

## 问题现象

1. **Checkpoint 卡住**: 作业在触发 Checkpoint 后停止响应，日志显示 "Triggering checkpoint 1" 后没有后续日志
2. **数据不写入**: DWD 表数据量保持在 32,217 条，没有新数据写入
3. **Stream Load 启动但未完成**: 日志显示 Stream Load 启动，但一直没有完成

## 问题分析

### 根本原因

通过测试和日志分析，发现了问题的根本原因：

**`no partition for this tuple` - 分区不匹配错误**

1. **Flink SQL 日期计算问题**: 
   - Flink SQL 中的 `TO_TIMESTAMP(FROM_UNIXTIME(ts / 1000))` 可能因为时区问题导致日期计算错误
   - 计算出的 `trade_date` 可能不在 Doris 表的分区范围内

2. **数据质量检查失败**:
   - 当数据的 `trade_date` 不在分区范围内时，Doris Stream Load 会过滤这些数据
   - 如果过滤率超过 `max_filter_ratio`（之前设置为 0.1，即 10%），Stream Load 会失败
   - 失败的 Stream Load 会导致 Checkpoint 一直等待，最终卡住

3. **Checkpoint 机制**:
   - Doris Connector 在 Checkpoint 时会 flush 所有缓冲的数据
   - 如果 Stream Load 失败，Checkpoint 会一直等待，导致作业卡住

## 解决方案

### 方案 1: 设置时区（推荐）

在 Flink Table Environment 中明确设置时区为中国时区（东八区）：

```java
// 设置时区为中国时区（东八区）- 重要：确保日期计算正确
tableEnv.getConfig().setLocalTimeZone(java.time.ZoneId.of("Asia/Shanghai"));
```

**优点**: 从根本上解决日期计算问题  
**缺点**: 需要重新编译代码

### 方案 2: 提高数据过滤容忍度

将 `max_filter_ratio` 从 0.1（10%）提高到 0.5（50%）：

```java
"'sink.properties.max_filter_ratio' = '0.5'\n" +  // 允许 50% 的数据过滤
```

**优点**: 即使有部分数据分区不匹配，Stream Load 也能成功  
**缺点**: 可能会丢失一些数据

### 方案 3: 增加重试次数

将 `sink.max-retries` 从 3 增加到 5：

```java
"'sink.max-retries' = '5',\n" +  // 增加重试次数
```

**优点**: 提高容错能力  
**缺点**: 不能解决根本问题

### 方案 4: 调整 Checkpoint 配置

增加 Checkpoint 间隔和超时时间：

```yaml
flink:
  checkpoint:
    interval: 60000  # 60 秒（从 30 秒增加到 60 秒）
    timeout: 300000  # 5 分钟（从 10 分钟减少到 5 分钟）
```

**优点**: 减少 Checkpoint 频率，降低失败概率  
**缺点**: 增加数据丢失风险

## 关键经验

1. **分区问题是隐藏的杀手**: 
   - Doris 的分区表如果数据不在分区范围内，会被过滤
   - 过滤率超过 `max_filter_ratio` 会导致 Stream Load 失败
   - 失败的 Stream Load 会导致 Checkpoint 卡住

2. **时区设置很重要**:
   - Flink SQL 的日期函数受时区影响
   - 必须明确设置时区，避免日期计算错误

3. **数据质量配置需要平衡**:
   - `max_filter_ratio` 太低：容易因为少量错误数据导致失败
   - `max_filter_ratio` 太高：可能丢失大量数据而不知道
   - 建议：开发环境 0.5，生产环境 0.1

4. **Checkpoint 配置需要调优**:
   - `interval` 太短：频繁 Checkpoint，增加失败概率
   - `interval` 太长：数据丢失风险增加
   - 建议：根据数据量和业务需求调整，一般 30-60 秒

5. **监控和日志很重要**:
   - 定期检查 Stream Load 任务状态
   - 监控数据写入速率
   - 查看 Checkpoint 完成情况
