## 问题分析

### 排查过程

1. **检查数据采集器** ✅
   - 数据采集器正常运行
   - 持续发送数据到 Kafka（Offset 持续增长）
   - 每秒发送约 5-10 条 BTC-USDT 行情数据

2. **检查 Kafka 状态** ✅
   - Kafka 容器正常运行
   - Topic `crypto_ticker` 存在
   - 分区 2 有大量数据（Offset > 70000）

3. **检查 Flink 作业** ⚠️
   - Flink 作业显示 RUNNING 状态
   - 但 Kafka 消费者组显示 "has no active members"
   - Kafka LAG 持续增加（从 98 到 247）

4. **检查 Doris 数据** ⚠️
   - 表数据量在增长（从 53,686 到 193,199）
   - 但最新时间停留在 17:30（旧数据）
   - 说明 Flink 在处理历史数据，但不消费新数据

### 根本原因 ⭐⭐⭐

**Kafka Source 配置使用了 `OffsetsInitializer.latest()` 消费模式**

```java
// 问题代码
KafkaSource.<String>builder()
    .setStartingOffsets(OffsetsInitializer.latest())  // ❌ 只消费新数据
    .build();
```

**问题说明**:
- `latest()` 模式：只消费 Flink 作业启动**之后**产生的新数据
- 数据采集器在 Flink 作业启动**之前**就已经运行
- Kafka 中已经有大量历史数据（Offset 0 → 70000+）
- Flink 启动后，从 latest offset 开始消费，但此时没有新数据产生
- 导致 Flink 一直等待新数据，Records Received = 0

**时间线**:
```
17:00 - 数据采集器启动，发送数据到 Kafka (Offset 0 → 50000)
17:30 - 数据采集器继续运行 (Offset 50000 → 70000)
01:00 - Flink 作业启动，使用 latest() 模式
01:00 - Flink 从 Offset 70000 开始消费（跳过所有历史数据）
01:00 - 此时数据采集器暂停，没有新数据产生
01:00 - Flink 等待新数据，Records Received = 0 ❌
```

## 解决方案

### 方案 1: 修改消费模式为 earliest（推荐）✅

**修改代码**，添加可配置的消费模式：

```java
/**
 * 创建 Kafka Source
 * 
 * 消费模式说明：
 * - earliest: 从最早的数据开始消费（适合处理历史数据）
 * - latest: 从最新的数据开始消费（适合实时处理）
 * - committed: 从上次提交的 offset 开始消费
 */
private static KafkaSource<String> createKafkaSource(ConfigLoader config) {
    // 读取消费模式配置，默认使用 earliest（从头开始消费）
    String startupMode = config.getString("kafka.consumer.startup-mode", "earliest");
    
    // 根据配置选择消费模式
    OffsetsInitializer offsetsInitializer;
    switch (startupMode.toLowerCase()) {
        case "latest":
            offsetsInitializer = OffsetsInitializer.latest();
            System.out.println("  Startup Mode: latest（从最新数据开始）");
            break;
        case "committed":
            offsetsInitializer = OffsetsInitializer.committedOffsets();
            System.out.println("  Startup Mode: committed（从上次提交的 offset 开始）");
            break;
        case "earliest":
        default:
            offsetsInitializer = OffsetsInitializer.earliest();
            System.out.println("  Startup Mode: earliest（从最早数据开始）");
            break;
    }
    
    return KafkaSource.<String>builder()
        .setBootstrapServers(config.getString("kafka.bootstrap-servers"))
        .setTopics(config.getString("kafka.topic.crypto-ticker"))
        .setGroupId(config.getString("kafka.consumer.group-id", "flink-ods-consumer"))
        .setStartingOffsets(offsetsInitializer)  // ✅ 使用配置的消费模式
        .setValueOnlyDeserializer(new SimpleStringSchema())
        .build();
}
```

**添加配置** (`config/application-dev.yml`):

```yaml
kafka:
  bootstrap-servers: localhost:9092
  consumer:
    startup-mode: earliest  # ✅ 从头开始消费
```

### 方案 2: 使用 committed 模式（生产环境推荐）

```yaml
kafka:
  consumer:
    startup-mode: committed  # 从上次提交的 offset 开始
```

**优点**:
- 避免重复消费历史数据
- 支持作业重启后继续消费
- 适合生产环境

**前提**:
- 消费者组必须已经提交过 offset
- 首次启动时会从 earliest 开始

## 技术要点

### Kafka 消费模式对比

| 模式 | 说明 | 适用场景 | 优点 | 缺点 |
|-----|------|---------|------|------|
| **earliest** | 从最早的数据开始 | 开发环境、数据回溯 | 不丢数据、可处理历史数据 | 首次启动慢、可能重复消费 |
| **latest** | 从最新的数据开始 | 实时处理、不关心历史 | 启动快、无重复消费 | 丢失历史数据 |
| **committed** | 从上次提交位置开始 | 生产环境 | 支持断点续传、不重复消费 | 首次启动需要 offset |

### 消费模式选择建议

**开发环境**:
```yaml
kafka:
  consumer:
    startup-mode: earliest  # 方便调试，可以看到所有数据
```

**生产环境**:
```yaml
kafka:
  consumer:
    startup-mode: committed  # 支持断点续传，避免重复消费
```

**实时监控场景**:
```yaml
kafka:
  consumer:
    startup-mode: latest  # 只关心新数据，不处理历史
```

### Flink Kafka Source 配置

```java
KafkaSource.<String>builder()
    .setBootstrapServers("localhost:9092")
    .setTopics("crypto_ticker")
    .setGroupId("flink-ods-consumer")
    
    // 消费模式配置（关键）
    .setStartingOffsets(OffsetsInitializer.earliest())  // 从头开始
    // .setStartingOffsets(OffsetsInitializer.latest())  // 从最新开始
    // .setStartingOffsets(OffsetsInitializer.committedOffsets())  // 从提交位置开始
    
    .setValueOnlyDeserializer(new SimpleStringSchema())
    .build();
```

## 相关问题

### 问题 1: 为什么 Doris 表数据在增长？

**原因**: Flink 作业之前运行过，消费了部分历史数据并写入 Doris。虽然当前作业使用 `latest()` 模式不消费新数据，但旧数据已经在表中了。

### 问题 2: 为什么 Kafka 消费者组显示 "has no active members"？

**原因**: Flink Kafka Source 使用的是新的 Consumer API，不会在消费者组中显示为 active member。这是正常现象，不影响消费。

### 问题 3: 为什么 Grafana 监控显示无数据？

**原因**: 
1. Flink 没有消费新数据，`numRecordsIn = 0`
2. Metrics Reporter 配置错误（已在问题 15 中修复）

### 问题 4: 如何避免重复消费？

**方案**:
1. 使用 `committed` 模式
2. 启用 Flink Checkpoint（已启用，60 秒间隔）
3. Checkpoint 会自动提交 Kafka offset

## 经验总结

### 1. Kafka 消费模式很重要 ⭐⭐⭐

- **默认使用 `earliest` 模式**，避免丢失数据
- 生产环境使用 `committed` 模式，支持断点续传
- 明确理解不同模式的行为差异

### 2. 数据流监控很重要

- 监控 Kafka Offset 和 LAG
- 监控 Flink 消费进度
- 监控 Doris 数据时间戳
- 三者结合才能发现问题

### 3. 配置要可调整

- 不要硬编码消费模式
- 使用配置文件管理
- 方便不同环境使用不同配置

### 4. 日志输出要清晰

- 输出关键配置信息（消费模式、并行度等）
- 方便排查问题
- 使用 System.out.println 确保输出可见
