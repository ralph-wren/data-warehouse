# Flink Web UI 显示问题 - 本地模式指标显示为 0

**时间**: 2026-03-20 01:45  
**问题类型**: Flink Web UI 显示  
**严重程度**: 低（不影响实际功能）  
**状态**: ✅ 已解决（已知问题，提供解决方案）

---

## 问题描述

### 现象

1. **Flink 作业正常运行**，数据正常消费和入库
2. **Kafka LAG = 0**，说明数据被完全消费
3. **Doris 表数据持续增长**，最新时间实时更新
4. **但 Flink Web UI 的 Overview 页面显示所有指标都是 0**：
   - Bytes Received: 0
   - Records Received: 0
   - Bytes Sent: 0
   - Records Sent: 0
5. **Grafana 监控也显示没有数据**

### 用户报告

> "FlinkODSJobDataStream 我启动这个程序，在正常消费 kafka 数据，正常入库，为什么 flink web 显示没接收数据呢"

### 截图证据

- Flink Web UI Overview 页面：所有指标显示 0
- Grafana Dashboard：图表显示无数据
- 但实际数据在持续流入 Doris

---

## 问题分析

### 排查过程

1. **检查 Pushgateway 指标** ⚠️
   ```bash
   curl -s http://localhost:9091/metrics | grep "numRecordsIn"
   # 结果：只有旧作业的指标，且值都是 0
   ```

2. **检查 Flink 作业状态** ✅
   ```bash
   curl -s http://localhost:8081/jobs
   # 结果：作业正常运行（RUNNING）
   ```

3. **检查 Kafka 消费状态** ✅
   ```bash
   docker exec kafka kafka-consumer-groups --describe --group flink-ods-consumer
   # 结果：LAG = 0，数据被完全消费
   ```

4. **检查 Doris 数据** ✅
   ```sql
   SELECT COUNT(*), MAX(timestamp) FROM ods_crypto_ticker_rt;
   # 结果：数据持续增长，时间实时更新
   ```

5. **检查 Flink REST API 指标** ✅ ⭐
   ```bash
   curl -s "http://localhost:8081/jobs/$JOB_ID/vertices/$VERTEX_ID/subtasks/0/metrics?get=Source__Kafka_Source.numRecordsIn"
   # 结果：[{'id': 'Source__Kafka_Source.numRecordsIn', 'value': '72154'}]
   ```

### 根本原因 ⭐⭐⭐

**Flink 本地模式（LocalStreamEnvironment）的 Web UI 指标显示 bug**

1. **Flink REST API 有正确的指标值**（72154 条记录）
2. **但 Web UI 的 Overview 页面显示为 0**
3. **这是 Flink 本地模式的已知问题**

**技术细节**:
- Flink 本地模式使用 `LocalStreamEnvironment`
- Web UI 的 Overview 页面可能显示的是**聚合指标**
- 在本地模式下，这些聚合指标的计算可能有 bug
- 或者显示的是某个特定 subtask 的指标（恰好是 subtask 1，值为 0）

**验证**:
```bash
# Subtask 0 的指标（处理了所有数据）
Source__Kafka_Source.numRecordsIn: 72154
ODS_Transform.numRecordsIn: 72154
Doris_ODS_Sink__Writer.numRecordsIn: 72154

# Subtask 1 的指标（没有处理数据）
Source__Kafka_Source.numRecordsIn: 0
ODS_Transform.numRecordsIn: 0
Doris_ODS_Sink__Writer.numRecordsIn: 0
```

**原因分析**:
- Kafka 只有一个分区（分区 2）有数据
- Flink 并行度为 2，有 2 个 subtask
- Subtask 0 处理了分区 2 的所有数据
- Subtask 1 没有分配到任何分区，所以指标为 0
- Web UI 可能显示的是 Subtask 1 的指标，所以显示为 0

## 解决方案

### 方案 1: 使用 Flink REST API 查看真实指标（推荐）✅

**创建指标检查脚本** (`test/check-flink-metrics.sh`):

```bash
#!/bin/bash

# 获取 Job ID
JOB_ID=$(curl -s http://localhost:8081/jobs | python -c "import sys, json; jobs = json.load(sys.stdin)['jobs']; print(jobs[0]['id']) if jobs else print('')")

# 获取 Vertex ID
VERTEX_ID=$(curl -s "http://localhost:8081/jobs/$JOB_ID" | python -c "import sys, json; data = json.load(sys.stdin); print(data['vertices'][0]['id']) if data.get('vertices') else print('')")

# 获取指标
curl -s "http://localhost:8081/jobs/$JOB_ID/vertices/$VERTEX_ID/subtasks/0/metrics?get=Source__Kafka_Source.numRecordsIn" | python -c "import sys, json; metrics = json.load(sys.stdin); [print(f'{m[\"id\"]}: {m[\"value\"]}') for m in metrics]"
```

**使用方法**:
```bash
bash test/check-flink-metrics.sh
```

5. 查看 "SubTasks" 标签页

**优点**:
- 可以看到每个 Operator 的详细指标
- 可以看到每个 Subtask 的指标
- 更直观，有图表展示

### 方案 3: 使用 Prometheus + Grafana 监控（生产环境推荐）✅

**配置 Prometheus Reporter**:
```java
MetricsConfig.configurePushgatewayReporter(
    flinkConfig,
    "localhost",

```

**优点**:
- 持久化指标数据
- 支持历史查询和告警
- 更适合生产环境

**注意**: 
- Pushgateway Reporter 在本地模式下可能也有延迟
- 需要等待 15-30 秒才能看到指标更新

**部署 Flink Standalone 集群**:

**优点**:

## 技术要点

### Flink 本地模式 vs Standalone 模式

| 特性 | 本地模式 | Standalone 模式 |
|-----|---------|----------------|
| **部署** | 简单，直接运行 | 需要启动集群 |
| **Web UI** | 有，但指标可能不准 | 完整功能 |
| **调试** | 方便，IDE 直接运行 | 需要打包提交 |
| **生产使用** | ❌ 不推荐 | ✅ 推荐 |
| **指标准确性** | ⚠️ 可能有问题 | ✅ 准确 |

### Flink Metrics 层次结构

```
Job
└── Vertex (Operator Chain)
    └── Subtask (Parallel Instance)
        └── Metrics
            ├── numRecordsIn
            ├── numRecordsOut
            ├── numBytesIn
            └── numBytesOut
```

### 如何正确查看 Flink 指标

1. **不要只看 Overview 页面**
   - Overview 页面的指标在本地模式下可能不准确
   - 应该查看具体的 Vertex 和 Subtask 指标

2. **使用 REST API**
   - REST API 提供最准确的指标
   - 可以编写脚本自动化监控

3. **查看 Subtask 级别的指标**
   - 每个 Subtask 可能处理不同数量的数据
   - 需要查看所有 Subtask 的指标并求和

4. **对比多个数据源**
   - Kafka 消费位置
   - Doris 数据量
   - Flink 指标
   - 三者应该一致

## 常见问题

### Q1: 为什么 Subtask 1 的指标是 0？

**A**: 因为 Kafka 只有一个分区（分区 2）有数据，而 Flink 并行度为 2。Kafka Source 会将分区分配给不同的 Subtask，Subtask 0 分配到了分区 2，Subtask 1 没有分配到任何分区。

**解决方案**:
- 增加 Kafka 分区数（推荐）
- 或者减少 Flink 并行度到 1

### Q2: 为什么 Pushgateway 中的指标是 0？

**A**: Pushgateway 中的指标可能是旧作业的，或者新作业的指标还没有推送。Pushgateway Reporter 有推送间隔（默认 15 秒），需要等待一段时间。

### Q3: 为什么 Grafana 显示没有数据？

**A**: Grafana 从 Prometheus 读取数据，Prometheus 从 Pushgateway 抓取数据。如果 Pushgateway 中没有数据，Grafana 就显示不出来。

### Q4: 生产环境应该怎么部署？

**A**: 生产环境推荐使用 Flink Standalone 集群或 Flink on YARN/Kubernetes，不要使用本地模式。

## 经验总结

### 1. Flink 本地模式的局限性 ⭐⭐⭐

- **适合开发调试**，不适合生产使用
- **Web UI 指标可能不准确**，需要使用 REST API 验证
- **Metrics Reporter 可能不工作**，需要额外配置

### 2. 如何验证数据流是否正常

不要只看 Flink Web UI，应该综合多个指标：
1. ✅ Kafka LAG = 0（数据被消费）
2. ✅ Doris 数据持续增长（数据被写入）
3. ✅ Flink REST API 指标 > 0（数据被处理）
4. ✅ 数据时间戳实时更新（不是旧数据）

### 3. 监控的重要性

- 不要依赖单一监控源
- 使用多种方式验证（Web UI、REST API、Kafka、Doris）
- 编写自动化监控脚本

### 4. 开发环境 vs 生产环境

| 环境 | 模式 | 监控方式 |
|-----|------|---------|
| **开发** | 本地模式 | REST API + 脚本 |
| **测试** | Standalone | Web UI + Prometheus |
| **生产** | YARN/K8s | Prometheus + Grafana + 告警 |
