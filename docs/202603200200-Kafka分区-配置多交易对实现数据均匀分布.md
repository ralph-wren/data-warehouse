## 问题描述

### 需求

用户有 4 个 Kafka 分区，希望数据能够均匀分布到不同分区，以便：
1. 提高 Flink 并行处理能力
2. 让所有 Flink subtask 都能处理数据
3. 提高整体吞吐量

### 当前状况

- Kafka Topic `crypto_ticker` 有 4 个分区
- 数据采集器只订阅 1 个交易对（BTC-USDT）
- 所有数据都发送到同一个分区（分区 2）
- Flink 并行度为 2，但只有 1 个 subtask 在工作

---

## 问题分析

### Kafka 分区机制

Kafka Producer 使用以下规则决定消息发送到哪个分区：

1. **如果指定了分区号**：直接发送到指定分区
2. **如果提供了 key**：使用 `hash(key) % partition_count` 计算分区
3. **如果没有 key**：使用轮询（round-robin）方式分配

### 当前实现

数据采集器使用交易对名称（`instId`）作为 key：

```java
// OKXWebSocketClient.java
kafkaProducer.send(instId, jsonData)
```

**问题**：
- 只订阅 1 个交易对（BTC-USDT）
- 所有消息的 key 都是 "BTC-USDT"
- `hash("BTC-USDT") % 4 = 2`，所有数据都发送到分区 2


## 解决方案

### 方案 1：订阅多个交易对（推荐）✅

**原理**：不同的交易对名称会 hash 到不同的分区

**实施步骤**：

#### 1. 修改数据采集器代码

修改 `DataCollectorMain.java` 的 `getSymbols` 方法：

```java
/**
 * 获取订阅的交易对列表
 * 
 * 注意：为了让数据均匀分布到 Kafka 的多个分区，建议订阅多个交易对
 * Kafka 使用 key（交易对名称）的 hash 值来决定分区，不同的交易对会分布到不同分区
 */
private static List<String> getSymbols(String[] args, ConfigLoader config) {
    // 如果命令行参数提供了交易对，使用命令行参数
    if (args.length > 0) {
        logger.info("Using symbols from command line arguments: {}", Arrays.toString(args));
        return Arrays.asList(args);
    }
    
    // 尝试从配置文件读取（支持逗号分隔的字符串）
    String symbolsConfig = config.getString("okx.symbols", "");
    if (!symbolsConfig.isEmpty()) {
        // 支持逗号分隔的多个交易对
        String[] symbolArray = symbolsConfig.split(",");
        List<String> symbols = new ArrayList<>();
        for (String symbol : symbolArray) {
            String trimmed = symbol.trim();
            if (!trimmed.isEmpty()) {
                symbols.add(trimmed);
            }
        }
        if (!symbols.isEmpty()) {
            logger.info("Using symbols from config: {}", symbols);
            return symbols;
        }
    }
    
    // 默认订阅 4 个主流交易对（对应 4 个 Kafka 分区）
    // 这样数据会均匀分布到不同分区
    List<String> defaultSymbols = Arrays.asList("BTC-USDT", "ETH-USDT", "SOL-USDT", "BNB-USDT");
    logger.info("Using default symbols: {}", defaultSymbols);
    return defaultSymbols;
}
```

### 方案 2：使用自定义分区器（高级）

**原理**：自定义分区逻辑，确保数据均匀分布

**实施步骤**：

#### 1. 创建自定义分区器

```java
package com.crypto.dw.kafka;

import org.apache.kafka.clients.producer.Partitioner;
import org.apache.kafka.common.Cluster;

import java.util.Map;

/**
 * 自定义 Kafka 分区器
 * 使用轮询（Round-Robin）方式均匀分配消息到各个分区
 */
public class RoundRobinPartitioner implements Partitioner {
    
    private int counter = 0;
    
    @Override
    public int partition(String topic, Object key, byte[] keyBytes, 
                        Object value, byte[] valueBytes, Cluster cluster) {
        // 获取分区数量
        int numPartitions = cluster.partitionCountForTopic(topic);
        
        // 轮询分配
        int partition = counter % numPartitions;
        counter++;
        
        return partition;
    }
    
    @Override
    public void close() {
        // 清理资源
    }
    
    @override
    public void configure(Map<String, ?> configs) {
        // 配置初始化
    }
}
```

#### 2. 配置 Producer 使用自定义分区器

```java
// KafkaProducerManager.java
props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, 
    "com.crypto.dw.kafka.RoundRobinPartitioner");
```

**优点**：
- ✅ 完全控制分区逻辑
- ✅ 可以实现完全均匀分布
- ✅ 适合单一交易对场景

**缺点**：
- ❌ 实现复杂
- ❌ 相同 key 的消息可能分布到不同分区（失去顺序保证）
- ❌ 不适合需要按 key 分组的场景

## 技术要点

### Kafka 分区策略

| 策略 | 说明 | 适用场景 |
|-----|------|---------|
| **默认分区器** | 使用 key 的 hash 值 | 需要保证相同 key 的消息顺序 |
| **轮询分区器** | 均匀分配到所有分区 | 不关心消息顺序，追求吞吐量 |
| **自定义分区器** | 自定义分区逻辑 | 特殊业务需求 |

### Hash 分区算法

```java
// Kafka 默认分区算法
int partition = hash(key) % numPartitions;

// 示例
hash("BTC-USDT") % 4 = 2  → 分区 2
hash("ETH-USDT") % 4 = 1  → 分区 1
hash("SOL-USDT") % 4 = 1  → 分区 1
hash("BNB-USDT") % 4 = 1  → 分区 1
```

### 分区数量建议

| 场景 | 分区数 | 说明 |
|-----|-------|------|
| **开发环境** | 2-4 | 方便调试 |
| **测试环境** | 4-8 | 接近生产 |
| **生产环境** | 8-16 | 根据吞吐量调整 |

**计算公式**：
```
分区数 = max(
    目标吞吐量 / 单分区吞吐量,
    消费者数量
)
```

### Flink 并行度配置

```yaml
flink:
  execution:
    parallelism: 4  # 应该 <= Kafka 分区数
```

**规则**：
- Flink 并行度 > Kafka 分区数：部分 subtask 空闲
- Flink 并行度 = Kafka 分区数：最佳配置
- Flink 并行度 < Kafka 分区数：部分 subtask 处理多个分区

## 常见问题

### Q1: 为什么订阅了 4 个交易对，但只有 2 个分区有数据？

**A**: 这是 Kafka 默认分区器的正常行为。Hash 算法不保证均匀分布，只保证相同 key 的消息发送到同一个分区。

**解决方案**：
- 订阅更多交易对（8-12 个）
- 或使用自定义分区器

### Q2: 如何确保所有分区都有数据？

**A**: 订阅足够多的交易对，或使用轮询分区器。

**经验值**：
- 4 个分区：订阅 8-12 个交易对
- 8 个分区：订阅 16-24 个交易对

### Q3: 分区不均匀会影响性能吗？

**A**: 会有一定影响：
- 数据倾斜：某些 subtask 处理更多数据
- 资源浪费：空闲的 subtask 不工作
- 吞吐量降低：整体处理速度受限于最慢的 subtask

**建议**：
- 开发环境：可以接受一定程度的不均匀
- 生产环境：应该追求均匀分布

### Q4: 可以动态调整分区数吗？

**A**: 可以，但有限制：
- 只能增加分区，不能减少
- 增加分区后，历史数据的分区不会改变
- 需要重启消费者才能感知新分区

```bash
# 增加分区到 8 个
docker exec kafka kafka-topics --bootstrap-server localhost:9092 \
    --alter --topic crypto_ticker --partitions 8
```

## 经验总结

### 1. 分区数量规划 ⭐⭐⭐

- **开始时不要设置太多分区**（4-8 个足够）
- **根据实际吞吐量调整**
- **预留一定余量**（2-3 倍）

### 2. 交易对选择

- **选择流动性好的交易对**（数据更新频繁）
- **覆盖不同类型**（主流币、山寨币、稳定币）
- **数量适中**（8-12 个）

### 3. 监控指标

- **分区数据量**：是否均匀
- **Flink subtask 处理量**：是否均衡
- **Kafka LAG**：是否有积压
- **吞吐量**：是否达到预期

### 4. 性能优化

- **Flink 并行度 = Kafka 分区数**
- **批量大小适中**（1000-5000）
- **压缩算法**（lz4 或 snappy）
