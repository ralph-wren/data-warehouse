# 监控过滤 - Grafana 面板只显示活跃 Job

## 问题描述

当前的 Grafana Flink 监控面板显示所有 job 的指标，包括已经 kill 的 job。这会导致：
- 面板中显示大量无用的历史数据
- 难以区分当前运行的作业
- 图表混乱，不利于监控

需要修改配置，只显示活跃的（运行中的）job。

## 问题分析

### Prometheus 指标标签

Flink 的 Prometheus 指标包含以下标签：
- `job_name`：作业名称
- `job_id`：作业 ID（唯一标识符）
- `task_name`：任务名称
- `tm_id`：TaskManager ID

**关键发现**：
- 运行中的 job 有 `job_id` 标签
- 已 kill 的 job 的指标会逐渐消失，但在 Prometheus 中可能还保留一段时间
- 通过过滤 `job_id!=""` 可以只显示有 job_id 的指标（即活跃的 job）

### 需要过滤的指标

以下指标需要添加 `job_id` 过滤：
1. 数据流入速率：`flink_taskmanager_job_task_numRecordsIn`
2. 数据流出速率：`flink_taskmanager_job_task_numRecordsOut`
3. Checkpoint 持续时间：`flink_jobmanager_job_lastCheckpointDuration`
4. Checkpoint 完成次数：`flink_jobmanager_job_numberOfCompletedCheckpoints`
5. Checkpoint 失败次数：`flink_jobmanager_job_numberOfFailedCheckpoints`
6. 缓冲池使用率：`flink_taskmanager_job_task_buffers_outPoolUsage`
7. 网络字节流入：`flink_taskmanager_job_task_numBytesIn`
8. 网络字节流出：`flink_taskmanager_job_task_numBytesOut`

**不需要过滤的指标**（集群级别）：
- `flink_jobmanager_numRunningJobs`：运行中的作业数
- `flink_jobmanager_numRegisteredTaskManagers`：TaskManager 数量
- `flink_jobmanager_taskSlotsAvailable`：可用任务槽
- `flink_jobmanager_taskSlotsTotal`：总任务槽
- `flink_taskmanager_Status_JVM_*`：JVM 指标（TaskManager 级别）

## 解决方案

### 1. 修改 Prometheus 查询表达式

在所有 job 相关的查询中添加 `{job_id!=""}` 过滤条件。

#### 修改前
```promql
rate(flink_taskmanager_job_task_numRecordsIn[1m])
```

#### 修改后
```promql
rate(flink_taskmanager_job_task_numRecordsIn{job_id!=""}[1m])
```

### 2. 具体修改内容

#### 面板 1：数据流入速率
```json
"targets": [{
  "expr": "rate(flink_taskmanager_job_task_numRecordsIn{job_id!=\"\"}[1m])",
  "legendFormat": "{{job_name}} - {{task_name}}",
  "refId": "A"
}]
```

#### 面板 2：数据流出速率
```json
"targets": [{
  "expr": "rate(flink_taskmanager_job_task_numRecordsOut{job_id!=\"\"}[1m])",
  "legendFormat": "{{job_name}} - {{task_name}}",
  "refId": "A"
}]
```

#### 面板 3：Checkpoint 持续时间
```json
"targets": [{
  "expr": "flink_jobmanager_job_lastCheckpointDuration{job_id!=\"\"}",
  "legendFormat": "{{job_name}}",
  "refId": "A"
}]
```

#### 面板 4：Checkpoint 统计
```json
"targets": [
  {
    "expr": "flink_jobmanager_job_numberOfCompletedCheckpoints{job_id!=\"\"}",
    "legendFormat": "完成 - {{job_name}}",
    "refId": "A"
  },
  {
    "expr": "flink_jobmanager_job_numberOfFailedCheckpoints{job_id!=\"\"}",
    "legendFormat": "失败 - {{job_name}}",
    "refId": "B"
  }
]
```

#### 面板 14：输出缓冲池使用率
```json
"targets": [{
  "expr": "flink_taskmanager_job_task_buffers_outPoolUsage{job_id!=\"\"}",
  "legendFormat": "{{job_name}} - {{task_name}}",
  "refId": "A"
}]
```

#### 面板 15：网络吞吐量
```json
"targets": [
  {
    "expr": "rate(flink_taskmanager_job_task_numBytesIn{job_id!=\"\"}[1m])",
    "legendFormat": "Bytes In - {{job_name}}",
    "refId": "A"
  },
  {
    "expr": "rate(flink_taskmanager_job_task_numBytesOut{job_id!=\"\"}[1m])",
    "legendFormat": "Bytes Out - {{job_name}}",
    "refId": "B"
  }
]
```

### 3. Prometheus 标签过滤语法

#### 基本语法
```promql
# 等于
metric{label="value"}

# 不等于
metric{label!="value"}

# 正则匹配
metric{label=~"regex"}

# 正则不匹配
metric{label!~"regex"}

# 多个条件（AND）
metric{label1="value1", label2="value2"}
```

#### 常用过滤示例
```promql
# 只显示特定作业
flink_taskmanager_job_task_numRecordsIn{job_name="Flink ODS Job"}

# 排除特定作业
flink_taskmanager_job_task_numRecordsIn{job_name!="Test Job"}

# 只显示有 job_id 的指标（活跃 job）
flink_taskmanager_job_task_numRecordsIn{job_id!=""}

# 使用正则匹配多个作业
flink_taskmanager_job_task_numRecordsIn{job_name=~"Flink.*Job"}
```

## 技术要点

### 1. Prometheus 标签过滤原理

Prometheus 使用标签（label）来区分不同的时间序列：
- 每个指标可以有多个标签
- 标签是键值对：`label_name="label_value"`
- 过滤条件在查询时应用，不影响数据存储

### 2. job_id 标签的生命周期

```
作业启动 → job_id 标签出现 → 指标开始推送
    ↓
作业运行 → job_id 标签存在 → 指标持续推送
    ↓
作业停止 → job_id 标签消失 → 指标停止推送
    ↓
Prometheus 保留期 → 旧数据过期 → 指标完全消失
```

### 3. 为什么使用 job_id!="" 而不是其他条件

| 过滤条件 | 说明 | 优缺点 |
|---------|------|--------|
| `{job_id!=""}` | job_id 不为空 | ✅ 简单有效，只显示活跃 job |
| `{job_name="xxx"}` | 指定作业名称 | ❌ 需要硬编码，不灵活 |
| `{job_name=~".*"}` | 正则匹配所有 | ❌ 仍会显示已 kill 的 job |
| 不过滤 | 显示所有 | ❌ 包含历史数据，混乱 |

### 4. Grafana 变量（可选优化）

可以添加 Grafana 变量，让用户选择要查看的作业：

```json
"templating": {
  "list": [
    {
      "name": "job_name",
      "type": "query",
      "datasource": "Prometheus",
      "query": "label_values(flink_taskmanager_job_task_numRecordsIn{job_id!=\"\"}, job_name)",
      "multi": true,
      "includeAll": true
    }
  ]
}
```

然后在查询中使用变量：
```promql
rate(flink_taskmanager_job_task_numRecordsIn{job_id!="", job_name=~"$job_name"}[1m])
```

## 优势

1. **清晰的监控视图**：只显示当前运行的作业，避免历史数据干扰
2. **自动更新**：作业启动/停止时，面板自动更新
3. **性能优化**：减少查询的数据量，提高 Grafana 响应速度
4. **易于维护**：不需要手动清理旧作业的指标

## 注意事项

### 1. Prometheus 数据保留期

Prometheus 默认保留 15 天的数据：
- 已 kill 的 job 的指标会在保留期后自动删除
- 如果需要长期保存历史数据，考虑使用 Thanos 或 Cortex

### 2. 指标消失的延迟

作业 kill 后，指标不会立即消失：
- Pushgateway 模式：指标会保留在 Pushgateway 中，直到手动删除
- Pull 模式：指标会在下次抓取时消失（通常 15-30 秒）

### 3. 清理 Pushgateway 中的旧指标

如果使用 Pushgateway，需要手动清理：
```bash
# 查看所有指标
curl http://localhost:9091/metrics

# 删除特定作业的指标
curl -X DELETE http://localhost:9091/metrics/job/flink-ods-job

# 删除所有指标
curl -X PUT http://localhost:9091/api/v1/admin/wipe
```

### 4. 集群级别指标不受影响

以下指标是集群级别的，不需要过滤：
- 运行中的作业数
- TaskManager 数量
- 任务槽统计
- JVM 指标（TaskManager 级别）

这些指标反映的是整个 Flink 集群的状态，而不是单个作业。

## 故障排查

### 问题 1：修改后面板显示为空

**原因**：可能没有运行中的作业

**解决**：
```bash
# 检查是否有运行中的作业
curl http://localhost:8081/jobs

# 启动一个作业
bash run-flink-ods-datastream.sh

# 等待 15 秒后查看面板
```

### 问题 2：Kill 作业后指标仍然显示

**原因**：Pushgateway 中的指标没有清理

**解决**：
```bash
# 清理 Pushgateway 中的指标
curl -X DELETE http://localhost:9091/metrics/job/flink-ods-job

# 或者重启 Pushgateway
docker-compose -f docker-compose-monitoring.yml restart pushgateway
```

### 问题 3：JSON 格式错误

**原因**：手动编辑时语法错误

**解决**：
```bash
# 恢复备份
cp monitoring/grafana/dashboards/flink-monitoring.json.backup2 \
   monitoring/grafana/dashboards/flink-monitoring.json

# 验证格式
python -m json.tool monitoring/grafana/dashboards/flink-monitoring.json
```
