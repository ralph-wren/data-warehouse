# Kafka JMX 详细监控实现

**时间**: 2026-03-20 23:20  
**问题**: 简化版 Kafka 监控只有基础指标，缺少消息速率、LAG、吞吐量等详细监控  
**状态**: ✅ 已完成

## 问题描述

当前使用的 `danielqsj/kafka-exporter` 只提供基础指标:
- ✅ Broker 数量
- ✅ Topic 数量和分区数
- ✅ 副本同步状态
- ✅ Leader 分布

缺少的关键指标:
- ❌ 消息流入/流出速率
- ❌ 字节吞吐量
- ❌ 请求延迟和性能
- ❌ 消费者 LAG
- ❌ JVM 和系统资源

## 解决方案

升级到 JMX Exporter 架构，通过 Kafka 的 JMX 接口获取完整的监控指标。

### 架构设计

```
Kafka (JMX enabled on port 9999)
    ↓
JMX Exporter (port 5556) ← 转换 JMX 指标为 Prometheus 格式
    ↓
Prometheus (port 9090) ← 抓取指标
    ↓
Grafana (port 3000) ← 可视化展示
```

### 双层监控策略

保留两个 Exporter 以获得最全面的监控:

1. **Kafka Exporter** (danielqsj/kafka-exporter)
   - 提供 Topic 级别的基础指标
   - 轻量级，无需配置
   - 端口: 9308

2. **JMX Exporter** (bitnami/jmx-exporter)
   - 提供详细的性能指标
   - 需要 Kafka 启用 JMX
   - 端口: 5556


## 实施步骤

### 1. 创建 JMX Exporter 配置文件

文件: `monitoring/jmx-exporter/kafka.yml`

配置了以下指标类别:
- **Broker Topic 指标**: 消息流入/流出速率、字节吞吐量
- **Replica Manager 指标**: 分区数、Leader 数、未充分复制的分区
- **Request 指标**: 请求延迟、队列时间、处理时间
- **Log Flush 指标**: 日志刷新性能
- **Controller 指标**: 控制器状态、Leader 选举
- **JVM 指标**: CPU、内存、GC

### 2. 更新 docker-compose-monitoring.yml

添加 JMX Exporter 服务:

```yaml
jmx-exporter:
  image: bitnami/jmx-exporter:latest
  container_name: kafka-jmx-exporter
  ports:
    - "5556:5556"
  volumes:
    - ./monitoring/jmx-exporter/kafka.yml:/etc/jmx-exporter/config.yml:ro
  command:
    - "5556"
    - "/etc/jmx-exporter/config.yml"
    - "host.docker.internal:9999"
```

### 3. 更新 Prometheus 配置

添加 JMX Exporter 抓取任务:

```yaml
- job_name: 'kafka-jmx'
  static_configs:
    - targets: ['jmx-exporter:5556']
```

### 4. 重启 Kafka 并启用 JMX

创建脚本 `restart-kafka-with-jmx.sh`:
- 停止现有 Kafka 容器
- 启动新容器并设置 JMX 环境变量
- 暴露 JMX 端口 9999
- 重新创建 Topic

关键环境变量:
```bash
KAFKA_JMX_OPTS="-Dcom.sun.management.jmxremote \
  -Dcom.sun.management.jmxremote.port=9999 \
  -Dcom.sun.management.jmxremote.authenticate=false \
  -Dcom.sun.management.jmxremote.ssl=false"
JMX_PORT=9999
```

### 5. 创建完整版监控面板

文件: `monitoring/grafana/dashboards/kafka-monitoring-advanced.json`

包含 20 个监控面板，分为 5 个类别:

**集群概览** (6 个面板):
- Broker 数量
- 分区总数
- Leader 分区数
- 未充分复制的分区
- 离线分区数
- 活跃控制器

**消息吞吐量** (4 个面板):
- 消息流入速率 (按 Topic)
- 字节吞吐量 (按 Topic)
- 总消息流入速率
- 总字节吞吐量

**请求性能** (4 个面板):
- 请求总时间 (Produce/Fetch)
- Produce 请求时间分解
- 请求速率 (按类型)
- 失败请求速率

**副本和分区** (2 个面板):
- crypto_ticker 同步副本数
- ISR 变化速率

**JVM 和系统资源** (4 个面板):
- CPU 使用率
- JVM 内存使用
- GC 频率
- GC 时间


## 使用方法

### 一键升级 (推荐)

```bash
# 运行升级脚本
./upgrade-kafka-monitoring.sh
```

脚本会自动完成:
1. 重启 Kafka 并启用 JMX
2. 重启监控服务
3. 验证服务状态
4. 测试 JMX 指标
5. 显示访问信息

### 手动升级

如果需要分步执行:

```bash
# 1. 重启 Kafka 并启用 JMX
./restart-kafka-with-jmx.sh

# 2. 重启监控服务
docker-compose -f docker-compose-monitoring.yml down
docker-compose -f docker-compose-monitoring.yml up -d

# 3. 等待服务启动
sleep 30

# 4. 测试 JMX 指标
./test/test-kafka-jmx-monitoring.sh
```

### 导入 Grafana 面板

1. 访问 Grafana: http://localhost:3000
2. 导航到 Dashboards → Import
3. 点击 "Upload JSON file"
4. 选择文件: `monitoring/grafana/dashboards/kafka-monitoring-advanced.json`
5. 选择 Prometheus 数据源
6. 点击 Import

## 关键指标说明

### 消息吞吐量指标

| 指标 | 说明 | 类型 | 用途 |
|-----|------|------|------|
| kafka_server_brokertopicmetrics_messagesinpersec_total | 消息流入总数 | Counter | 计算消息速率 |
| kafka_server_brokertopicmetrics_bytesinpersec_total | 字节流入总数 | Counter | 计算流入吞吐量 |
| kafka_server_brokertopicmetrics_bytesoutpersec_total | 字节流出总数 | Counter | 计算流出吞吐量 |

使用 `rate()` 函数计算速率:
```promql
rate(kafka_server_brokertopicmetrics_messagesinpersec_total[1m])
```

### 请求性能指标

| 指标 | 说明 | 分位数 |
|-----|------|--------|
| kafka_network_requestmetrics_totaltimems | 请求总时间 | Mean, 50th, 95th, 99th |
| kafka_network_requestmetrics_requestqueuetimems | 请求队列等待时间 | Mean, 50th, 95th, 99th |
| kafka_network_requestmetrics_localtimems | 本地处理时间 | Mean, 50th, 95th, 99th |
| kafka_network_requestmetrics_responsequeuetimems | 响应队列时间 | Mean, 50th, 95th, 99th |

关注 P99 延迟以发现性能问题:
```promql
kafka_network_requestmetrics_totaltimems{request="Produce",quantile="99thPercentile"}
```

### 副本健康指标

| 指标 | 说明 | 正常值 | 告警阈值 |
|-----|------|--------|---------|
| kafka_server_replicamanager_underreplicatedpartitions | 未充分复制的分区 | 0 | > 0 |
| kafka_controller_kafkacontroller_offlinepartitionscount | 离线分区数 | 0 | > 0 |
| kafka_controller_kafkacontroller_activecontrollercount | 活跃控制器数 | 1 | != 1 |

### JVM 健康指标

| 指标 | 说明 | 建议值 |
|-----|------|--------|
| jvm_process_cpu_load | 进程 CPU 使用率 | < 0.8 |
| jvm_memory_heap_used / jvm_memory_heap_max | 堆内存使用率 | < 0.8 |
| rate(jvm_gc_collection_time_ms_total[1m]) | GC 时间占比 | < 100ms/s |

## 故障排查

### 问题 1: JMX Exporter 无法连接

**现象**: JMX Exporter 日志显示连接失败

**检查**:
```bash
# 检查 Kafka JMX 端口
netstat -ano | grep 9999

# 检查 Kafka 环境变量
docker exec kafka env | grep JMX

# 查看 JMX Exporter 日志
docker logs kafka-jmx-exporter
```

**解决**:
```bash
# 重启 Kafka 并确保 JMX 启用
./restart-kafka-with-jmx.sh
```

### 问题 2: Prometheus 没有抓取到 JMX 指标

**检查**:
```bash
# 查看 Prometheus targets
curl http://localhost:9090/api/v1/targets | grep kafka-jmx

# 检查 JMX Exporter 是否可访问
curl http://localhost:5556/metrics
```

**解决**:
```bash
# 重启 Prometheus
docker-compose -f docker-compose-monitoring.yml restart prometheus
```

### 问题 3: Grafana 面板显示 "No data"

**检查步骤**:

1. 在 Prometheus 中验证指标存在:
   ```
   http://localhost:9090/graph
   查询: kafka_server_brokertopicmetrics_messagesinpersec_total
   ```

2. 在 Grafana Explore 中测试查询:
   ```
   Explore → Prometheus → 输入查询 → Run query
   ```

3. 检查时间范围:
   - 确保时间范围包含数据
   - 尝试 "Last 5 minutes"

4. 检查数据源配置:
   - Configuration → Data Sources → Prometheus
   - 测试连接

### 问题 4: 指标值为 0 或不变化

**原因**: Kafka 没有流量

**解决**: 生成测试数据
```bash
# 启动数据采集器
./run-collector.sh

# 或手动发送测试消息
docker exec -it kafka /opt/kafka/bin/kafka-console-producer.sh \
  --bootstrap-server localhost:9092 \
  --topic crypto_ticker
```

## 后续优化建议

### 1. 添加告警规则

创建 `monitoring/prometheus/alerts/kafka.yml`:

```yaml
groups:
  - name: kafka
    interval: 30s
    rules:
      # 未充分复制的分区
      - alert: KafkaUnderReplicatedPartitions
        expr: kafka_server_replicamanager_underreplicatedpartitions > 0
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "Kafka 有未充分复制的分区"
          
      # 离线分区
      - alert: KafkaOfflinePartitions
        expr: kafka_controller_kafkacontroller_offlinepartitionscount > 0
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Kafka 有离线分区"
          
      # 控制器异常
      - alert: KafkaNoActiveController
        expr: kafka_controller_kafkacontroller_activecontrollercount != 1
        for: 1m
        labels:
          severity: critical
        annotations:
          summary: "Kafka 控制器异常"
```

### 2. 添加消费者 LAG 监控

```yaml
  volumes:
    - ./monitoring/kafka-lag-exporter/application.conf:/opt/docker/conf/application.conf
```

用于告警通知 (邮件、Slack、钉钉等):

```yaml
    - /var/log:/var/log
```
