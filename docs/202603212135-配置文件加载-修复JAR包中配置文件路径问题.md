# 配置文件加载 - 修复 JAR 包中配置文件路径问题

**时间**: 2026-03-21 21:35  
**问题**: 配置文件在 JAR 包中无法正确加载  
**状态**: ✅ 已解决

---

## 问题描述

使用 Program Args 成功传递 `APP_ENV=docker` 后，日志显示：

```
Program Arguments: [--APP_ENV, docker]
Found APP_ENV in program arguments: docker
Loading configuration for environment: docker
Warning: Config file not found: application-docker.yml
Using base configuration only
```

说明环境参数传递成功，但配置文件无法加载。

---

## 根本原因 ⭐⭐

**ClassLoader 资源加载路径问题**

原始代码使用单一路径加载配置文件：
```java
InputStream inputStream = getClass().getClassLoader()
    .getResourceAsStream("config/" + filename);
```

这在本地开发时可以工作，但在 JAR 包中可能因为路径问题无法加载。

---

## 解决方案

### 修改 ConfigLoader.loadYamlFile 方法 ✅

添加多路径尝试加载机制：

```java
/**
 * 加载 YAML 文件
 * 支持从 classpath 加载配置文件
 */
private Map<String, Object> loadYamlFile(String filename) {
    try {
        Yaml yaml = new Yaml();
        
        // 尝试多种路径加载配置文件
        InputStream inputStream = null;
        String[] paths = {
            "config/" + filename,  // 标准路径
            filename,              // 根路径
            "/" + filename,        // 绝对路径
            "/config/" + filename  // 绝对路径 + config
        };
        
        for (String path : paths) {
            inputStream = getClass().getClassLoader().getResourceAsStream(path);
            if (inputStream != null) {
                log.info("Successfully loaded config file from path: " + path);
                break;
            }
        }
        
        if (inputStream == null) {
            log.info("Warning: Config file not found: " + filename);
            log.info("Tried paths: " + String.join(", ", paths));
            return new HashMap<>();
        }
        
        Map<String, Object> data = yaml.load(inputStream);
        return data != null ? data : new HashMap<>();
    } catch (Exception e) {
        log.error("Error loading config file: " + filename);
        e.printStackTrace();
        return new HashMap<>();
    }
}
```

## 关键改进

### 1. 多路径尝试机制

尝试以下路径顺序加载配置文件：
1. `config/application-docker.yml` - 标准路径
2. `application-docker.yml` - 根路径
3. `/application-docker.yml` - 绝对路径
4. `/config/application-docker.yml` - 绝对路径 + config

### 2. 详细日志输出

- 成功加载时输出实际使用的路径
- 失败时输出所有尝试过的路径
- 方便排查问题

### 3. 兼容性

- 兼容本地开发环境
- 兼容 JAR 包运行环境
- 兼容不同的 ClassLoader 实现

## 技术要点

### 1. ClassLoader 资源加载

Java ClassLoader 加载资源时：
- 相对路径：`config/application.yml`
- 绝对路径：`/config/application.yml`
- 不同 ClassLoader 可能有不同的行为

### 2. JAR 包资源路径

在 JAR 包中，资源文件的路径结构：
```
realtime-crypto-datawarehouse-1.0.0.jar
├── config/
│   ├── application.yml
│   ├── application-dev.yml
│   ├── application-docker.yml
│   └── application-prod.yml
├── com/
│   └── crypto/
│       └── dw/
│           └── ...
└── META-INF/
    └── ...
```

### 3. Maven Resources Plugin

Maven 会将 `src/main/resources` 下的文件复制到 `target/classes`，然后打包到 JAR 的根目录。

## 常见问题

### Q1: 为什么需要尝试多个路径？

**A**: 不同的运行环境和 ClassLoader 实现可能使用不同的路径规则。多路径尝试可以提高兼容性。

### Q2: 配置文件应该放在哪里？

**A**: 放在 `src/main/resources/config/` 目录下，Maven 会自动打包到 JAR 的 `config/` 目录。

### Q3: 如何验证配置文件是否正确打包？

**A**: 使用命令：
```bash
jar tf target/realtime-crypto-datawarehouse-1.0.0.jar | grep "application.*yml"
```

### Q4: 如果配置文件仍然无法加载怎么办？

**A**: 查看日志中的 "Tried paths" 信息，确认尝试了哪些路径，然后检查 JAR 包中的实际路径。
