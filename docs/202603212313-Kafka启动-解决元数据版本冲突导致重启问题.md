# Kafka 启动 - 解决元数据版本冲突导致重启问题

## 问题描述

Kafka 容器一直重启，无法正常启动，同时 Zookeeper 也没有启动。

## 问题现象

1. **Kafka 容器重启循环**: 容器启动后立即退出，然后重新启动
2. **Zookeeper 未启动**: 没有看到 Zookeeper 容器运行
3. **旧的 Kafka 配置残留**: 监控系统中的 kafka-exporter 和 kafka-jmx-exporter 仍在运行

## 根本原因

### 1. 多个 Kafka 配置冲突

项目中存在多个 Kafka 配置：
- `docker-compose-kafka.yml` - 新的独立 Kafka 配置（包含 Zookeeper）
- `docker-compose-monitoring.yml` - 监控系统中的 Kafka 配置（只有 exporter）

旧的监控配置启动了 kafka-exporter，但没有启动实际的 Kafka 服务。

### 2. 元数据版本冲突 ⭐⭐⭐

Kafka 启动失败的核心错误：

```
java.lang.RuntimeException: Found unexpected version in loaded `meta.properties`: 
{cluster.id=MkU3OEVBNTcwNTJENDM2Qg, node.id=1, version=1}. 
Zk-based brokers only support version 0 (which is implicit when the `version` field is missing).
```

**原因分析**:
- Kafka 的数据卷中保存了旧的元数据文件 `meta.properties`
- 旧数据使用的是 `version=1`（KRaft 模式的元数据格式）
- 当前配置使用 Zookeeper 模式，只支持 `version=0`
- 两种模式的元数据格式不兼容

**Kafka 两种模式**:
- **Zookeeper 模式**: 传统模式，使用外部 Zookeeper 管理元数据（version=0）
- **KRaft 模式**: 新模式，Kafka 自己管理元数据，不需要 Zookeeper（version=1）

## 改进措施

### 1. 更新 manage-kafka.sh 脚本

添加 `clean` 操作，方便以后清理数据卷：

```bash
#!/bin/bash

# Kafka 管理脚本
# 用法: ./manage-kafka.sh [start|stop|restart|clean]
# 默认: restart
# clean: 停止并清理所有数据卷（用于解决启动问题）

set -e

ACTION=${1:-restart}

case "$ACTION" in
    start)
        echo "启动 Kafka 集群..."
        docker-compose -f docker-compose-kafka.yml up -d
        
        echo "等待 Kafka 启动..."
        sleep 15
        
        # 创建 Topic
        echo "创建 crypto_ticker Topic..."
        docker exec kafka kafka-topics \
            --create \
            --topic crypto_ticker \
            --bootstrap-server localhost:9092 \
            --partitions 4 \
            --replication-factor 1 \
            --config retention.ms=604800000 \
            --config segment.ms=3600000 \
            --config compression.type=lz4 \
            --if-not-exists 2>/dev/null || echo "Topic 已存在"
        
        echo "✅ Kafka 启动完成"
        ;;
        
    stop)
        echo "停止 Kafka 集群..."
        docker-compose -f docker-compose-kafka.yml down
        echo "✅ Kafka 已停止"
        ;;
        
    clean)
        echo "停止 Kafka 并清理数据卷..."
        docker-compose -f docker-compose-kafka.yml down -v
        echo "✅ Kafka 已停止，数据卷已清理"
        echo "⚠️  注意：所有 Kafka 数据已删除，下次启动将是全新环境"
        ;;
        
    restart)
        echo "重启 Kafka 集群..."
        $0 stop
        sleep 3
        $0 start
        ;;
        
    *)
        echo "错误: 未知操作 '$ACTION'"
        echo "用法: $0 [start|stop|restart|clean]"
        echo ""
        echo "操作说明:"
        echo "  start   - 启动 Kafka 和 Zookeeper"
        echo "  stop    - 停止 Kafka 和 Zookeeper"
        echo "  restart - 重启 Kafka 和 Zookeeper"
        echo "  clean   - 停止并清理所有数据（解决启动问题）"
        exit 1
        ;;
esac

echo ""
echo "Kafka 状态:"
docker-compose -f docker-compose-kafka.yml ps
```

### 2. 使用方法

```bash
# 正常启动
bash manage-kafka.sh start

# 停止
bash manage-kafka.sh stop

# 重启
bash manage-kafka.sh restart

# 清理数据并重新开始（解决启动问题）
bash manage-kafka.sh clean
bash manage-kafka.sh start
```

## 常见问题

### Q1: 为什么需要清理数据卷？

**A**: 当 Kafka 的配置模式改变时（如从 KRaft 切换到 Zookeeper），旧的元数据文件会导致启动失败。清理数据卷可以删除旧的元数据，让 Kafka 以全新状态启动。

### Q2: 清理数据卷会丢失什么？

**A**: 会丢失：
- Kafka 中的所有消息数据
- Topic 配置
- Consumer Group 的 offset
- Zookeeper 中的元数据

**建议**: 只在开发环境或确认可以丢失数据时使用。

### Q3: 如何避免这个问题？

**A**: 
1. 统一使用一种模式（Zookeeper 或 KRaft）
2. 不要在不同模式之间切换
3. 如果必须切换，先清理数据卷
4. 生产环境使用持久化存储，并做好备份

### Q4: Zookeeper 模式 vs KRaft 模式，选哪个？

**A**: 
- **Zookeeper 模式**: 成熟稳定，生产环境推荐（Kafka 3.x 之前的标准）
- **KRaft 模式**: 简化架构，未来趋势（Kafka 3.3+ 生产可用）

本项目使用 Zookeeper 模式，因为更稳定且兼容性好。
