# 架构问题 - 修复 Flink 作业数据流转错误

## 问题发现

用户发现了一个严重的架构问题:**三个 Flink 作业都在消费同一个 Kafka Topic**,这违反了数据仓库分层架构的设计原则。

## 当前错误架构

```
┌─────────────────────────────────────────────────────────────┐
│                      当前错误架构                            │
└─────────────────────────────────────────────────────────────┘

OKX WebSocket → Kafka (crypto-ticker)
                    ↓
        ┌───────────┼───────────┐
        ↓           ↓           ↓
    ODS 作业    DWD 作业    DWS 作业  ❌ 三个作业都消费 Kafka
        ↓           ↓           ↓
    ODS 表      DWD 表      DWS 表
```

### 问题分析

**1. FlinkODSJobDataStream.java** ✅ 正确
```java
// 消费 Kafka Topic
KafkaSource<String> kafkaSource = KafkaSource.<String>builder()
    .setTopics(config.getString("kafka.topic.crypto-ticker"))  // ✅ 正确
    .build();

// 写入 ODS 表
DorisSink<String> dorisSink = createDorisSink(config);  // → ods_crypto_ticker_rt
```

**2. FlinkDWDJobSQL.java** ❌ 错误
```java
// 错误：直接消费 Kafka Topic
CREATE TABLE kafka_ods_source (
    ...
) WITH (
    'connector' = 'kafka',
    'topic' = 'crypto-ticker',  // ❌ 错误：应该从 ODS 表读取
    ...
)

// 写入 DWD 表
INSERT INTO doris_dwd_sink  // → dwd_crypto_ticker_detail
SELECT ... FROM kafka_ods_source;  // ❌ 错误：应该从 ODS 表读取
```

**3. FlinkDWSJob1MinSQL.java** ❌ 错误
```java
// 错误：直接消费 Kafka Topic
CREATE TABLE kafka_ods_source (
    ...
) WITH (
    'connector' = 'kafka',
    'topic' = 'crypto-ticker',  // ❌ 错误：应该从 DWD 表读取
    ...
)

// 写入 DWS 表
INSERT INTO doris_dws_sink  // → dws_crypto_ticker_1min
SELECT ... FROM kafka_ods_source;  // ❌ 错误：应该从 DWD 表读取
```

### 问题影响

1. **数据重复消费**: 三个作业都消费同一个 Kafka Topic,造成资源浪费
2. **架构混乱**: 违反了数据仓库分层架构的设计原则
3. **数据不一致**: DWD 和 DWS 层没有使用 ODS 层的数据,可能导致数据不一致
4. **维护困难**: 数据转换逻辑分散在多个作业中,难以维护
5. **无法回溯**: DWD 和 DWS 层无法从 ODS 层重新计算历史数据

---

## 正确的架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                      正确的架构设计                          │
└─────────────────────────────────────────────────────────────┘

1. 数据采集层
   OKX WebSocket → Kafka (crypto-ticker)
                       ↓
                   ODS 作业 (消费 Kafka)
                       ↓
                   ods_crypto_ticker_rt (Doris)

2. 数据清洗层
   ods_crypto_ticker_rt (Doris)
                       ↓
                   DWD 作业 (读取 ODS 表)
                       ↓
                   dwd_crypto_ticker_detail (Doris)

3. 数据汇总层
   dwd_crypto_ticker_detail (Doris)
                       ↓
                   DWS 作业 (读取 DWD 表)
                       ↓
                   dws_crypto_ticker_1min (Doris)
```

### 架构优势

1. **分层清晰**: 每一层只依赖上一层的数据
2. **单一数据源**: Kafka 只被 ODS 作业消费一次
3. **数据一致性**: DWD 和 DWS 层使用 ODS 层的数据,保证数据一致
4. **易于维护**: 每一层的转换逻辑独立,易于维护和调试
5. **支持回溯**: 可以从 ODS 层重新计算 DWD 和 DWS 层的历史数据

## 修复方案

### 方案 1: 使用 Doris Connector 读取上游表 (推荐) ⭐⭐⭐

**优势**:
- 直接从 Doris 表读取数据
- 支持增量读取和全量读取
- 性能好,延迟低

**DWD 作业修改**:
```java
// 修改前：从 Kafka 读取
CREATE TABLE kafka_ods_source (...) WITH ('connector' = 'kafka', ...)

// 修改后：从 Doris ODS 表读取
CREATE TABLE doris_ods_source (
    inst_id STRING,
    `timestamp` BIGINT,
    last_price DECIMAL(20, 8),
    bid_price DECIMAL(20, 8),
    ask_price DECIMAL(20, 8),
    bid_size DECIMAL(20, 8),
    ask_size DECIMAL(20, 8),
    volume_24h DECIMAL(30, 8),
    high_24h DECIMAL(20, 8),
    low_24h DECIMAL(20, 8),
    open_24h DECIMAL(20, 8),
    data_source STRING,
    ingest_time BIGINT
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'table.identifier' = 'crypto_dw.ods_crypto_ticker_rt',
    'username' = 'root',
    'password' = ''
);

// 从 Doris ODS 表读取并写入 DWD 表
INSERT INTO doris_dwd_sink
SELECT 
    inst_id,
    `timestamp`,
    -- 数据清洗和转换逻辑
    ...
FROM doris_ods_source;
```

**DWS 作业修改**:
```java
// 从 Doris DWD 表读取
CREATE TABLE doris_dwd_source (
    inst_id STRING,
    `timestamp` BIGINT,
    trade_date DATE,
    trade_hour INT,
    last_price DECIMAL(20, 8),
    ...
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'table.identifier' = 'crypto_dw.dwd_crypto_ticker_detail',
    'username' = 'root',
    'password' = ''
);

// 窗口聚合并写入 DWS 表
INSERT INTO doris_dws_sink
SELECT 
    inst_id,
    TUMBLE_START(ts, INTERVAL '1' MINUTE) as window_start,
    TUMBLE_END(ts, INTERVAL '1' MINUTE) as window_end,
    -- 聚合逻辑
    ...
FROM doris_dwd_source
GROUP BY inst_id, TUMBLE(ts, INTERVAL '1' MINUTE);
```

### 方案 2: 使用 CDC (Change Data Capture) ⭐⭐

**优势**:
- 实时捕获 Doris 表的变更
- 支持增量同步
- 适合实时数据仓库

**实现方式**:
- 使用 Flink CDC Connector for Doris (如果支持)
- 或者使用 Debezium + Kafka Connect

**缺点**:
- Doris 的 CDC 支持可能不完善
- 配置复杂

### 方案 3: 使用中间 Kafka Topic ⭐

**优势**:
- 解耦各层作业
- 支持多个下游消费者

**实现方式**:
```
OKX → Kafka (crypto-ticker) → ODS 作业 → ODS 表
                                          ↓
                                    Kafka (ods-topic)
                                          ↓
                                      DWD 作业 → DWD 表
                                                  ↓
                                            Kafka (dwd-topic)
                                                  ↓
                                              DWS 作业 → DWS 表
```

**缺点**:
- 增加了 Kafka Topic 数量
- 增加了系统复杂度
- 需要额外的 Kafka 资源

## 注意事项

### 1. Doris Connector 读取模式

Doris Flink Connector 支持两种读取模式:

**批量读取 (Batch Read)**:
- 一次性读取全部数据
- 适合离线计算和数据回溯
- 作业会自动结束

**流式读取 (Stream Read)**:
- 持续读取新数据
- 适合实时计算
- 作业持续运行

**配置方式**:
```java
// 批量读取（默认）
'scan.mode' = 'batch'

// 流式读取（需要 Doris 支持 CDC）
'scan.mode' = 'stream'
```

### 2. 数据延迟

使用 Doris Connector 读取数据会有一定延迟:
- ODS 作业写入 Doris 的延迟: ~1-5 秒
- DWD 作业从 Doris 读取的延迟: ~1-5 秒
- 总延迟: ~2-10 秒

如果对延迟要求极高,可以考虑使用 Kafka 中间 Topic。

### 3. 数据一致性

使用 Doris Connector 读取数据时,需要注意:
- Doris 的数据可见性延迟
- Flink Checkpoint 和 Doris 事务的一致性
- 建议启用 Flink Checkpoint 保证 Exactly-Once 语义
