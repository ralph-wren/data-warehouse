# Doris 增量读取 - 官方文档分析和解决方案

**时间**: 2026-03-23 00:30  
**问题**: 根据 Doris 官方文档修改支持增量读取  
**结论**: Doris Source 不支持 CDC 增量读取,提供了多种替代方案

---

## 问题背景

用户希望 Flink 从 Doris 表读取数据时,能够支持增量读取,而不是每次都读取全表。

**用户需求**:
- Flink 作业从 Doris 表读取数据
- 只读取新增的数据 (增量读取)
- 避免重复处理已处理的数据

---

## 官方文档分析

### 1. Doris Source 的限制 ⚠️

根据 Doris 官方文档 (https://doris.apache.org/zh-CN/docs/3.x/ecosystem/flink-doris-connector):

> **Doris Source 目前是有界流,不支持以 CDC 的方式持续读取。**

**关键信息**:
- Doris Source 是有界流 (Bounded Stream)
- 读取的是表的快照数据 (Snapshot Read)
- 不支持 CDC (Change Data Capture) 方式的增量读取
- 读取完成后 Source 就结束了
- 不会持续监听表的变化

### 2. Doris Source 的读取方式

Doris Connector 支持两种读取方式:

#### 方式 1: Thrift 读取 (默认)

```java
CREATE TABLE doris_source (
    inst_id STRING,
    timestamp BIGINT,
    ...
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'table.identifier' = 'crypto_dw.ods_crypto_ticker_rt',
    'username' = 'root',
    'password' = '',
    'doris.read.field' = '*'  -- 指定读取字段
)
```

**特点**:
- 兼容性最好,支持所有 Doris 版本
- 性能中等
- 适合小数据量场景

#### 方式 2: ArrowFlightSQL 读取 (推荐,Doris 2.1+)

```java
CREATE TABLE doris_source (
    inst_id STRING,
    timestamp BIGINT,
    ...
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'table.identifier' = 'crypto_dw.ods_crypto_ticker_rt',
    'username' = 'root',
    'password' = '',
    -- ArrowFlightSQL 配置
    'doris.deserialize.arrow.async' = 'true',
    'doris.deserialize.queue.size' = '64',
    'doris.request.query.timeout.s' = '3600',
    'doris.read.field' = '*'
)
```

**特点**:
- 性能提升 2-10 倍 (相比 Thrift)
- 内存占用更少
- 支持更大的数据量
- 异步反序列化,提高吞吐量
- 要求 Doris 2.1+ 版本

### 3. 官方推荐的增量同步方案

官方文档提供了 **Flink CDC 整库同步** 功能:

> 如果需要增量同步,可以使用 Flink CDC 整库同步功能。该功能集成了 Flink CDC,可以同步 MySQL、Oracle、PostgreSQL、SQLServer 等数据库到 Doris。

**架构**:
```
MySQL (源数据库)
  ↓ (Flink CDC 监听 Binlog)
Flink CDC 整库同步工具
  ↓ (实时写入)
Doris (目标数据库)
```

**适用场景**:
- 从 MySQL 等数据库实时同步到 Doris
- 需要 CDC 增量同步能力
- 需要整库或多表同步

**限制**:
- 只支持从外部数据库同步到 Doris
- **不支持 Doris 表之间的 CDC 同步** ⚠️
- 需要源数据库支持 CDC (如 MySQL Binlog)

---

## 解决方案

由于 Doris Source 不支持 CDC 增量读取,我们提供了以下替代方案:

### 方案 1: 定时批处理 (推荐) ⭐⭐⭐

**原理**: 使用调度工具定时启动 Flink 作业,配合时间过滤只处理最近的数据

**实现**:
```bash
# 使用 Cron 定时任务,每小时执行一次
0 * * * * /path/to/run-flink-dwd-sql.sh

# 在 Flink SQL 中使用时间过滤
SELECT * FROM ods_crypto_ticker_rt 
WHERE ingest_time >= UNIX_TIMESTAMP(DATE_SUB(NOW(), INTERVAL 1 HOUR)) * 1000 
  AND ingest_time < UNIX_TIMESTAMP(NOW()) * 1000
```

**优点**:
- 简单易实现
- 不需要额外工具
- 可以精确控制处理的数据范围

**缺点**:
- 实时性较低 (分钟级)
- 每次都需要重新启动作业

**适用场景**:
- 离线分析
- 定时 ETL
- 对实时性要求不高的场景

### 方案 2: Kafka 中间层 (推荐,当前架构) ⭐⭐⭐

**原理**: 实时层使用 Kafka (无界流),批处理层使用 Doris Source (有界流)

**架构**:
```
实时层 (持续运行 7x24):
├── WebSocket → Kafka (无界流)
└── Kafka → Doris ODS (无界流)

批处理层 (定时运行):
├── Doris ODS → Doris DWD (有界流,定时)
├── Doris DWD → Doris DWS (有界流,定时)
└── Doris DWS → Doris ADS (有界流,定时)
```

**优点**:
- 实时性高 (秒级)
- 架构清晰,职责分明
- Kafka 提供持久化和回溯能力

**缺点**:
- 需要维护 Kafka 集群
- 架构相对复杂

**适用场景**:
- 实时数据仓库 (当前项目) ✅
- 需要高实时性的场景
- 需要数据回溯的场景

### 方案 3: Flink CDC 整库同步 (官方推荐) ⭐⭐

**原理**: 使用 Flink CDC 监听 MySQL Binlog,实时同步到 Doris

**实现**:
```yaml
# flink-cdc-sync-config.yaml
source:
  type: mysql
  hostname: localhost
  port: 3306
  username: root
  password: password
  database-name: source_db
  table-name: .*  # 同步所有表

sink:
  type: doris
  fenodes: 127.0.0.1:8030
  username: root
  password: ""
  database-name: target_db
  sink.enable-2pc: true
```

```bash
# 启动 Flink CDC 整库同步
flink run \
  -c com.ververica.cdc.cli.CliFrontend \
  flink-cdc-cli.jar \
  --config flink-cdc-sync-config.yaml
```

**优点**:
- 真正的 CDC 增量同步
- 实时性高 (秒级)
- 支持 INSERT/UPDATE/DELETE

**缺点**:
- 只支持从外部数据库同步到 Doris
- 不支持 Doris 表之间的 CDC 同步
- 需要源数据库支持 CDC

**适用场景**:
- MySQL → Doris 实时同步
- 需要 CDC 能力的场景
- 整库或多表同步

### 方案 4: 增量标记表 (自己实现) ⭐

**原理**: 使用标记表记录上次处理的时间戳,只读取新数据

**实现**:
```sql
-- 创建增量标记表
CREATE TABLE ods_process_watermark (
    job_name VARCHAR(100),
    last_process_time BIGINT,
    update_time DATETIME
);

-- 只读取新数据
SELECT * FROM ods_crypto_ticker_rt 
WHERE ingest_time > (
    SELECT last_process_time 
    FROM ods_process_watermark 
    WHERE job_name = 'flink-dwd-job'
);

-- 处理完成后更新标记
UPDATE ods_process_watermark 
SET last_process_time = UNIX_TIMESTAMP(NOW()) * 1000 
WHERE job_name = 'flink-dwd-job';
```

**优点**:
- 简单易实现
- 不需要额外工具
- 可以精确控制增量范围

**缺点**:
- 需要手动管理标记表
- 不是真正的 CDC (无法捕获 UPDATE/DELETE)
- 依赖时间戳字段

**适用场景**:
- 简单的增量处理
- 只需要处理 INSERT 操作
- 有时间戳字段的表

## 方案对比

### 读取方式对比

| 读取方式 | 性能 | 兼容性 | 适用场景 | Doris 版本要求 |
|---------|------|--------|---------|---------------|
| Thrift | 中等 | 最好 | 小数据量,兼容性优先 | 所有版本 |
| ArrowFlightSQL | 高 (2-10倍) | 好 | 大数据量,性能优先 | 2.1+ |

### 增量读取方案对比

| 方案 | 实时性 | 复杂度 | CDC 支持 | 适用场景 |
|------|--------|--------|---------|---------|
| 定时批处理 | 低 (分钟级) | 低 | ❌ | 离线分析,定时 ETL |

## 推荐方案

### 当前项目推荐: Kafka 中间层 (方案 2) ✅

**理由**:
1. 实时性高 (秒级)
2. 架构清晰,职责分明
3. Kafka 提供持久化和回溯能力
4. 已经在使用,不需要额外改造

**架构**:
```
实时层 (持续运行):
├── WebSocket → Kafka (无界流)
└── Kafka → Doris ODS (无界流)

批处理层 (定时运行):
├── Doris ODS → Doris DWD (有界流,定时)
├── Doris DWD → Doris DWS (有界流,定时)
└── Doris DWS → Doris ADS (有界流,定时)
```

### 性能优化建议

1. **使用 ArrowFlightSQL 读取方式** (Doris 2.1+)
   - 性能提升 2-10 倍
   - 内存占用更少

2. **使用分区读取**
   ```sql
   SELECT * FROM ods_crypto_ticker_rt 
   WHERE trade_date = '2026-03-22'  -- 只读取特定分区
   ```

3. **使用时间过滤**
   ```sql
   SELECT * FROM ods_crypto_ticker_rt 
   WHERE ingest_time >= UNIX_TIMESTAMP(DATE_SUB(NOW(), INTERVAL 1 HOUR)) * 1000
   ```

4. **配置合理的并行度**
   ```yaml
   flink:
     execution:
       parallelism: 4  # 根据数据量调整
   ```

## 总结

### 关键发现

1. **Doris Source 不支持 CDC 增量读取** (官方文档明确说明)
2. **Doris Source 是有界流** (读取完就结束)
3. **官方推荐使用 Flink CDC 整库同步** (从 MySQL 等数据库同步到 Doris)
4. **当前项目架构已经是最佳实践** (Kafka 中间层)
