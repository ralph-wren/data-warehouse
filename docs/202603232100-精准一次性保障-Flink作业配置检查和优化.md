## 二、Doris Source 的精准一次性保障 ⭐⭐⭐

### Doris Source 的特殊性

**重要说明**: Doris 作为 Source 时，精准一次性保障机制与 Kafka/Flink 不同。

### 1. Doris Source 的读取模式

**当前使用**: ArrowFlightSQL 批量读取模式

```java
'connector' = 'doris',
'doris.deserialize.arrow.async' = 'true',  // 异步反序列化
'doris.deserialize.queue.size' = '64',     // 反序列化队列大小
'doris.request.query.timeout.s' = '3600',  // 查询超时时间
```

**特点**:
- ✅ **快照读取**: 读取 Doris 表的快照数据（某个时间点的完整数据）
- ✅ **幂等性**: 多次读取同一快照，结果完全一致
- ✅ **无重复**: 同一条数据不会被读取多次
- ❌ **不支持 CDC**: 无法捕获增量变更（INSERT/UPDATE/DELETE）
- ❌ **不支持实时**: 需要重新启动作业才能读取新数据

### 2. Doris Source 的精准一次性保障

**保障机制**:

1. **Checkpoint 恢复时重新读取**
   - Flink 作业从 Checkpoint 恢复时，Doris Source 会重新执行查询
   - 由于读取的是快照数据，重新读取的结果与之前完全一致
   - 不会产生重复数据

2. **分区裁剪和过滤下推**
   - Doris Connector 支持分区裁剪（Partition Pruning）
   - WHERE 条件会下推到 Doris 执行，减少数据传输
   - 保证每次读取的数据范围一致

3. **与 Checkpoint 的配合**
   ```
   Checkpoint N:
   ├─ Doris Source: 读取快照 [0, T1]
   ├─ 处理数据
   └─ Doris Sink: 写入数据
   
   故障恢复 (从 Checkpoint N):
   ├─ Doris Source: 重新读取快照 [0, T1]  ← 结果完全一致
   ├─ 重新处理数据
   └─ Doris Sink: 重新写入数据（使用相同 Label，去重）
   ```

### 3. 端到端精准一次性保障

**数据流**: Doris ODS → Flink DWD → Doris DWD

```
┌─────────────────────────────────────────────────────────┐
│              端到端精准一次性保障                          │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌──────────────┐      ┌──────────────┐      ┌────────┐│
│  │  Doris ODS   │─────►│  Flink DWD   │─────►│ Doris  ││
│  │   (Source)   │      │   (Process)  │      │  DWD   ││
│  │              │      │              │      │ (Sink) ││
│  └──────────────┘      └──────────────┘      └────────┘│
│         │                     │                    │    │
│         │                     │                    │    │
│    快照读取              Checkpoint           两阶段提交  │
│    (幂等性)              (状态恢复)           (去重)     │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**保障要点**:

1. **Doris Source (读取端)**:
   - ✅ 快照读取，幂等性保证
   - ✅ Checkpoint 恢复时重新读取相同数据
   - ✅ 无需额外配置

2. **Flink Processing (处理端)**:
   - ✅ Checkpoint 保存处理状态
   - ✅ 故障恢复时从 Checkpoint 恢复
   - ✅ 需要配置 `CheckpointingMode.EXACTLY_ONCE`

3. **Doris Sink (写入端)**:
   - ✅ 使用 Label 机制去重
   - ✅ 两阶段提交（2PC）保证原子性
   - ✅ 需要配置 `sink.enable-2pc = true`

### 4. 关键配置

**Doris Source 配置** (无需额外配置):
```java
// Doris Source 天然支持精准一次性
'connector' = 'doris',
'fenodes' = '127.0.0.1:8030',
'table.identifier' = 'crypto_dw.ods_crypto_ticker_rt',
// 快照读取模式，幂等性保证
'doris.deserialize.arrow.async' = 'true',
```

**Doris Sink 配置** (需要启用 2PC):
```java
// Doris Sink 需要启用两阶段提交
'connector' = 'doris',
'fenodes' = '127.0.0.1:8030',
'benodes' = '127.0.0.1:8040',
'table.identifier' = 'crypto_dw.dwd_crypto_ticker_detail',
'sink.label-prefix' = 'dwd_' + System.currentTimeMillis(),  // Label 去重
'sink.enable-2pc' = 'true',  // ⭐ 启用两阶段提交（关键）
'sink.properties.format' = 'json',
'sink.properties.read_json_by_line' = 'true'
```

### 5. 注意事项

**优点**:
- ✅ 实现简单，无需额外配置
- ✅ 性能高效，批量读取
- ✅ 数据一致性强

**限制**:
- ❌ 不支持实时增量读取（CDC）
- ❌ 需要重启作业才能读取新数据
- ❌ 不适合高频更新的场景

**适用场景**:
- ✅ 批量 ETL 作业
- ✅ 定时调度作业
- ✅ 数据分层处理（ODS → DWD → DWS）
- ❌ 实时 CDC 同步

**方案 3: 使用 Kafka 作为中间层**

## 三、优化方案

### 方案 1: 修复 Kafka Sink（FlinkDataCollectorJob）

```java
// 修改前
.setDeliveryGuarantee(DeliveryGuarantee.AT_LEAST_ONCE)

// 修改后
.setDeliveryGuarantee(DeliveryGuarantee.EXACTLY_ONCE)
.setTransactionalIdPrefix("flink-collector-txn-" + System.currentTimeMillis())
```

**注意**: 需要在 Kafka Producer 配置中添加：
```yaml
kafka:
  producer:
    transaction-timeout-ms: 900000  # 15分钟
    enable-idempotence: true
```

### 方案 2: 完善 Checkpoint 配置（FlinkEnvironmentFactory）

```java
// 在 createStreamEnvironment 和 createTableEnvironment 中添加

// 1. 显式设置 Checkpoint 模式
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// 2. 设置 Checkpoint 超时时间
env.getCheckpointConfig().setCheckpointTimeout(300000);  // 5分钟

// 3. 设置最小间隔
env.getCheckpointConfig().setMinPauseBetweenCheckpoints(500);

// 4. 设置最大并发 Checkpoint 数
env.getCheckpointConfig().setMaxConcurrentCheckpoints(1);

// 5. 设置 Checkpoint 失败容忍次数
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);

// 6. 设置 Checkpoint 清理策略（作业取消时保留 Checkpoint）
env.getCheckpointConfig().setExternalizedCheckpointCleanup(
    CheckpointConfig.ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
);

// 7. 配置状态后端
StateBackend stateBackend = new EmbeddedRocksDBStateBackend(true);  // 启用增量 Checkpoint
env.setStateBackend(stateBackend);

// 8. 设置 Checkpoint 存储
String checkpointDir = config.getString("flink.state.checkpoint-dir");
env.getCheckpointConfig().setCheckpointStorage(checkpointDir);
```

### 方案 3: 优化 Doris Sink 配置（FlinkODSJobDataStream）

```java
DorisExecutionOptions executionOptions = DorisExecutionOptions.builder()
    .setStreamLoadProp(streamLoadProp)
    .setMaxRetries(3)
    .setBufferSize(10 * 1024 * 1024)
    .setBufferCount(3)
    .setBufferFlushMaxRows(batchSize)
    .setBufferFlushIntervalMs(batchIntervalMs)
    .setLabelPrefix("flink-ods-" + System.currentTimeMillis())
    
    // 新增：启用 Checkpoint 时刷新数据
    .setCheckpointFlush(true)  // Checkpoint 时强制刷新缓冲区
    
    // 新增：设置 Stream Load 超时时间
    .setStreamLoadTimeout(600000)  // 10分钟
    
    .build();
```

**重要**: 同时需要在 FlinkTableFactory 中启用 Doris Sink 的两阶段提交：

```java
// 在 createDorisSinkTable 方法中添加
"'sink.enable-2pc' = 'true',\n" +  // ⭐ 启用两阶段提交（精准一次性必需）
"'sink.properties.format' = 'json',\n" +
```

## 七、注意事项

### 1. Kafka 事务性能影响

启用 `EXACTLY_ONCE` 会影响性能:
- 吞吐量降低约 20-30%
- 延迟增加约 50-100ms

**建议**:
- 如果对数据重复容忍度高，可以使用 `AT_LEAST_ONCE`
- 如果必须保证精准一次性，接受性能损失

### 2. Checkpoint 间隔选择

- 间隔太短：频繁 Checkpoint，影响性能
- 间隔太长：故障恢复时间长，数据丢失风险高

**建议**:
- 开发环境：60秒
- 生产环境：30-60秒
- 高吞吐场景：120秒

### 3. 状态后端选择

- **MemoryStateBackend**: 适合小状态（<100MB）
- **FsStateBackend**: 适合中等状态（100MB-1GB）
- **RocksDBStateBackend**: 适合大状态（>1GB）

**当前使用**: RocksDB（正确选择）
