# Doris Source 精准一次性保障机制说明

**日期**: 2026-03-23 21:10  
**问题**: Doris 作为 Source 时如何保证精准一次性语义  
**结论**: Doris Source 通过快照读取和幂等性天然支持精准一次性

---

## 一、Doris Source 的特殊性

### 与 Kafka Source 的区别

| 特性 | Kafka Source | Doris Source |
|-----|-------------|--------------|
| 读取模式 | 流式读取（持续消费） | 批量读取（快照） |
| 增量支持 | ✅ 支持 CDC | ❌ 不支持 CDC |
| 实时性 | ✅ 实时 | ❌ 需要重启作业 |
| 精准一次性 | 需要配置 Offset 提交 | 天然支持（幂等性） |
| 重复读取 | 可能重复（如果 Offset 未提交） | 不会重复（快照一致） |

### Doris Source 的读取模式

**当前使用**: ArrowFlightSQL 批量读取模式

```java
'connector' = 'doris',
'doris.deserialize.arrow.async' = 'true',  // 异步反序列化
'doris.deserialize.queue.size' = '64',     // 反序列化队列大小
'doris.request.query.timeout.s' = '3600',  // 查询超时时间
```

**工作原理**:
1. Flink 作业启动时，Doris Source 执行一次 SELECT 查询
2. 读取 Doris 表的快照数据（某个时间点的完整数据）
3. 将数据分批传输到 Flink
4. 读取完成后，Source 结束（有界流）

## 二、精准一次性保障机制

### 1. 快照读取的幂等性

**核心原理**: 多次读取同一快照，结果完全一致

```
时间线:
T0: Doris 表状态 [A, B, C]
T1: Flink 作业启动，读取快照 [A, B, C]
T2: Doris 表新增数据 [D, E]
T3: Flink 作业故障，从 Checkpoint 恢复
T4: Doris Source 重新读取快照 [A, B, C]  ← 结果完全一致
```

**关键点**:
- ✅ 快照数据不会改变（已经持久化）
- ✅ 重新读取的结果与之前完全一致
- ✅ 不会产生重复数据
- ⚠️ 新增数据（D, E）不会被读取（需要重启作业）

### 2. Checkpoint 恢复机制

**Checkpoint 保存的状态**:
```
Checkpoint N:
├─ Doris Source State:
│  ├─ 查询 SQL: SELECT * FROM ods_crypto_ticker_rt
│  ├─ 读取进度: 已读取 10000 行
│  └─ 分区信息: p20260323
├─ Processing State:
│  └─ 窗口状态、聚合状态等
└─ Doris Sink State:
   └─ 待提交的 Label: dwd_1234567890
```

**故障恢复流程**:
```
1. Flink 作业故障
2. 从 Checkpoint N 恢复
3. Doris Source 重新执行查询（从读取进度继续）
4. 重新处理数据
5. Doris Sink 使用相同 Label 写入（去重）
```

### 3. 端到端精准一次性保障

**数据流**: Doris ODS → Flink DWD → Doris DWD

```
┌─────────────────────────────────────────────────────────┐
│              端到端精准一次性保障                          │
├─────────────────────────────────────────────────────────┤
│                                                          │
│  ┌──────────────┐      ┌──────────────┐      ┌────────┐│
│  │  Doris ODS   │─────►│  Flink DWD   │─────►│ Doris  ││
│  │   (Source)   │      │   (Process)  │      │  DWD   ││
│  │              │      │              │      │ (Sink) ││
│  └──────────────┘      └──────────────┘      └────────┘│
│         │                     │                    │    │
│         │                     │                    │    │
│    快照读取              Checkpoint           两阶段提交  │
│    (幂等性)              (状态恢复)           (去重)     │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**保障要点**:

1. **Doris Source (读取端)**:
   - ✅ 快照读取，幂等性保证
   - ✅ Checkpoint 恢复时重新读取相同数据
   - ✅ 无需额外配置

2. **Flink Processing (处理端)**:
   - ✅ Checkpoint 保存处理状态
   - ✅ 故障恢复时从 Checkpoint 恢复
   - ✅ 需要配置 `CheckpointingMode.EXACTLY_ONCE`

3. **Doris Sink (写入端)**:
   - ✅ 使用 Label 机制去重
   - ✅ 两阶段提交（2PC）保证原子性
   - ✅ 需要配置 `sink.enable-2pc = true`

## 三、关键配置

### 1. Doris Source 配置

**无需额外配置**，天然支持精准一次性：

```java
CREATE TABLE doris_ods_source (
    inst_id STRING,
    `timestamp` BIGINT,
    last_price DECIMAL(20, 8),
    -- ... 其他字段
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'table.identifier' = 'crypto_dw.ods_crypto_ticker_rt',
    'username' = 'root',
    'password' = '',
    -- 快照读取模式，幂等性保证
    'doris.deserialize.arrow.async' = 'true',
    'doris.deserialize.queue.size' = '64',
    'doris.request.query.timeout.s' = '3600'
)
```

### 2. Doris Sink 配置

**必须启用两阶段提交**：

```java
CREATE TABLE doris_dwd_sink (
    inst_id STRING,
    `timestamp` BIGINT,
    trade_date DATE,
    -- ... 其他字段
) WITH (
    'connector' = 'doris',
    'fenodes' = '127.0.0.1:8030',
    'benodes' = '127.0.0.1:8040',
    'table.identifier' = 'crypto_dw.dwd_crypto_ticker_detail',
    'username' = 'root',
    'password' = '',
    'sink.label-prefix' = 'dwd_' + System.currentTimeMillis(),  // Label 去重
    'sink.enable-2pc' = 'true',  // ⭐ 启用两阶段提交（关键）
    'sink.properties.format' = 'json',
    'sink.properties.read_json_by_line' = 'true'
)
```

### 3. Flink Checkpoint 配置

```java
// 启用 Checkpoint
env.enableCheckpointing(60000);  // 60 秒

// 设置 Checkpoint 模式
env.getCheckpointConfig().setCheckpointingMode(CheckpointingMode.EXACTLY_ONCE);

// 设置 Checkpoint 超时时间
env.getCheckpointConfig().setCheckpointTimeout(300000);  // 5 分钟

// 设置 Checkpoint 失败容忍次数
env.getCheckpointConfig().setTolerableCheckpointFailureNumber(3);

// 设置 Checkpoint 清理策略
env.getCheckpointConfig().setExternalizedCheckpointCleanup(
    CheckpointConfig.ExternalizedCheckpointCleanup.RETAIN_ON_CANCELLATION
);
```

## 五、注意事项

### 优点

1. **实现简单**: 无需额外配置，天然支持精准一次性
2. **性能高效**: 批量读取，吞吐量高
3. **数据一致性强**: 快照读取，幂等性保证
4. **无重复风险**: 不会产生重复数据

### 限制

1. **不支持实时增量读取**: 无法捕获 CDC 变更
2. **需要重启作业**: 读取新数据需要重启作业
3. **不适合高频更新**: 适合批量 ETL，不适合实时同步
4. **有界流**: 读取完成后 Source 结束

### 适用场景

**适合**:
- ✅ 批量 ETL 作业
- ✅ 定时调度作业
- ✅ 数据分层处理（ODS → DWD → DWS）
- ✅ 离线数据分析

**不适合**:
- ❌ 实时 CDC 同步
- ❌ 高频数据更新
- ❌ 无界流处理

## 六、如果需要实时增量读取

### 方案 1: 使用 Flink CDC Connector（推荐）

```xml
<dependency>
    <groupId>com.ververica</groupId>
    <artifactId>flink-connector-mysql-cdc</artifactId>
    <version>2.4.2</version>
</dependency>
```

```java
// 使用 MySQL CDC 读取 Doris（Doris 兼容 MySQL 协议）
MySqlSource<String> mySqlSource = MySqlSource.<String>builder()
    .hostname("127.0.0.1")
    .port(9030)
    .databaseList("crypto_dw")
    .tableList("crypto_dw.ods_crypto_ticker_rt")
    .username("root")
    .password("")
    .deserializer(new JsonDebeziumDeserializationSchema())
    .build();
```

### 方案 2: 使用时间戳增量读取

```sql
-- 每次只读取新增数据
SELECT * FROM ods_crypto_ticker_rt
WHERE ingest_time > ${last_checkpoint_time}
ORDER BY ingest_time
LIMIT 10000
```

**实现方式**:
1. 在 Checkpoint 中保存最后读取的时间戳
2. 恢复时从该时间戳继续读取
3. 使用 LIMIT 控制每批读取的数据量

### 方案 3: 使用 Kafka 作为中间层

```
Doris → Kafka → Flink → Doris
  ↑                        ↓
  └────── CDC 捕获 ─────────┘
```

**架构**:
1. 使用 Debezium 或 Canal 捕获 Doris 变更
2. 将变更发送到 Kafka
3. Flink 从 Kafka 消费数据
4. 处理后写回 Doris

## 七、总结

### Doris Source 精准一次性保障

| 保障机制 | 说明 | 配置要求 |
|---------|------|---------|
