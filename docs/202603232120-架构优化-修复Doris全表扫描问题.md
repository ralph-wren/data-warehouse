# 架构优化 - 修复 Doris 全表扫描问题

**日期**: 2026-03-23 21:20  
**问题**: DWD/DWS 作业从 Doris 读取数据导致全表扫描，性能差  
**结论**: 应该改为所有作业都从 Kafka 读取，实时流式处理

---

## 一、问题分析

### 当前架构（错误）❌

```
Kafka → ODS作业 → Doris ODS → DWD作业 → Doris DWD → DWS作业 → Doris DWS
              ↓                    ↓                    ↓
           实时流              全表扫描❌            全表扫描❌
```

**数据流**:
1. **ODS 作业**: Kafka → Doris ODS（实时流，正确✅）
2. **DWD 作业**: Doris ODS → Doris DWD（批量读取，错误❌）
3. **DWS 作业**: Doris DWD → Doris DWS（批量读取，错误❌）

**问题**:
1. **全表扫描**: DWD/DWS 作业每次启动都要扫描全表
2. **性能差**: 数据量大时，启动时间长（可能几小时）
3. **资源浪费**: 重复读取已处理的数据
4. **不是实时**: 无法实时处理新数据，需要重启作业
5. **Checkpoint 无意义**: 恢复时仍然要全表扫描

### 正确架构（推荐）✅

```
                    ┌─→ ODS作业 → Doris ODS
                    │
Kafka (crypto_ticker)─→ DWD作业 → Doris DWD
                    │
                    └─→ DWS作业 → Doris DWS
```

**数据流**:
1. **ODS 作业**: Kafka → Doris ODS（实时流）
2. **DWD 作业**: Kafka → Doris DWD（实时流，直接从 Kafka 读取）
3. **DWS 作业**: Kafka → Doris DWS（实时流，直接从 Kafka 读取）

**优点**:
- ✅ **实时处理**: 所有作业都是实时流式处理
- ✅ **无全表扫描**: 只处理新数据
- ✅ **Checkpoint 有效**: 恢复时从 Kafka Offset 继续
- ✅ **性能高**: 启动快，资源占用少
- ✅ **精准一次性**: Kafka Offset + Doris Label 保证

## 二、架构对比

### 方案 1: 批量读取 Doris（当前方案，不推荐）❌

**优点**:
- 数据分层清晰（ODS → DWD → DWS）
- 可以利用 Doris 的计算能力

**缺点**:
- ❌ 全表扫描，性能差
- ❌ 不是实时处理
- ❌ Checkpoint 恢复慢
- ❌ 资源浪费

**适用场景**:
- 离线批量 ETL
- 定时调度作业（每天一次）
- 数据量小（< 100万行）

### 方案 2: 流式读取 Kafka（推荐）✅

**优点**:
- ✅ 实时流式处理
- ✅ 只处理新数据
- ✅ Checkpoint 恢复快
- ✅ 精准一次性保障

**缺点**:
- 需要在 Flink 中实现数据清洗和聚合逻辑
- 无法利用 Doris 的计算能力

**适用场景**:
- 实时数据处理
- 流式 ETL
- 数据量大（> 100万行）
- 需要低延迟

### 方案 3: 混合架构（折中方案）⚠️

```
Kafka → ODS作业 → Doris ODS
   ↓
   ├─→ DWD作业 → Doris DWD（从 Kafka 读取）
   │
   └─→ DWS作业 → Doris DWS（从 Kafka 读取）
```

**说明**:
- ODS 作业：Kafka → Doris ODS（保留，用于数据归档）
- DWD 作业：Kafka → Doris DWD（改为从 Kafka 读取）
- DWS 作业：Kafka → Doris DWS（改为从 Kafka 读取）

**优点**:
- ✅ 实时流式处理
- ✅ Doris ODS 作为数据归档
- ✅ 可以从 Doris ODS 补数据（离线场景）

## 三、推荐方案：流式读取 Kafka

### 架构图

```
┌─────────────────────────────────────────────────────────┐
│                  实时流式处理架构                          │
├─────────────────────────────────────────────────────────┤
│                                                          │
│                    Kafka Topic                           │
│                  (crypto_ticker)                         │
│                         │                                │
│         ┌───────────────┼───────────────┐               │
│         │               │               │               │
│         ▼               ▼               ▼               │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐         │
│   │   ODS   │     │   DWD   │     │   DWS   │         │
│   │  作业   │     │  作业   │     │  作业   │         │
│   └────┬────┘     └────┬────┘     └────┬────┘         │
│        │               │               │               │
│        ▼               ▼               ▼               │
│   ┌─────────┐     ┌─────────┐     ┌─────────┐         │
│   │Doris ODS│     │Doris DWD│     │Doris DWS│         │
│   └─────────┘     └─────────┘     └─────────┘         │
│                                                          │
│   原始数据         清洗数据         聚合数据              │
│   (归档)          (明细)          (指标)                │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

### 数据流转

1. **Kafka Topic**: 存储原始 Ticker 数据
2. **ODS 作业**: 读取 Kafka → 写入 Doris ODS（数据归档）
3. **DWD 作业**: 读取 Kafka → 清洗转换 → 写入 Doris DWD
4. **DWS 作业**: 读取 Kafka → 窗口聚合 → 写入 Doris DWS

### 关键特性

- **并行处理**: 三个作业并行运行，互不影响
- **实时性**: 所有作业都是实时流式处理
- **精准一次性**: Kafka Offset + Doris Label 保证
- **可扩展**: 可以添加更多作业（如 ADS 作业）

| 作业 | Consumer Group ID | 说明 |
|-----|------------------|------|
| ODS 作业 | `flink-ods-consumer` | 原始数据归档 |

| ADS 作业 | `flink-ads-consumer` | 实时指标 |

**重要**: 每个作业使用独立的 Consumer Group，互不影响。

## 六、性能对比

### 场景：处理 1000 万条数据

| 指标 | 批量读取 Doris | 流式读取 Kafka |
|-----|---------------|---------------|
| 启动时间 | 30-60 分钟 | 5-10 秒 |
| 处理延迟 | 不适用（批量） | < 1 秒 |
| Checkpoint 恢复 | 30-60 分钟 | 5-10 秒 |
| 资源占用 | 高（全表扫描） | 低（增量处理） |
| 实时性 | 无 | 实时 |
| 精准一次性 | 支持 | 支持 |

### 场景：作业故障恢复

**批量读取 Doris**:
```
故障发生 → 从 Checkpoint 恢复 → 重新扫描全表（30分钟）→ 继续处理
```

**流式读取 Kafka**:
```
故障发生 → 从 Checkpoint 恢复 → 从 Kafka Offset 继续（5秒）→ 继续处理
```

## 七、注意事项

### 1. Kafka 数据保留

**问题**: Kafka 数据有保留期限（默认 7 天）

**解决方案**:
```yaml
# 配置 Kafka Topic 保留时间
kafka:
  topic:
    crypto-ticker:
      retention.ms: 604800000  # 7 天
      retention.bytes: 107374182400  # 100 GB
```

**建议**:
- 开发环境：7 天
- 生产环境：30 天（根据需求调整）

### 2. 数据补录

**场景**: 需要重新处理历史数据

**方案 1**: 从 Kafka 重新消费
```bash
# 重置 Consumer Group Offset 到最早
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --group flink-dwd-consumer \
  --reset-offsets --to-earliest \
  --topic crypto_ticker \
  --execute
```

**方案 2**: 从 Doris ODS 补录（离线批量）
```java
// 创建一个临时的批量作业
// 从 Doris ODS 读取历史数据 → 写入 Doris DWD
```

### 3. 数据一致性

**问题**: ODS/DWD/DWS 三个作业并行运行，数据可能不一致

**解决方案**:
- 使用相同的 Kafka Offset（从同一个 Topic 读取）
- 使用 Watermark 保证事件时间一致
- 使用 Checkpoint 保证故障恢复一致

### 4. 资源规划

**并行运行三个作业**:
- ODS 作业：2 个 TaskManager（简单写入）
- DWD 作业：4 个 TaskManager（数据清洗）
- DWS 作业：4 个 TaskManager（窗口聚合）

**总计**: 10 个 TaskManager，每个 2 核 4GB，共 20 核 40GB
