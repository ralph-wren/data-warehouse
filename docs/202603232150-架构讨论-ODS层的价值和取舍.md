# 架构讨论 - ODS 层的价值和取舍

**日期**: 2026-03-23 21:50  
**问题**: 既然所有作业都从 Kafka 读取，ODS 层还有必要吗？  
**结论**: 取决于具体需求，可以保留也可以去掉

---

## 一、当前架构

### 现状

```
                    ┌─→ ODS作业 → Doris ODS (Kafka 原始数据副本)
                    │
Kafka (crypto_ticker)─→ DWD作业 → Doris DWD (清洗后的数据)
                    │
                    └─→ DWS作业 → Doris DWS (聚合后的数据)
```

**问题**: ODS 层只是 Kafka 数据的副本，似乎没有太大价值？

---

## 二、ODS 层的价值分析

### 优点 ✅

#### 1. 数据归档和长期存储

**Kafka 的局限**:
- Kafka 数据有保留期限（默认 7 天，最多 30 天）
- 超过保留期限的数据会被自动删除
- 无法查询历史数据

**ODS 的价值**:
- 永久保存原始数据
- 可以查询任意时间段的历史数据
- 支持数据回溯和审计

**示例**:
```sql
-- 查询 3 个月前的数据（Kafka 已删除，但 ODS 还有）
SELECT * FROM ods_crypto_ticker_rt
WHERE ingest_time BETWEEN '2025-12-23' AND '2025-12-24';
```

#### 2. 数据补录和重新处理

**场景**: DWD/DWS 作业出现 bug，需要重新处理历史数据

**使用 Kafka**:
- ❌ 如果数据已过期，无法重新处理
- ❌ 需要重置 Consumer Group Offset（可能影响其他作业）

**使用 ODS**:
- ✅ 可以从 ODS 读取历史数据重新处理
- ✅ 不影响 Kafka 和其他作业

**示例**:
```java
// 创建临时作业，从 ODS 读取历史数据重新处理
String odsSourceDDL = tableFactory.createDorisSourceTable(
    "ods_source",
    "ods",
    TableSchemas.DORIS_ODS_SOURCE_SCHEMA
);

// 重新处理 2025-12-23 的数据
String sql = "INSERT INTO dwd_sink SELECT ... FROM ods_source " +
             "WHERE DATE(FROM_UNIXTIME(timestamp/1000)) = '2025-12-23'";
```

#### 3. 数据质量检查和对账

**场景**: 需要对比原始数据和处理后的数据

**使用 ODS**:
```sql
-- 对账：检查 ODS 和 DWD 的数据量是否一致
SELECT 
    DATE(FROM_UNIXTIME(timestamp/1000)) as date,
    COUNT(*) as ods_count
FROM ods_crypto_ticker_rt
GROUP BY date;

SELECT 
    trade_date,
    COUNT(*) as dwd_count
FROM dwd_crypto_ticker_detail
GROUP BY trade_date;

-- 如果数量不一致，说明 DWD 作业有问题
```

#### 4. 数据探索和分析

**场景**: 数据分析师需要查看原始数据

**使用 ODS**:
```sql
-- 查看某个交易对的原始数据
SELECT * FROM ods_crypto_ticker_rt
WHERE inst_id = 'BTC-USDT'
ORDER BY timestamp DESC
LIMIT 100;

-- 分析数据分布
SELECT 
    inst_id,
    COUNT(*) as count,
    MIN(last_price) as min_price,
    MAX(last_price) as max_price
FROM ods_crypto_ticker_rt
GROUP BY inst_id;
```

#### 5. 灾难恢复

**场景**: Kafka 集群故障，数据丢失

**使用 ODS**:
- ✅ ODS 作为数据备份
- ✅ 可以从 ODS 恢复数据到 Kafka
- ✅ 保证数据不丢失

### 缺点 ❌

#### 1. 存储成本

**问题**: ODS 层存储了大量原始数据

**成本**:
- 假设每天 1000 万条数据
- 每条数据约 500 字节
- 每天约 5 GB
- 每年约 1.8 TB

**解决方案**:
- 使用 Doris 的数据压缩（压缩比约 5:1）
- 定期清理过期数据（如保留 1 年）
- 使用冷热分离存储

#### 2. 维护成本

**问题**: 需要维护额外的 ODS 作业和表

**成本**:
- 需要监控 ODS 作业的运行状态
- 需要处理 ODS 作业的故障
- 需要管理 ODS 表的分区和索引

#### 3. 数据冗余

**问题**: ODS 数据与 Kafka 数据重复

**影响**:
- 在 Kafka 保留期内，数据是重复的
- 浪费存储空间

## 三、是否需要 ODS 层？

### 方案 1: 保留 ODS 层 ✅ (推荐)

**适用场景**:
- 需要长期保存原始数据（> 30 天）
- 需要数据回溯和审计
- 需要数据补录和重新处理
- 数据质量要求高，需要对账

**架构**:
```
                    ┌─→ ODS作业 → Doris ODS (数据归档)
                    │
Kafka (crypto_ticker)─→ DWD作业 → Doris DWD (数据清洗)
                    │
                    └─→ DWS作业 → Doris DWS (数据聚合)
```

**优点**:
- ✅ 数据永久保存
- ✅ 支持数据回溯
- ✅ 支持数据补录
- ✅ 支持数据对账

**缺点**:
- ❌ 存储成本高
- ❌ 维护成本高

### 方案 2: 去掉 ODS 层 ⚠️

**适用场景**:
- 只需要短期数据（< 30 天）
- 不需要数据回溯
- 不需要数据补录
- 存储成本敏感

**架构**:
```
Kafka (crypto_ticker) ─→ DWD作业 → Doris DWD (数据清洗)
                     │
                     └─→ DWS作业 → Doris DWS (数据聚合)
```

**优点**:
- ✅ 架构简单
- ✅ 存储成本低
- ✅ 维护成本低

**缺点**:
- ❌ 无法查询历史数据（> Kafka 保留期）
- ❌ 无法数据补录
- ❌ 无法数据对账

### 方案 3: 混合方案 ⭐ (折中)

**适用场景**:
- 需要长期保存部分数据
- 存储成本敏感

**架构**:
```
                    ┌─→ ODS作业 → Doris ODS (采样归档，如每分钟一条)
                    │
Kafka (crypto_ticker)─→ DWD作业 → Doris DWD (全量数据)
                    │
                    └─→ DWS作业 → Doris DWS (聚合数据)
```

**ODS 采样策略**:
```java
// 只保存每分钟的第一条数据
dataStream
    .keyBy(data -> data.getInstId())
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .reduce((value1, value2) -> value1)  // 只保留第一条
    .sinkTo(odsSink);
```

**优点**:
- ✅ 存储成本降低 99%（每分钟 1 条 vs 每秒 1 条）
- ✅ 仍然可以查询历史趋势
- ✅ 维护成本低

**缺点**:
- ❌ 无法查询完整的历史数据
- ❌ 数据补录精度降低

## 四、推荐方案

### 生产环境推荐：保留 ODS 层 ✅

**理由**:
1. **数据是资产**: 原始数据是最宝贵的资产，应该永久保存
2. **合规要求**: 金融数据通常需要保存 3-7 年
3. **数据回溯**: 业务需求变化时，可以重新处理历史数据
4. **数据对账**: 保证数据质量，及时发现问题

**优化建议**:
1. **数据压缩**: 使用 Doris 的 ZSTD 压缩（压缩比 5:1）
2. **冷热分离**: 
   - 热数据（最近 30 天）：SSD 存储
   - 冷数据（30 天以上）：HDD 存储
3. **定期清理**: 保留 1-3 年的数据，超过后归档到对象存储
4. **采样存储**: 对于高频数据，可以采样存储

### 开发/测试环境：去掉 ODS 层 ⚠️

**理由**:
1. 开发环境不需要长期保存数据
2. 降低存储和维护成本
3. 简化架构，加快开发速度

**架构**:
```
Kafka (7天保留) ─→ DWD作业 → Doris DWD
                │
                └─→ DWS作业 → Doris DWS
```

## 五、ODS 层的优化建议

### 1. 数据压缩

```sql
-- 创建 ODS 表时启用压缩
CREATE TABLE ods_crypto_ticker_rt (
    -- 字段定义
) DUPLICATE KEY(inst_id, `timestamp`)
DISTRIBUTED BY HASH(inst_id) BUCKETS 10
PROPERTIES (
    "compression" = "ZSTD",  -- 使用 ZSTD 压缩
    "replication_num" = "1"  -- 单副本（开发环境）
);
```

### 2. 分区管理

```sql
-- 按天分区
CREATE TABLE ods_crypto_ticker_rt (
    -- 字段定义
) DUPLICATE KEY(inst_id, `timestamp`)
PARTITION BY RANGE(`timestamp`) (
    -- 动态分区，自动创建
)
DISTRIBUTED BY HASH(inst_id) BUCKETS 10
PROPERTIES (
    "dynamic_partition.enable" = "true",
    "dynamic_partition.time_unit" = "DAY",
    "dynamic_partition.start" = "-30",  -- 保留 30 天
    "dynamic_partition.end" = "3",      -- 提前创建 3 天
    "dynamic_partition.prefix" = "p",
    "dynamic_partition.buckets" = "10"
);
```

### 3. 定期清理

```sql
-- 删除 90 天前的分区
ALTER TABLE ods_crypto_ticker_rt 
DROP PARTITION p20251223;

-- 或者使用动态分区自动清理
ALTER TABLE ods_crypto_ticker_rt 
SET ("dynamic_partition.start" = "-90");  -- 只保留 90 天
```

### 4. 采样存储

```java
// 只保存每分钟的第一条数据
dataStream
    .keyBy(data -> data.getInstId())
    .window(TumblingEventTimeWindows.of(Time.minutes(1)))
    .reduce((value1, value2) -> value1)
    .sinkTo(odsSink);

// 存储成本降低 99%
```

**生产环境**: ✅ 保留 ODS 层

- 支持数据回溯和补录
