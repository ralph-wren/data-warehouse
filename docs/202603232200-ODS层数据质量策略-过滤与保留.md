# ODS 层数据质量策略 - 过滤与保留

**日期**: 2026-03-23 22:00  
**问题**: ODS 层的数据是全部保留吗？会不会做一些过滤？过滤掉的数据怎么办？  
**结论**: ODS 层应该保留所有原始数据，但可以标记异常数据

---

## 一、ODS 层的设计原则

### 核心原则：保留所有原始数据 ⭐⭐⭐⭐⭐

**理由**:
1. **ODS = Operational Data Store（操作数据存储）**
   - ODS 层的职责是**原样保存**原始数据
   - 不做任何业务逻辑处理
   - 不做任何数据过滤

2. **数据是资产**
   - 今天看起来是"脏数据"，明天可能是有价值的数据
   - 过滤规则可能会变化
   - 一旦过滤，数据永久丢失

3. **可追溯性**
   - 需要知道系统接收到了什么数据
   - 需要分析数据质量问题的根源
   - 需要审计和合规

### 错误的做法 ❌

```java
// ❌ 错误：在 ODS 层过滤数据
dataStream
    .filter(data -> data.getLastPrice() > 0)  // 过滤掉价格为 0 的数据
    .filter(data -> data.getBidPrice() > 0)   // 过滤掉买价为 0 的数据
    .sinkTo(odsSink);

// 问题：
// 1. 过滤掉的数据永久丢失
// 2. 无法分析为什么会有价格为 0 的数据
// 3. 如果过滤规则错误，无法恢复
```

### 正确的做法 ✅

```java
// ✅ 正确：ODS 层保存所有原始数据
dataStream
    .sinkTo(odsSink);  // 不做任何过滤

// 数据质量检查在 DWD 层进行
```

## 三、ODS 层的数据质量标记（推荐）⭐

### 方案：保留所有数据 + 添加质量标记

**设计思路**:
- 保存所有原始数据（不过滤）
- 添加数据质量标记字段
- DWD 层根据质量标记过滤

### 2. ODS 作业添加质量检查

```java
public static class ODSTransformFunction implements MapFunction<String, String> {
    
    private final ObjectMapper objectMapper = new ObjectMapper();
    
    @Override
    public String map(String value) throws Exception {
        try {
            // 解析 Ticker 数据
            TickerData ticker = objectMapper.readValue(value, TickerData.class);
            
            // 数据质量检查
            String dataQuality = "GOOD";
            String qualityReason = "";
            boolean isValid = true;
            
            // 检查 1: 价格是否有效
            if (ticker.getLast() <= 0) {
                dataQuality = "BAD";
                qualityReason = "last_price_invalid";
                isValid = false;
            }
            
            // 检查 2: 买卖价是否有效
            if (ticker.getBidPx() <= 0 || ticker.getAskPx() <= 0) {
                dataQuality = "BAD";
                qualityReason = qualityReason.isEmpty() ? "bid_ask_price_invalid" 
                    : qualityReason + ",bid_ask_price_invalid";
                isValid = false;
            }
            
            // 检查 3: 时间戳是否有效
            long currentTime = System.currentTimeMillis();
            if (Math.abs(ticker.getTs() - currentTime) > 3600000) {  // 超过 1 小时
                dataQuality = "BAD";
                qualityReason = qualityReason.isEmpty() ? "timestamp_invalid" 
                    : qualityReason + ",timestamp_invalid";
                isValid = false;
            }
            
            // 构建 Doris JSON 格式（包含质量标记）
            StringBuilder json = new StringBuilder();
            json.append("{");
            json.append("\"inst_id\":\"").append(ticker.getInstId()).append("\",");
            json.append("\"timestamp\":").append(ticker.getTs()).append(",");
            json.append("\"last_price\":").append(ticker.getLast()).append(",");
            json.append("\"bid_price\":").append(ticker.getBidPx()).append(",");
            json.append("\"ask_price\":").append(ticker.getAskPx()).append(",");
            json.append("\"bid_size\":").append(ticker.getBidSz()).append(",");
            json.append("\"ask_size\":").append(ticker.getAskSz()).append(",");
            json.append("\"volume_24h\":").append(ticker.getVol24h()).append(",");
            json.append("\"high_24h\":").append(ticker.getHigh24h()).append(",");
            json.append("\"low_24h\":").append(ticker.getLow24h()).append(",");
            json.append("\"open_24h\":").append(ticker.getOpen24h()).append(",");
            json.append("\"data_source\":\"OKX\",");
            json.append("\"ingest_time\":").append(System.currentTimeMillis()).append(",");
            // 添加质量标记
            json.append("\"data_quality\":\"").append(dataQuality).append("\",");
            json.append("\"quality_reason\":\"").append(qualityReason).append("\",");
            json.append("\"is_valid\":").append(isValid);
            json.append("}");
            
            return json.toString();
            
        } catch (Exception e) {
            logger.error("Failed to transform data: {}", e.getMessage());
            // 即使解析失败，也保存原始数据
            return buildErrorRecord(value, e.getMessage());
        }
    }
    
    // 构建错误记录
    private String buildErrorRecord(String rawData, String errorMsg) {
        return "{" +
            "\"inst_id\":\"UNKNOWN\"," +
            "\"timestamp\":" + System.currentTimeMillis() + "," +
            "\"last_price\":0," +
            "\"bid_price\":0," +
            "\"ask_price\":0," +
            "\"bid_size\":0," +
            "\"ask_size\":0," +
            "\"volume_24h\":0," +
            "\"high_24h\":0," +
            "\"low_24h\":0," +
            "\"open_24h\":0," +
            "\"data_source\":\"OKX\"," +
            "\"ingest_time\":" + System.currentTimeMillis() + "," +
            "\"data_quality\":\"BAD\"," +
            "\"quality_reason\":\"parse_error:" + errorMsg + "\"," +
            "\"is_valid\":false" +
            "}";
    }
}
```

## 四、异常数据的处理策略

### 策略 1: 保存到 ODS + 质量标记（推荐）⭐⭐⭐

**流程**:
```
原始数据 → 质量检查 → ODS（全部保存 + 质量标记）
                    ↓
                DWD（只处理 is_valid=true 的数据）
```

**优点**:
- ✅ 所有数据都保存，不丢失
- ✅ 可以分析异常数据的原因
- ✅ 可以统计数据质量指标
- ✅ 如果过滤规则错误，可以重新处理

**查询示例**:
```sql
-- 查询所有数据
SELECT * FROM ods_crypto_ticker_rt;

-- 只查询有效数据
SELECT * FROM ods_crypto_ticker_rt WHERE is_valid = true;

-- 只查询异常数据
SELECT * FROM ods_crypto_ticker_rt WHERE is_valid = false;

-- 统计数据质量
SELECT 
    data_quality,
    quality_reason,
    COUNT(*) as count
FROM ods_crypto_ticker_rt
WHERE DATE(FROM_UNIXTIME(timestamp/1000)) = CURDATE()
GROUP BY data_quality, quality_reason;
```

### 策略 2: 分表存储（备选）⭐⭐

**流程**:
```
原始数据 → 质量检查 → 有效数据 → ods_crypto_ticker_rt
                    ↓
                异常数据 → ods_crypto_ticker_error
```

**表结构**:
```sql
-- 正常数据表
CREATE TABLE ods_crypto_ticker_rt (
    -- 正常字段
);

-- 异常数据表
CREATE TABLE ods_crypto_ticker_error (
    -- 正常字段
    error_type STRING COMMENT '错误类型',
    error_reason STRING COMMENT '错误原因',
    raw_data STRING COMMENT '原始数据'
);
```

**优点**:
- ✅ 正常数据和异常数据分开存储
- ✅ 查询正常数据时性能更好
- ✅ 可以单独分析异常数据

**缺点**:
- ❌ 需要维护两张表
- ❌ 数据分散，不便于统一分析

### 策略 3: 只保存有效数据（不推荐）❌

**流程**:
```
原始数据 → 质量检查 → 有效数据 → ODS
                    ↓
                异常数据 → 丢弃（或写入日志）
```

**缺点**:
- ❌ 异常数据永久丢失
- ❌ 无法分析数据质量问题
- ❌ 无法追溯数据来源

## 六、最佳实践

### 1. ODS 层设计原则

✅ **应该做的**:
- 保存所有原始数据（不过滤）
- 添加数据质量标记
- 添加数据来源标记
- 添加接收时间戳

❌ **不应该做的**:
- 过滤数据
- 修改原始数据
- 做业务逻辑处理
- 做数据转换

### 2. 数据质量检查规则

```java
// 基本检查
- 价格 > 0
- 时间戳在合理范围内（当前时间 ± 1 小时）
- 必填字段不为空

// 业务检查
- 买价 < 卖价（正常情况）
- 价格在合理范围内（如 BTC 价格在 10000-100000 之间）
- 成交量 > 0
```

### 3. 异常数据处理流程

```
1. 接收原始数据
2. 进行质量检查
3. 添加质量标记
4. 保存到 ODS（全部保存）
5. DWD 层根据质量标记过滤
6. 定期分析异常数据
7. 优化数据源或过滤规则
```

## 七、实际案例

### 案例 1: 价格异常数据

**场景**: 接收到价格为 0 的数据

**处理**:
```
1. ODS 层保存（is_valid=false, quality_reason="last_price_invalid"）
2. DWD 层过滤掉
3. 分析发现：交易所 API 在维护期间返回 0
4. 优化：添加交易所状态检查
```

**如果过滤掉**:
- ❌ 无法发现问题
- ❌ 无法统计异常数据比例
- ❌ 无法优化数据源

### 案例 2: 时间戳异常

**场景**: 接收到未来时间的数据

**处理**:
```
1. ODS 层保存（is_valid=false, quality_reason="timestamp_invalid"）
2. 分析发现：服务器时间不同步
3. 修复：同步服务器时间
4. 重新处理：从 ODS 读取数据，重新标记质量
```

## 八、总结

### ODS 层数据策略

| 策略 | 是否过滤 | 质量标记 | 推荐度 |
|-----|---------|---------|--------|
| 保存所有 + 质量标记 | ❌ | ✅ | ⭐⭐⭐⭐⭐ |
| 分表存储 | ❌ | ✅ | ⭐⭐⭐ |
| 只保存有效数据 | ✅ | ❌ | ⭐ |

### 推荐方案

**ODS 层**: 保存所有原始数据 + 添加质量标记
- 不过滤任何数据
- 添加 `data_quality`、`quality_reason`、`is_valid` 字段
- 所有数据都保存，便于追溯和分析

**DWD 层**: 根据质量标记过滤
- 只处理 `is_valid = true` 的数据
- 过滤掉异常数据
- 进行数据清洗和转换

**监控**: 实时监控数据质量
- 统计异常数据比例
- 分析异常原因
- 优化数据源和过滤规则

### 核心原则

**数据是资产，不要轻易丢弃！**
