# 从配置文件读取状态后端并注入运行时

## 1. 问题

用户希望修改代码，让 `FlinkEnvironmentFactory` 能真正从配置文件读取状态后端配置，而不是只在 YAML 里声明、运行时却没有生效。

## 2. 原因分析

修改前存在两个问题：

1. `application-*.yml` 中虽然已经配置了：
   - `flink.state.backend`
   - `flink.state.checkpoint-dir`
   - `flink.state.savepoint-dir`
2. 但是 `FlinkEnvironmentFactory` 只做了：
   - 创建 `StreamExecutionEnvironment`
   - 设置并行度
   - 开启 Checkpoint
   - 配置 Web UI 和 Metrics

也就是说，**配置文件中的状态后端参数并没有真正注入到 Flink 的 `Configuration` 中**。

## 3. 解决方案

本次改动采用“配置驱动”方案，而不是在代码里写死某个状态后端实现类。

这样做的好处：

1. `DataStream` 和 `Table` 两种环境共用一套配置逻辑
2. 不需要额外在业务代码里写死 `EmbeddedRocksDBStateBackend`
3. 可以通过不同环境配置文件切换状态后端
4. 更适合后续接入 StreamPark / Flink 集群统一配置

## 4. 具体修改

### 4.1 修改 `FlinkEnvironmentFactory`

文件：

- `src/main/java/com/crypto/dw/flink/factory/FlinkEnvironmentFactory.java`

新增了 `configureStateBackend()` 方法，负责读取并注入以下配置：

1. `flink.state.backend` -> `state.backend.type`
2. `flink.state.checkpoint-dir` -> `state.checkpoints.dir`
3. `flink.state.savepoint-dir` -> `state.savepoints.dir`
4. `flink.state.incremental` -> `state.backend.incremental`
5. `flink.state.local-recovery` -> `state.backend.local-recovery`

同时在 `createFlinkConfiguration()` 中调用该方法，保证所有通过工厂创建的 Flink 环境都统一应用状态配置。

### 4.2 兼容旧配置值

新增了 `normalizeStateBackend()` 方法。

原因：

旧版本配置里有时会写 `filesystem`，但在 Flink 1.17 的推荐写法中，更常见的是 `hashmap` 和 `rocksdb`。

处理方式：

- 如果配置为 `filesystem`
- 代码会自动映射为 `hashmap`
- 并输出警告日志，提示这是兼容旧写法

### 4.3 补充环境配置文件

修改了以下配置文件：

- `src/main/resources/config/application-dev.yml`
- `src/main/resources/config/application-docker.yml`
- `src/main/resources/config/application-prod.yml`

补充或明确了：

- `backend`
- `checkpoint-dir`
- `savepoint-dir`
- `incremental`
- `local-recovery`

这样不同环境下的状态后端行为更清晰。

### 4.4 增强配置自检输出

修改文件：

- `src/main/java/com/crypto/dw/ConfigTest.java`

新增日志输出：

- `State Backend`
- `Checkpoint Dir`
- `Savepoint Dir`
- `Incremental`
- `Local Recovery`

后续运行 `ConfigTest` 时，可以直接确认 YAML 是否被正确读取。

## 5. 当前效果

修改完成后：

1. `FlinkEnvironmentFactory` 会真正读取配置文件中的状态后端参数
2. `rocksdb` / `hashmap` 等配置会进入 Flink Runtime `Configuration`
3. checkpoint/savepoint 路径也会同步注入运行时
4. 不再出现“配置文件写了，但工厂代码没有生效”的问题
