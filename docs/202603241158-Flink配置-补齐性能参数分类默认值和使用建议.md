# Flink 配置补齐：性能参数分类、默认值和使用建议

## 1. 本次目标

这次修改的目标不是只补一个 `state.backend`，而是把 `FlinkEnvironmentFactory` 里和运行性能、稳定性、恢复能力直接相关的参数一起收拢，做到：

1. 配置按功能分类
2. 参数有默认值
3. 参数有注释
4. 作业统一走工厂配置
5. 初学者可以知道哪些参数值得优先调

## 3. 参数分类说明

### 3.1 execution：执行层参数

这组参数决定作业的基本运行方式和延迟/吞吐折中。

#### 已补充参数

- `flink.execution.runtime-mode`
- `flink.execution.parallelism`
- `flink.execution.max-parallelism`
- `flink.execution.buffer-timeout-ms`
- `flink.execution.auto-watermark-interval-ms`

#### 默认建议

- `runtime-mode: STREAMING`
- `parallelism: dev/docker=4, prod=8`
- `max-parallelism: dev/docker=128, prod=256`
- `buffer-timeout-ms: 100`
- `auto-watermark-interval-ms: 200`

#### 使用建议

1. **优先调 `parallelism`**
   - 吞吐不足时先加它
   - 但前提是 Kafka 分区数、TaskManager slot 数也要够

2. **`max-parallelism` 尽量早定**
   - 它影响 keyed state 的逻辑分片
   - 后续改动会增加状态恢复复杂度

3. **`buffer-timeout-ms` 决定延迟/吞吐折中**
   - 更低：更实时，但吞吐下降、网络 flush 更频繁
   - 更高：吞吐更好，但端到端延迟更高
   - 一般先用 `100ms`

4. **事件时间作业建议保留 `auto-watermark-interval-ms`**
   - 你的作业涉及 watermark，建议保留 `200ms`

### 3.2 checkpoint：一致性与恢复点参数

这组参数决定作业的容错能力、快照频率和状态存储压力。

#### 已补充参数

- `flink.checkpoint.interval`
- `flink.checkpoint.mode`
- `flink.checkpoint.timeout`
- `flink.checkpoint.min-pause`
- `flink.checkpoint.max-concurrent`
- `flink.checkpoint.tolerable-failed-checkpoints`
- `flink.checkpoint.externalized-retention`
- `flink.checkpoint.snapshot-compression`
- `flink.checkpoint.checkpoints-after-tasks-finish`
- `flink.checkpoint.unaligned.enabled`
- `flink.checkpoint.unaligned.aligned-timeout-ms`
- `flink.checkpoint.unaligned.max-subtasks-per-channel-state-file`

#### 默认建议

开发环境：

- `interval: 60000`
- `mode: EXACTLY_ONCE`
- `timeout: 300000`
- `min-pause: 10000`
- `max-concurrent: 1`
- `tolerable-failed-checkpoints: 3`
- `externalized-retention: DELETE_ON_CANCELLATION`
- `snapshot-compression: true`
- `unaligned.enabled: false`

生产环境：

- `interval: 120000`
- `timeout: 600000`
- `min-pause: 30000`
- `externalized-retention: RETAIN_ON_CANCELLATION`
- 其余建议与开发保持一致

#### 使用建议

1. **不要把 `interval` 设得太短**
   - Checkpoint 太频繁会直接吞掉吞吐
   - 先从 `60s` 或 `120s` 开始

2. **`max-concurrent` 一般保持 `1`**
   - 配置简单，问题少，最适合当前项目

3. **`min-pause` 很重要**
   - 它能避免 checkpoint 连续轰炸状态后端
   - 比你之前的 `500ms` 更推荐 `10000ms`

4. **`snapshot-compression` 建议开**
   - 尤其是远程 checkpoint 存储

5. **`unaligned checkpoint` 不建议默认开**
   - 它主要用于严重反压场景
   - 开了以后 channel state 会增多，排障也更复杂
   - 所以本次默认关闭，只在需要时再开

6. **生产环境建议保留 externalized checkpoint**
   - 取消作业后可以继续恢复
   - 但要注意定期清理历史目录

### 3.3 state：状态后端参数

这组参数决定状态存储方式和 checkpoint/savepoint 落盘位置。

#### 已补充参数

- `flink.state.backend`
- `flink.state.checkpoint-dir`
- `flink.state.savepoint-dir`
- `flink.state.incremental`
- `flink.state.local-recovery`

#### 默认建议

- `backend: rocksdb`
- `incremental: true`
- `local-recovery: true`

#### 使用建议

1. **当前项目建议继续用 RocksDB**
   - 你的作业是长期运行的流式作业
   - 后续状态增长后，RocksDB 更稳

2. **`incremental` 建议开启**
   - 尤其是 RocksDB + 远程存储场景
   - 可以显著减少 checkpoint 体积

3. **`local-recovery` 建议开启**
   - 本地重启恢复更快

4. **`checkpoint-dir` 和 `savepoint-dir` 要分环境**
   - dev: 本地文件系统
   - docker: 容器内挂载目录
   - prod: HDFS 或共享存储

### 3.4 restart：重启策略参数

这组参数决定作业失败后的恢复方式。

#### 已补充参数

- `flink.restart.strategy`
- `flink.restart.fixed-delay.attempts`
- `flink.restart.fixed-delay.delay-ms`
- `flink.restart.failure-rate.max-failures-per-interval`
- `flink.restart.failure-rate.failure-rate-interval-ms`
- `flink.restart.failure-rate.delay-ms`

#### 默认建议

开发环境：

- `strategy: fixed-delay`
- `attempts: 3`
- `delay-ms: 10000`

Docker / 生产环境：

- `strategy: failure-rate`
- `max-failures-per-interval: 3`
- `failure-rate-interval-ms: 600000`
- `delay-ms: 30000`

#### 使用建议

1. **开发环境用 `fixed-delay`**
   - 行为最直观
   - 好定位问题

2. **长期运行环境用 `failure-rate`**
   - 更能容忍偶发网络抖动和外部系统短故障

3. **不要一味拉高重试次数**
   - 如果是配置错误或 SQL 错误，再多次重启也没意义
