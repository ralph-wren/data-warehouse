# Flink 修复：空值下游传播、重连退出、端口冲突

## 1. 本次修复的问题

这次主要修复了前一次代码检查中发现的 4 个实际问题：

1. ODS DataStream 转换异常时返回 `null`，但下游不会自动过滤
2. ODS DataStream 人为关闭算子链，影响吞吐
3. WebSocket Source 断线后可能直接退出，不会稳定重连
4. DWS 1 分钟作业和 ODS DataStream 作业占用同一个 Web UI 端口

## 2. 修复内容

### 2.1 修复 ODS 转换空值继续下游的问题

修改文件：

- `src/main/java/com/crypto/dw/flink/FlinkODSJobDataStream.java`

原来的问题：

1. 使用的是 `MapFunction`
2. 解析失败时返回 `null`
3. 注释写“返回 null 会被过滤掉”

但实际上：

**`map()` 不会自动过滤 `null`，`null` 仍然可能继续流向下游，导致 Sink 侧报错。**

修复方式：

1. 改成 `FlatMapFunction`
2. 出现异常时不再返回 `null`
3. 直接丢弃坏数据，不输出到下游

这样坏数据会在转换阶段被拦住，不会污染后续链路。

### 2.2 去掉无必要的 `disableChaining()`

同样修改：

- `src/main/java/com/crypto/dw/flink/FlinkODSJobDataStream.java`

原来的问题：

- `ODS Transform` 显式调用了 `disableChaining()`

影响：

1. 增加算子边界
2. 增加网络传输和序列化开销
3. 降低吞吐

修复方式：

- 删除 `disableChaining()`
- 让 Flink 自动做链优化

这对当前这个简单转换算子更合理。

### 2.3 修复 WebSocket Source 断线后直接退出的问题

修改文件：

- `src/main/java/com/crypto/dw/flink/source/OKXWebSocketSourceFunction.java`

原来的问题：

外层循环是：

```java
while (running.get() && wsClient.isOpen())
```

这会导致：

1. 一旦连接断开
2. `wsClient.isOpen()` 变成 `false`
3. 循环直接退出
4. 后面的重连逻辑可能根本没有机会执行

修复方式：

1. 外层循环改为只看 `running`
2. 在循环内部判断连接状态
3. 如果断开，就在循环内部统一重连
4. 重连时复用配置文件中的：
   - `okx.websocket.reconnect.max-retries`
   - `okx.websocket.reconnect.initial-delay`
   - `okx.websocket.reconnect.max-delay`
5. 每次重连都重新创建 WebSocket 客户端实例，避免复用已关闭对象

这次还顺手把关闭逻辑提取成了公共方法，供 `cancel()` 和异常恢复共用。

### 2.4 修复本地 Web UI 端口冲突

修改文件：

- `src/main/java/com/crypto/dw/flink/FlinkDWSJob1MinSQL.java`

原来的问题：

- `FlinkODSJobDataStream` 使用 `8083`
- `FlinkDWSJob1MinSQL` 也使用 `8083`

如果本地同时运行两个作业，会直接端口冲突。

修复方式：

- 将 `FlinkDWSJob1MinSQL` 改为 `8084`

这样当前本地端口分配变为：

- ODS SQL: `8081`
- DWD SQL: `8082`
- ODS DataStream: `8083`
- DWS 1Min SQL: `8084`
- Data Collector: `8085`
- ADS Realtime Metrics: `8086`

## 3. 顺带优化的实现方式

### 3.1 Doris JSON 构建更稳

之前是手写 `StringBuilder` 拼 JSON。

风险：

1. 数值字段为空时会生成非法 JSON
2. 某些字段格式不对时，下游才报错
3. 维护起来容易漏逗号、漏引号

现在改为：

1. 先把字符串字段解析成正确的 `long` / `BigDecimal`
2. 再用 `ObjectNode` 输出 JSON

这样更稳，也更容易发现坏数据。

### 3.2 转换阶段就校验数值合法性

这次在 ODS 转换里增加了：

1. 必填字段校验
2. Long 解析校验
3. Decimal 解析校验

如果字段值非法，直接丢弃该记录，而不是把错误推迟到 Doris Sink 阶段。

## 4. 当前效果

修复后：

1. ODS 转换失败的脏数据不会再以 `null` 形式流到下游
2. ODS Transform 不再强制断链，吞吐更好
3. WebSocket Source 断线后会在运行循环内稳定重连
4. 本地同时启动 ODS DataStream 和 DWS 1Min 不会再因为 8083 冲突而失败

## 5. 后续建议

### 5.1 建议把 Web UI 端口也做成配置项

这次先修成不冲突的固定值，已经够用。

但更好的长期方案是：

- 每个作业从配置文件读取自己的 Web UI 端口

这样后续新增作业时不需要再去源码里改端口。

### 5.2 建议给脏数据加一个单独输出

当前处理方式是：

- 转换失败直接丢弃

如果你后续想追查脏数据来源，建议再做一版：

1. 主流正常写 Doris
2. 脏数据写日志文件、Kafka 死信队列或单独表
