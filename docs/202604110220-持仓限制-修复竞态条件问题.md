# 持仓限制 - 修复竞态条件问题

## 问题描述

用户配置了 `max-positions: 1`，但仍然看到最多有 3 个币对同时下单。

## 问题分析

### 1. 观察到的现象

从交易日志 `logs/trading_20260411.csv` 中可以看到：
- 大部分订单都失败了（`spotOrderId=null` 或 `swapOrderId=null`）
- 只有一个订单成功：`OL` 币种在 13:29:34 成功开仓
- Redis 中的持仓计数为 0

### 2. 根本原因：Check-Then-Act 竞态条件 ⭐⭐⭐

原来的代码存在经典的 **Check-Then-Act 竞态条件**：

```java
// 原来的代码（有问题）
// 1. 检查持仓数量
String countStr = redisManager.get(POSITION_COUNT_KEY);
int currentPositions = countStr != null ? Integer.parseInt(countStr) : 0;

if (currentPositions >= maxPositions) {
    return;  // 已达上限
}

// 2. 下单（可能失败）
spotOrderId = tradingService.buySpot(...);
swapOrderId = tradingService.shortSwap(...);

// 3. 下单成功后才增加计数
if (spotOrderId != null && swapOrderId != null) {
    redisManager.incr(POSITION_COUNT_KEY);  // 增加计数
}
```

**问题场景**：
1. 时刻 T1：币种 A 检查持仓数量 = 0，通过检查
2. 时刻 T2：币种 B 检查持仓数量 = 0，通过检查（因为 A 还没下单成功）
3. 时刻 T3：币种 C 检查持仓数量 = 0，通过检查（因为 A、B 都还没下单成功）
4. 时刻 T4：A、B、C 同时开始下单
5. 时刻 T5：只有下单成功的才增加计数

**结果**：多个币种同时通过检查，导致超过最大持仓数量限制。

### 3. 为什么只有 1 个成功但看到 3 个下单？

- 3 个币种同时通过了持仓检查
- 它们几乎同时开始下单
- 但由于网络、API 限制等原因，大部分订单失败了
- 只有 1 个订单成功，所以最终持仓数量是 1

## 解决方案

### 核心思路：先预留名额，失败再回退 ⭐⭐⭐

使用 Redis 的原子性操作，在检查的同时就预留持仓名额：

```java
// 新的代码（正确）
boolean positionReserved = false;

// 1. 先原子性增加计数（预留持仓名额）
long newCount = redisManager.incr(POSITION_COUNT_KEY);

// 2. 检查是否超过上限
if (newCount > maxPositions) {
    // 超过上限，立即回退计数
    redisManager.decr(POSITION_COUNT_KEY);
    logger.warn("⏸ 已达到最大持仓数量，跳过开仓");
    return;
}

// 3. 成功预留持仓名额
positionReserved = true;
logger.info("📊 预留持仓名额: {} -> {}/{}", symbol, newCount, maxPositions);

// 4. 下单
spotOrderId = tradingService.buySpot(...);
swapOrderId = tradingService.shortSwap(...);

// 5. 如果下单失败，回退预留的名额
if (!orderSuccess && positionReserved) {
    long currentCount = redisManager.decr(POSITION_COUNT_KEY);
    logger.info("↩️ 开仓失败，回退持仓名额: {} | 当前持仓: {}/{}", 
        symbol, currentCount, maxPositions);
}
```

### 关键改进

1. **原子性预留**：使用 `INCR` 命令原子性增加计数，立即预留名额
2. **立即检查**：增加后立即检查是否超过上限
3. **超限回退**：如果超过上限，立即使用 `DECR` 回退
4. **失败回退**：如果下单失败，也要回退预留的名额
5. **多处回退**：在所有可能失败的地方都要回退名额
   - 计算合约张数失败
   - 计算的张数为 0
   - 下单失败

## 代码修改

### 1. 开仓前预留持仓名额

```java
// 检查全局持仓数量是否已达上限（原子性操作：先增加再检查）
// 这样可以避免竞态条件，确保不会超过最大持仓数量
boolean positionReserved = false;
if (redisManager != null) {
    try {
        // 先原子性增加计数（预留持仓名额）
        long newCount = redisManager.incr(POSITION_COUNT_KEY);
        
        // 检查是否超过上限
        if (newCount > maxPositions) {
            // 超过上限，回退计数
            redisManager.decr(POSITION_COUNT_KEY);
            logger.warn("⏸ 已达到最大持仓数量 {}/{}，跳过开仓: {}", 
                newCount - 1, maxPositions, opp.symbol);
            return;
        }
        
        // 成功预留持仓名额
        positionReserved = true;
        logger.info("📊 预留持仓名额: {} -> {}/{}", opp.symbol, newCount, maxPositions);
        
    } catch (Exception e) {
        logger.error("检查持仓数量失败: {}", e.getMessage(), e);
        // 如果 Redis 失败,为了安全起见,不开仓
        return;
    }
}
```

## 技术要点

### 1. Redis 原子性操作

Redis 的 `INCR` 和 `DECR` 命令是原子性的：
- 即使多个客户端同时调用，也能保证计数准确
- 不会出现两个客户端同时读到相同的值

### 2. 先预留后检查模式

这是解决竞态条件的经典模式：
```
传统模式（有问题）：Check -> Act -> Update
新模式（正确）：Reserve -> Check -> Act (or Rollback)
```

### 3. 完整的回退机制

在所有可能失败的地方都要回退：
- ✅ 超过上限时回退
- ✅ 计算失败时回退
- ✅ 张数为 0 时回退
- ✅ 下单失败时回退

### 4. 日志清晰

每个关键步骤都有清晰的日志：
- `📊 预留持仓名额`: 成功预留
- `⏸ 已达到最大持仓数量`: 超过上限
- `✅ 开仓成功`: 订单成功
- `↩️ 回退持仓名额`: 失败回退

## 测试验证

### 1. 正常场景

```
时刻 T1: 币种 A 预留名额 (0 -> 1)
时刻 T2: 币种 B 尝试预留 (1 -> 2)，超过上限，立即回退 (2 -> 1)
时刻 T3: 币种 C 尝试预留 (1 -> 2)，超过上限，立即回退 (2 -> 1)
时刻 T4: 币种 A 下单成功，保持名额 (1)
```

**结果**：只有 1 个币种成功开仓 ✅

### 2. 失败场景

```
时刻 T1: 币种 A 预留名额 (0 -> 1)
时刻 T2: 币种 A 下单失败，回退名额 (1 -> 0)
时刻 T3: 币种 B 预留名额 (0 -> 1)
时刻 T4: 币种 B 下单成功，保持名额 (1)
```

**结果**：失败的订单会释放名额，其他币种可以继续尝试 ✅

### 3. 并发场景

```
时刻 T1: 币种 A、B、C 同时尝试预留
  - A: INCR -> 1 (成功)
  - B: INCR -> 2 (超限，回退 -> 1)
  - C: INCR -> 2 (超限，回退 -> 1)
时刻 T2: 只有 A 继续下单
```

**结果**：Redis 原子性操作保证只有 1 个币种能预留成功 ✅
