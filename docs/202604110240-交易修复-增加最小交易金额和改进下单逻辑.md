# 交易修复 - 增加最小交易金额和改进下单逻辑

## 时间
2026-04-11 02:40

## 问题描述
用户报告两个问题:
1. **现货下单失败**: 错误码 51020 - "Your order should meet or exceed the minimum order amount"
2. **下了多个币种**: 虽然配置了 `max-positions: 1`,但仍然下了 3 个币种的订单

## 问题分析

### 1. 现货下单失败
查看交易日志 `trading/order_detail_2026041113.csv`:
```csv

```

**分析**:
- 所有订单的 `spotOrderId=null`,说明现货下单失败
- 但 `swapOrderId` 有值,说明合约单还是下了
- OKX API 返回错误: `{"code":"1","data":[{"sCode":"51020","sMsg":"Your order should meet or exceed the minimum order amount."}]}`

**根本原因**:
- 配置的 `trade-amount-usdt: 10` (10 USDT)
- 计算出的现货数量太小,不满足 OKX 最小下单金额要求
- OKX 现货最小下单金额通常是 5-10 USDT,还要考虑手续费
- 例如: RSR 价格 0.001527 USDT,10 USDT 只能买 6547 个,但可能不满足最小金额要求

### 2. 下了多个币种


**分析**:
- 虽然配置了 `max-positions: 1`,但下了 3 个币种的订单
- 所有订单都失败了 (FAILED)

**根本原因**:
- 现货下单失败后,`orderSuccess = false`
- 失败时会回退持仓计数: `redisManager.decr(POSITION_COUNT_KEY)`
- 回退后持仓数量又变成 0,所以下一个币种又可以尝试开仓
- 循环往复,导致多个币种都尝试下单

### 3. 下单逻辑问题
当前代码逻辑:
```java
// 先下现货单
spotOrderId = tradingService.buySpot(opp.symbol, contractSize);
// 再下合约单
swapOrderId = tradingService.shortSwap(opp.symbol, contractSize, 1);

// 检查两个订单是否都成功
if (spotOrderId != null && swapOrderId != null) {
    orderSuccess = true;
}
```

**问题**:
- 即使现货下单失败 (`spotOrderId = null`),合约单还是会下
- 这导致合约单成功但现货单失败,无法完成对冲
- 浪费手续费,增加风险

## 解决方案

### 1. 增加最小交易金额
修改 `application-dev.yml`:
```yaml
arbitrage:
  trading:
    trade-amount-usdt: 50  # 从 10 增加到 50 USDT
```

**理由**:
- OKX 现货最小下单金额通常是 5-10 USDT
- 50 USDT 可以满足大部分币种的最小下单要求
- 对于低价币种(如 RSR 0.001527 USDT),50 USDT 可以买 32,745 个,足够满足最小金额
- 对于中价币种(如 IMX 0.1457 USDT),50 USDT 可以买 343 个,也足够
- 对于高价币种(如 BTC),可能需要更大的金额,但这些币种通常不会出现在套利列表中

### 2. 改进下单逻辑
修改 `FlinkADSArbitrageJob.java` 的 `openPosition()` 方法:
```java
try {
    if (opp.arbitrageDirection.contains("做多现货")) {
        // 策略 A: 做多现货 + 做空合约
        direction = "LONG_SPOT_SHORT_SWAP";
        // 先下现货单
        spotOrderId = tradingService.buySpot(opp.symbol, contractSize);
        // 检查现货单是否成功
        if (spotOrderId == null) {
            errorMsg = "现货下单失败,跳过合约下单";
            logger.error("❌ {}: {}", opp.symbol, errorMsg);
        } else {
            // 现货单成功,再下合约单
            swapOrderId = tradingService.shortSwap(opp.symbol, contractSize, 1);
        }
    } else {
        // 策略 B: 做空现货 + 做多合约
        direction = "SHORT_SPOT_LONG_SWAP";
        // 先下现货单
        spotOrderId = tradingService.sellSpot(opp.symbol, contractSize);
        // 检查现货单是否成功
        if (spotOrderId == null) {
            errorMsg = "现货下单失败,跳过合约下单";
            logger.error("❌ {}: {}", opp.symbol, errorMsg);
        } else {
            // 现货单成功,再下合约单
            swapOrderId = tradingService.longSwap(opp.symbol, contractSize, 1);
        }
    }
    
    // 检查两个订单是否都成功
    if (spotOrderId != null && swapOrderId != null) {
        orderSuccess = true;
        // ... 记录日志和保存订单
    }
} catch (Exception e) {
    errorMsg = e.getMessage();
    logger.error("❌ 开仓失败: {}", errorMsg, e);
}
```

**改进点**:
1. **先下现货单,检查成功后再下合约单**: 避免现货失败但合约成功的情况
2. **现货失败时跳过合约单**: 不浪费手续费,不增加风险
3. **记录详细错误信息**: 方便排查问题

### 3. 失败黑名单机制
现有的失败黑名单机制已经可以防止重复尝试失败的币种:
```java
// 如果下单失败,记录日志并加入黑名单
if (!orderSuccess) {
    logTradeToCsv(..., "FAILED", errorMsg);
    
    // 加入失败黑名单
    failedSymbols.add(opp.symbol);
    logger.warn("🚫 {} 已加入失败黑名单,不再尝试交易", opp.symbol);
}
```

**工作流程**:
1. 币种 A 尝试开仓,现货下单失败
2. 回退持仓计数,加入失败黑名单
3. 币种 B 尝试开仓,检查黑名单,不在黑名单中,继续
4. 币种 A 再次出现套利机会,检查黑名单,在黑名单中,跳过

## 预期效果

### 1. 现货下单成功率提高
- 50 USDT 的交易金额可以满足大部分币种的最小下单要求

- 失败的币种加入黑名单,不再重复尝试
- 持仓计数正确,不会因为失败回退而导致多次尝试
