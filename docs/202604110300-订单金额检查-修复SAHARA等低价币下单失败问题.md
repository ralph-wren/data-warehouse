# 订单金额检查 - 修复 SAHARA 等低价币下单失败问题

## 问题描述

套利作业在尝试交易 SAHARA-USDT 时持续失败,错误代码 51020:
```
Your order should meet or exceed the minimum order amount.
```

实际请求:
```json
{
  "instId": "SAHARA-USDT",
  "tdMode": "cross",
  "ccy": "USDT",
  "side": "sell",
  "ordType": "market",
  "sz": "45"
}
```

## 问题分析

### 根本原因

1. **SAHARA 币价值太低**: 45 个 SAHARA 可能只值 0.45 USDT
2. **低于 OKX 最小限额**: OKX 要求最小下单金额通常为 5-10 USDT
3. **代码缺少金额检查**: 下单前没有验证订单的 USDT 价值

### 为什么 Python 测试成功?

Python 测试脚本使用的是 BTC:
- 测试数量: 0.00001 BTC
- BTC 价格: ~100,000 USDT
- 订单价值: 0.00001 × 100,000 = 1 USDT (仍然可能失败)
- 实际测试可能用了更大的数量

### 对比分析

| 币种 | 数量 | 价格 | 订单价值 | 是否满足最小限额 |
|------|------|------|----------|-----------------|
| BTC | 0.0001 | 100,000 USDT | 10 USDT | ✅ 满足 |
| SAHARA | 45 | 0.01 USDT | 0.45 USDT | ❌ 不满足 |
| ETH | 0.003 | 3,500 USDT | 10.5 USDT | ✅ 满足 |

## 解决方案

### 1. 自动调整订单数量以满足最小金额

修改 `OKXTradingService.java` 的 `buySpot()` 和 `sellSpot()` 方法:

```java
// 检查订单金额是否满足最小要求,如果不满足则调整为最小金额
BigDecimal orderValue = size.multiply(price);
BigDecimal okxMinAmount = new BigDecimal("10");  // OKX 最小 10 USDT

// 从配置读取交易金额
ConfigLoader config = ConfigLoader.getInstance();
BigDecimal configTradeAmount = new BigDecimal(
    config.getString("arbitrage.trading.trade-amount-usdt", "10")
);

// 取两者中的较大值作为最小订单金额
BigDecimal minOrderAmount = okxMinAmount.max(configTradeAmount);

// 如果订单金额小于最小要求,调整数量以满足最小金额
if (orderValue.compareTo(minOrderAmount) < 0) {
    logger.warn("⚠ 订单金额 {} USDT < 最小要求 {} USDT,自动调整数量", 
        orderValue, minOrderAmount);
    
    // 重新计算数量: 数量 = 最小金额 / 价格
    size = minOrderAmount.divide(price, 8, RoundingMode.UP);
    orderValue = size.multiply(price);
    
    logger.info("  └─ 调整后: {} {} × {} USDT = {} USDT", 
        size, symbol, price, orderValue);
}
```

### 2. 修改方法签名传入价格

修改 `buySpot()` 和 `sellSpot()` 方法签名:

```java
// 修改前
public String buySpot(String symbol, BigDecimal size)
public String sellSpot(String symbol, BigDecimal size)

// 修改后
public String buySpot(String symbol, BigDecimal size, BigDecimal price)
public String sellSpot(String symbol, BigDecimal size, BigDecimal price)
```

### 3. 更新调用方传入价格

修改 `FlinkADSArbitrageJob.java` 的 `openPosition()` 方法:

```java
// 策略 A: 做多现货 + 做空合约
spotOrderId = tradingService.buySpot(opp.symbol, contractSize, opp.spotPrice);

// 策略 B: 做空现货 + 做多合约
spotOrderId = tradingService.sellSpot(opp.symbol, contractSize, opp.spotPrice);
```

## 最小订单金额逻辑

### 计算规则

```
最小订单金额 = max(OKX最小限额, 配置的交易金额)
```

### 自动调整示例

假设配置文件中 `trade-amount-usdt: 10`:

#### 示例 1: SAHARA (低价币)
- 原始数量: 45 SAHARA
- 价格: 0.01 USDT
- 原始订单价值: 0.45 USDT
- 最小要求: 10 USDT
- **自动调整**: 1000 SAHARA (10 / 0.01)
- 调整后订单价值: 10 USDT ✅

#### 示例 2: BTC (高价币)
- 原始数量: 0.0001 BTC
- 价格: 100,000 USDT
- 原始订单价值: 10 USDT
- 最小要求: 10 USDT
- **无需调整**: 保持 0.0001 BTC ✅

#### 示例 3: 配置更大交易金额
- 配置: `trade-amount-usdt: 20`
- 原始数量: 0.0001 BTC
- 价格: 100,000 USDT
- 原始订单价值: 10 USDT
- 最小要求: 20 USDT (取配置值)
- **自动调整**: 0.0002 BTC (20 / 100,000)
- 调整后订单价值: 20 USDT ✅

### 优势

1. **满足 OKX 要求**: 确保订单金额 >= 10 USDT
2. **尊重用户配置**: 如果用户配置了更大的交易金额,使用配置值
3. **自动处理低价币**: 自动增加数量以满足最小金额
4. **不会拒绝交易**: 所有币种都能正常交易
