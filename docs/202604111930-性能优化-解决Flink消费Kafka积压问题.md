# Flink 消费 Kafka 积压问题排查与优化

## 问题描述

用户发现 Kafka 里每秒都在推送数据，但 Flink 消费有积压，导致套利机会频繁超时（30秒超时）。从日志来看：

```
2026-04-11 18:51:23 - 🔍 发现新套利机会: AEVO | 价差率: -0.557900%
2026-04-11 18:51:25 - ⏳ 套利机会持续中: AEVO | 已持续 2.3 秒
2026-04-11 18:51:27 - ⏳ 套利机会持续中: AEVO | 已持续 4.7 秒
2026-04-11 18:51:57 - ⏱ 套利机会超时: AEVO | 上次持续时间 13.1 秒 | 重新开始观察
```

套利机会持续时间从 4.7 秒掉到 3.5 秒，然后频繁超时，说明数据处理链路存在严重延迟。

## 问题分析

### 1. CSV 文件写入是同步阻塞操作（主要瓶颈）

在 `TradingDecisionProcessor` 类中，每次开仓、平仓、订单更新都会调用：
- `logOrderDetail()` - 记录订单明细
- `logPositionSummary()` - 记录持仓汇总

这些方法的问题：
```java
orderDetailWriter.write(...);
orderDetailWriter.flush();  // ❌ 每次写入都立即刷新到磁盘，严重影响性能
```

**影响**：
- 每次写入都会触发磁盘 I/O
- 在高频交易场景下，磁盘 I/O 会成为严重瓶颈
- 导致 Flink 算子处理速度变慢，Kafka 消费积压

### 2. Redis 操作频繁

每次处理套利机会都会调用 Redis：
- `redisManager.getCounter(POSITION_COUNT_KEY)` - 检查持仓数量
- `redisManager.hget("okx:margin:support", symbol)` - 读取杠杆支持信息
- `redisManager.incr(POSITION_COUNT_KEY)` - 增加持仓计数

虽然 Redis 很快，但在高频场景下仍会累积延迟。

### 3. OKX API 调用是同步的

`openPosition()` 方法中调用：
- `tradingService.buySpot()` - 同步 HTTP 请求
- `tradingService.shortSwap()` - 同步 HTTP 请求
- `tradingService.sellSpot()` - 同步 HTTP 请求
- `tradingService.longSwap()` - 同步 HTTP 请求

这些都是阻塞操作，会严重影响 Flink 算子的吞吐量。

### 4. Interval Join 窗口过大

当前配置：
```java
.between(Time.seconds(-10), Time.seconds(10))  // ±10 秒窗口
```

这会保留大量状态数据，增加内存压力和状态后端负担。

## 优化方案

### 优化 1：移除 CSV 文件的立即刷新（已实施）

**修改内容**：
1. 增大 BufferedWriter 缓冲区到 64KB（默认 8KB）
2. 移除所有 `flush()` 调用，让缓冲区自动刷新
3. 只在关闭文件时才强制刷新

**代码修改**：
```java
// 修改前
orderDetailWriter = new BufferedWriter(new FileWriter(orderDetailFile, true));
orderDetailWriter.write(...);
orderDetailWriter.flush();  // ❌ 每次都刷新

// 修改后
orderDetailWriter = new BufferedWriter(new FileWriter(orderDetailFile, true), 65536);  // 64KB 缓冲区
orderDetailWriter.write(...);
// 移除 flush()，让缓冲区积累数据
```

**预期效果**：
- 减少磁盘 I/O 次数 90% 以上
- 提升 Flink 算子吞吐量 2-5 倍
- 降低 Kafka 消费延迟

**风险**：
- 如果程序异常退出，缓冲区中的数据可能丢失
- 建议：在 `close()` 方法中确保 `flush()` 被调用

### 优化 2：Redis 操作优化（建议）

**当前问题**：
- 每次处理套利机会都查询 Redis
- 杠杆支持信息很少变化，不需要每次查询

**优化方案**：
1. 使用本地缓存（Caffeine）缓存杠杆支持信息
2. 设置 TTL 为 1 小时，减少 Redis 查询
3. 持仓计数仍然使用 Redis，确保准确性

**代码示例**：
```java
// 使用 Caffeine 缓存
private transient Cache<String, Boolean> marginSupportCache;

@Override
public void open(Configuration parameters) throws Exception {
    marginSupportCache = Caffeine.newBuilder()
        .maximumSize(1000)
        .expireAfterWrite(1, TimeUnit.HOURS)
        .build();
}

// 查询时先查缓存
Boolean marginSupported = marginSupportCache.get(symbol, key -> {
    String flags = redisManager.hget("okx:margin:support", key);
    return flags != null && flags.contains("1");
});
```

### 优化 3：异步 API 调用（建议）

**当前问题**：
- OKX API 调用是同步的，会阻塞 Flink 算子
- 在高频交易场景下，API 延迟会累积

**优化方案**：
1. 使用 Flink 的 `AsyncDataStream` API
2. 将 API 调用改为异步非阻塞
3. 使用线程池管理异步任务

**代码示例**：
```java
// 使用 AsyncDataStream
AsyncDataStream.unorderedWait(
    opportunityStream,
    new AsyncFunction<ArbitrageOpportunity, TradeRecord>() {
        @Override
        public void asyncInvoke(ArbitrageOpportunity input, ResultFuture<TradeRecord> resultFuture) {
            CompletableFuture.supplyAsync(() -> {
                // 异步调用 API
                return tradingService.buySpot(...);
            }, executorService).thenAccept(result -> {
                resultFuture.complete(Collections.singleton(result));
            });
        }
    },
    60, TimeUnit.SECONDS
);
```

### 优化 4：减小 Interval Join 窗口（建议）

**当前配置**：
```java
.between(Time.seconds(-10), Time.seconds(10))  // ±10 秒
```

**优化方案**：
- 根据实际数据延迟调整窗口大小
- 如果数据延迟通常在 2-3 秒，可以调整为 ±5 秒
- 减少状态后端压力

**代码修改**：
```java
.between(Time.seconds(-5), Time.seconds(5))  // ±5 秒
```

### 优化 5：增加并行度（建议）

**当前配置**：
```yaml
flink:
  execution:
    parallelism: 4  # dev 环境
```

**优化方案**：
- 如果 CPU 资源充足，可以增加到 8 或 16
- 检查 Flink Web UI 的反压指标，确认瓶颈在哪个算子
- 针对性地调整特定算子的并行度

## 验证方法

### 1. 检查 Flink Web UI

访问 `http://localhost:8083`（或对应端口），查看：
- **Backpressure**：检查哪个算子有反压
- **Records Sent/Received**：检查数据流速率
- **Checkpoint Duration**：检查 Checkpoint 是否正常

### 2. 检查 Kafka Lag

```bash
# 查看消费者组的 Lag
docker exec -it kafka kafka-consumer-groups.sh \
  --bootstrap-server localhost:9092 \
  --group ads-arbitrage \
  --describe
```

### 3. 监控日志

观察套利机会的持续时间是否稳定：
```
⏳ 套利机会持续中: AEVO | 价差率: -0.557900% | 已持续 2.3 秒 / 需要 5 秒
⏳ 套利机会持续中: AEVO | 价差率: -0.557900% | 已持续 4.7 秒 / 需要 5 秒
```

如果持续时间稳定增长，说明消费延迟已解决。

1. **数据丢失风险**：移除 `flush()` 后，如果程序异常退出，缓冲区中的数据可能丢失。建议在 `close()` 方法中确保 `flush()` 被调用。

2. **监控指标**：实施优化后，持续监控 Flink Web UI 和 Kafka Lag，确认优化效果。

3. **渐进式优化**：先实施优化 1，观察效果后再决定是否实施其他优化。

4. **测试环境**：建议先在 dev 环境测试，确认无问题后再部署到 prod 环境。
