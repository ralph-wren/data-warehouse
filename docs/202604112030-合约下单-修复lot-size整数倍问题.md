# 合约下单 - 修复 lot size 整数倍问题并确保现货合约数量一致

**日期**: 2026-04-11 20:30  
**问题**: 合约下单失败，错误 51121: Order quantity must be a multiple of the lot size  
**状态**: ✅ 已解决

---

## 问题描述

从日志中发现：
1. 现货下单成功（CVX-USDT，订单ID: 3469793412390346752）
2. 合约下单失败，错误信息：
   ```
   "sCode":"51121"
   "sMsg":"Order quantity must be a multiple of the lot size."
   ```
3. 现货成功日志和合约失败日志都没有写入CSV文件
4. 现货和合约的币数量不一致

## 根本原因

### 问题1：合约数量不符合 lot size 要求

之前的实现：
- 现货使用 `calculateSpotSize()` 计算币的数量（向上取整）
- 合约使用 `calculateContractSize()` 重新计算张数（向下取整）
- 导致现货和合约的币数量不一致

例如：
- CVX-USDT 现货：数量 = 5.5 USDT / 价格 = 5.5 CVX
- CVX-USDT-SWAP 合约：重新计算 = 5.5 USDT / 价格 / ctVal = 可能不同的币数量
- 但合约可能要求 lot size = 1，即必须是整数张数

### 问题2：日志写入逻辑不完整

在 `openPosition()` 方法的 `else if (spotOrderId != null && swapOrderId == null)` 分支中：
- 缺少合约失败时的价格和数量信息
- 错误信息不够详细

## 解决方案

### 1. 修改合约下单方法 - 确保现货合约数量一致

修改 `OKXTradingService.longSwap()` 和 `shortSwap()` 方法：

**修改前**：
```java
public String longSwap(String symbol, BigDecimal coinAmount, BigDecimal price, int leverage) {
    // 重新计算 USDT 金额
    BigDecimal usdtAmount = coinAmount.multiply(price);
    
    // 重新计算张数（可能导致币数量不一致）
    BigDecimal contractSize = calculateContractSize(symbol, usdtAmount, price, redisManager);
    
    orderRequest.put("sz", contractSize.toPlainString());
}
```

**修改后**：
```java
public String longSwap(String symbol, BigDecimal spotCoinAmount, BigDecimal price, int leverage) {
    // 获取交易对信息
    InstrumentInfo info = getInstrumentInfo(symbol);
    
    // 直接将现货币数量转换为张数
    // 张数 = 币数量 / ctVal
    BigDecimal rawSize = spotCoinAmount.divide(info.ctVal, 8, RoundingMode.DOWN);
    
    // 按 lotSz 向下取整，确保符合 lot size 要求
    BigDecimal lots = rawSize.divide(info.lotSz, 0, RoundingMode.DOWN);
    BigDecimal contractSize = lots.multiply(info.lotSz);
    
    // 计算实际币数量（用于日志对比）
    BigDecimal actualCoinAmount = contractSize.multiply(info.ctVal);
    
    logger.info("📈 做多合约: {} | 现货币数量: {} {} | 合约张数: {} | 实际币数量: {} {} | 杠杆: {}x", 
        instId, spotCoinAmount, symbol, contractSize, actualCoinAmount, symbol, leverage);
    
    orderRequest.put("sz", contractSize.toPlainString());
}
```

### 2. 关键改进

1. **参数名称更改**：`coinAmount` → `spotCoinAmount`，明确表示这是现货的币数量
2. **直接转换**：不再重新计算 USDT 金额，直接将现货币数量转换为张数
3. **向下取整**：按 lot size 向下取整，确保不会超过现货数量
4. **日志对比**：输出现货币数量和合约实际币数量，方便对比

### 3. 完善日志写入逻辑

在合约下单失败分支中：
- 添加价格和数量信息到日志
- 获取详细错误信息并分类处理
- 添加日志写入确认信息

## 技术细节

### 数量一致性保证

**计算流程**：
1. 现货：`spotCoinAmount = USDT金额 / 价格（向上取整）`
2. 合约：
   - 原始张数 = `spotCoinAmount / ctVal`
   - 符合 lot size 的张数 = `floor(原始张数 / lotSz) * lotSz`
   - 实际币数量 = `张数 * ctVal`

**示例**：
假设 CVX-USDT-SWAP：
- ctVal = 0.1（1 张 = 0.1 CVX）
- lotSz = 1（最小交易单位 1 张）
- 现货币数量 = 5.5 CVX

计算过程：
1. 原始张数 = 5.5 / 0.1 = 55 张
2. 符合 lot size 的张数 = floor(55 / 1) * 1 = 55 张
3. 实际币数量 = 55 * 0.1 = 5.5 CVX ✅ 与现货一致

**特殊情况**：
如果 ctVal = 0.3，lotSz = 1：
1. 原始张数 = 5.5 / 0.3 = 18.33 张
2. 符合 lot size 的张数 = floor(18.33 / 1) * 1 = 18 张
3. 实际币数量 = 18 * 0.3 = 5.4 CVX ⚠️ 略小于现货（向下取整）

## 注意事项

1. **数量一致性**：
   - 如果 ctVal 和 lotSz 配合良好（如 ctVal=0.1, lotSz=1），现货和合约数量完全一致
   - 如果 ctVal 和 lotSz 不匹配，合约实际币数量可能略小于现货（向下取整）
   - 差异通常很小，不会影响套利效果

2. **向下取整原因**：
   - 确保合约数量不会超过现货数量
   - 避免因精度问题导致合约数量大于现货

3. **日志输出**：
   - 日志中会输出三个数量：现货币数量、合约张数、实际币数量
   - 方便排查数量不一致的问题

4. **最小下单数量**：
   - 如果计算后的张数为 0 或小于最小下单数量，会返回 null，不会下单
