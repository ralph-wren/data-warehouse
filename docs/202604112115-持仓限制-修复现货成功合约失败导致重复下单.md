# 持仓限制 - 修复现货成功合约失败导致重复下单

## 问题描述

从 CSV 日志发现，ZRX 和 S 两个币对都出现了现货下单成功但合约下单失败的情况，导致持仓超过 1 个。

## 问题现象

查看最新的 `order_detail_*.csv` 日志：

```
2026-04-11 21:10:20,ZRX,OPEN,SHORT_SPOT_LONG_SWAP,SPOT,3469884767854682112,0.10368,5.5,SUCCESS,现货订单已提交
2026-04-11 21:10:20,ZRX,OPEN,SHORT_SPOT_LONG_SWAP,SWAP,N/A,0.10316,5.5,FAILED,合约下单失败

2026-04-11 21:10:22,S,OPEN,SHORT_SPOT_LONG_SWAP,SPOT,3469884827581571072,0.04192,5.5,SUCCESS,现货订单已提交
2026-04-11 21:10:22,S,OPEN,SHORT_SPOT_LONG_SWAP,SWAP,N/A,0.04171,5.5,FAILED,合约下单失败
```

**问题**：
1. ZRX 现货下单成功，合约下单失败
2. 2 秒后，S 现货下单成功，合约下单失败
3. 导致持仓超过 1 个（配置的最大持仓数）

## 根本原因

在 `TradingDecisionProcessor.openPosition()` 方法中：

```java
if (spotOrderId != null && swapOrderId != null) {
    // 两个订单都成功 - 创建临时持仓状态
    PositionState tempPosition = new PositionState();
    tempPosition.setOpen(true);  // 标记为持仓中
    positionState.update(tempPosition);
    
} else if (spotOrderId != null && swapOrderId == null) {
    // 现货单成功但合约单失败
    // ⚠️ 问题：没有创建持仓状态！
    recordFailure(opp.symbol, "合约下单失败");
    // 写入日志...
}
```

**问题分析**：
1. 当现货下单成功但合约下单失败时，代码只是记录了失败日志
2. 但没有创建持仓状态（`positionState.update()`）
3. 导致系统认为没有持仓（`position == null`）
4. 当下一个套利机会（S）出现时，系统检查持仓状态发现 `position == null`
5. 于是继续下单，导致持仓超过 1 个

**持仓检查逻辑**：
```java
@Override
public void processElement1(
        ArbitrageOpportunity opportunity,
        Context ctx,
        Collector<TradeRecord> out) throws Exception {
    
    PositionState position = positionState.value();
    
    // 决策 1: 开仓逻辑
    if (position == null || !position.isOpen()) {
        // 没有持仓，可以开仓
        if (shouldOpen(opportunity)) {
            openPosition(opportunity, out);
        }
    }
}
```

## 解决方案

在现货下单成功但合约下单失败时，也要创建临时持仓状态，防止继续下单：

```java
} else if (spotOrderId != null && swapOrderId == null) {
    // 现货单成功但合约单失败
    logger.error("❌ 合约下单失败,现货单已提交: {}", spotOrderId);
    
    // 记录失败
    recordFailure(opp.symbol, "合约下单失败");
    
    // ⚠️ 关键修复：创建临时持仓状态，防止继续下单导致持仓超过1个
    // 虽然合约下单失败，但现货已经提交，需要标记为"持仓中"避免重复下单
    PositionState tempPosition = new PositionState();
    tempPosition.setSymbol(opp.symbol);
    tempPosition.setOpen(true);  // 标记为持仓中
    tempPosition.setDirection(direction);
    tempPosition.setAmount(tradeAmount);
    tempPosition.setEntrySpotPrice(opp.spotPrice);
    tempPosition.setEntrySwapPrice(BigDecimal.ZERO);  // 合约未成交，设置为0
    tempPosition.setOpenTime(System.currentTimeMillis());
    tempPosition.setSpotOrderId(spotOrderId);
    tempPosition.setSwapOrderId(null);  // 合约订单失败，设置为null
    tempPosition.setLastLogTime(System.currentTimeMillis());
    positionState.update(tempPosition);
    
    logger.warn("⚠️ 已创建临时持仓状态（现货成功，合约失败），防止重复下单");
    
    // 写入日志...
}
```

## 关键改动

1. **创建临时持仓状态**：
   - 设置 `setOpen(true)` 标记为持仓中
   - 设置现货订单 ID：`setSpotOrderId(spotOrderId)`
   - 设置合约订单 ID 为 null：`setSwapOrderId(null)`
   - 设置合约价格为 0：`setEntrySwapPrice(BigDecimal.ZERO)`

2. **防止重复下单**：
   - 当下一个套利机会出现时，系统检查持仓状态发现 `position != null && position.isOpen()`
   - 不会再执行开仓逻辑，避免持仓超过 1 个

3. **保持一致性**：
   - 无论现货和合约是否都成功，只要现货下单成功，就创建持仓状态
   - 确保持仓限制检查的准确性

## 后续处理

虽然创建了临时持仓状态，但这是一个"不完整"的持仓（只有现货，没有合约）。后续需要：

1. **手动平仓**：
   - 管理员需要手动平掉现货持仓
   - 或者等待订单自动过期

2. **监控告警**：
   - 可以添加监控，当检测到 `swapOrderId == null` 的持仓时发出告警
   - 提醒管理员处理单边持仓

3. **自动回滚**（可选）：
   - 未来可以考虑添加自动取消现货订单的功能
   - 但需要注意订单可能已经成交，取消会失败
