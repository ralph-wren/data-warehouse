# 订单状态 - 添加订单失败处理逻辑

## 问题描述

用户发现 COMP 现货和合约订单都提交成功（返回 orderId），但现货订单后续因为"借币额度不足"失败了，系统却没有清除临时持仓状态，导致后续又下单了 ZK。

## 问题分析

### CSV 日志分析

```csv
2026-04-11 22:25:00,COMP,OPEN,SHORT_SPOT_LONG_SWAP,SWAP,3470035081543049216,20.57,0.26582891,PENDING,订单已提交
2026-04-11 22:25:00,COMP,OPEN,SHORT_SPOT_LONG_SWAP,SPOT,3470035072181362688,20.69,0.26582891,PENDING,订单已提交
2026-04-11 22:25:13,COMP,OPEN,SHORT_SPOT_LONG_SWAP,SPOT,N/A,N/A,N/A,FAILED,Your order failed because it requires borrowing COMP; and the platform's loan quota is insufficient.
2026-04-11 22:26:12,ZK,OPEN,SHORT_SPOT_LONG_SWAP,SPOT,3470037487999803392,0.0159,345.69453175,PENDING,订单已提交
```

### 问题根本原因

1. **订单提交成功 ≠ 订单执行成功**
   - OKX 返回的 orderId 只表示订单已提交到交易所
   - 订单后续可能因为各种原因失败（借币额度不足、余额不足、数量不符合要求等）

2. **缺少订单状态监听**
   - 代码中有 `processElement2` 方法处理订单更新
   - 但只处理了 `filled`（成交）状态
   - 没有处理 `canceled`（取消）或 `failed`（失败）状态

3. **临时持仓状态未清除**
   - 订单提交成功后，创建了临时持仓状态（防止重复下单）
   - 订单失败后，临时持仓状态没有被清除
   - 导致系统认为还有持仓，不会继续下单
   - 但当持仓状态过期或被清理后，系统又继续下单

## 解决方案

### 1. 修改 processElement2 方法

添加订单失败和取消的处理逻辑：

```java
@Override
public void processElement2(
        OrderUpdate order,
        Context ctx,
        Collector<TradeRecord> out) throws Exception {
    
    // 处理订单更新
    if ("filled".equals(order.state)) {
        handleOrderFilled(order, out);
    } else if ("canceled".equals(order.state) || "failed".equals(order.state)) {
        // 处理订单取消或失败
        handleOrderFailed(order, out);
    }
}
```

### 2. 添加 handleOrderFailed 方法

处理订单失败或取消的逻辑：

```java
/**
 * 处理订单失败或取消
 * 
 * 功能：
 * 1. 检查是否是待确认订单
 * 2. 如果是开仓订单失败，清除临时持仓状态
 * 3. 记录失败日志
 * 4. 加入黑名单
 */
private void handleOrderFailed(OrderUpdate order, Collector<TradeRecord> out) 
        throws Exception {
    
    PendingOrder pending = pendingOrders.get(order.orderId);
    if (pending == null) {
        return;
    }
    
    logger.warn("⚠️ 订单失败: orderId={}, symbol={}, state={}", 
        order.orderId, pending.symbol, order.state);
    
    // 如果是开仓订单失败
    if ("OPEN".equals(pending.action)) {
        // 清除临时持仓状态
        PositionState position = positionState.value();
        if (position != null && position.isOpen() && pending.symbol.equals(position.getSymbol())) {
            logger.warn("⚠️ 清除临时持仓状态: symbol={}", pending.symbol);
            positionState.clear();
        }
        
        // 加入黑名单
        String failureReason = "订单失败: " + order.state;
        recordFailure(pending.symbol, failureReason);
        
        // 写入失败日志
        String timestamp = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
        
        // 判断是现货还是合约订单失败
        String orderType = order.orderId.equals(pending.spotOrderId) ? "SPOT" : "SWAP";
        String errorLog = String.format("%s,%s,OPEN,%s,%s,%s,N/A,N/A,FAILED,%s",
            timestamp, pending.symbol, "UNKNOWN", orderType, order.orderId, failureReason.replace(",", ";"));
        orderDetailWriter.write(errorLog);
    }
    
    // 清理待确认订单
    pendingOrders.remove(pending.spotOrderId);
    pendingOrders.remove(pending.swapOrderId);
}
```

## 工作流程

1. **订单提交**：现货和合约订单提交成功，创建临时持仓状态
2. **订单更新**：OKX WebSocket 推送订单状态更新
3. **状态判断**：
   - 如果订单成交（`filled`），调用 `handleOrderFilled` 确认开仓
   - 如果订单失败（`failed`）或取消（`canceled`），调用 `handleOrderFailed` 清除临时持仓状态
4. **清除状态**：清除临时持仓状态，允许继续下单
5. **加入黑名单**：将失败的币对加入黑名单，避免短时间内重复下单

## 优势

1. **及时清除临时持仓**：订单失败后立即清除临时持仓状态，避免阻塞后续下单
2. **防止重复下单**：失败的币对加入黑名单，避免短时间内重复下单
3. **详细日志记录**：记录订单失败的详细信息，方便排查问题
4. **状态一致性**：确保持仓状态与实际订单状态一致

## 注意事项

1. **订单状态更新依赖 WebSocket**：需要确保 OKX 私有频道 WebSocket 连接正常
2. **黑名单时间**：失败的币对会加入黑名单，默认持续 60 分钟（可配置）
3. **订单状态**：OKX 订单状态包括：`live`（待成交）、`filled`（完全成交）、`partially_filled`（部分成交）、`canceled`（已取消）、`failed`（失败）
