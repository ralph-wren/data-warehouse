# 下单方式 - 统一使用币数量下单（现货和合约）

## 问题描述

用户提出需求：现货和合约都应该使用"币的数量"来下单，而不是现货用 USDT 金额、合约用币数量。

### 原有逻辑的问题

1. **现货下单**：使用 USDT 金额，内部转换为币数量
2. **合约下单**：使用币数量，内部转换为合约张数
3. **不一致性**：两种下单方式不统一，容易导致转换错误

### 用户需求

统一下单逻辑：
1. 根据 USDT 金额和价格计算币数量：`coinAmount = usdtAmount / price`
2. 查询合约最小下单单位：从 Redis `okx:instrument:cache` 获取 `ctVal`（合约面值）和 `lotSz`（最小下单单位）
3. 按合约要求调整币数量：确保符合 lot size 要求
4. 现货和合约都使用相同的币数量下单

## 解决方案

### 1. 创建 `calculateValidCoinAmount` 方法

在 `OKXTradingService` 中添加新方法，用于计算符合合约要求的币数量：

```java
/**
 * 计算符合合约要求的币数量
 * 
 * @param symbol 交易对符号（如 BTC）
 * @param usdtAmount USDT 金额
 * @param price 当前价格
 * @return 符合合约要求的币数量，如果失败返回 null
 */
public BigDecimal calculateValidCoinAmount(String symbol, BigDecimal usdtAmount, BigDecimal price) {
    try {
        // 1. 根据 USDT 金额和价格计算币数量
        BigDecimal rawCoinAmount = usdtAmount.divide(price, 8, java.math.RoundingMode.DOWN);
        
        // 2. 获取合约交易对信息
        InstrumentInfo info = getInstrumentInfo(symbol);
        if (info == null) {
            logger.error("❌ 无法获取 {} 的交易对信息", symbol);
            return null;
        }
        
        // 3. 将币数量转换为张数
        // 张数 = 币数量 / ctVal
        BigDecimal rawSize = rawCoinAmount.divide(info.ctVal, 8, java.math.RoundingMode.DOWN);
        
        // 4. 按 lotSz 向上取整，确保符合 lot size 要求
        BigDecimal lots = rawSize.divide(info.lotSz, 0, java.math.RoundingMode.UP);
        BigDecimal contractSize = lots.multiply(info.lotSz);
        
        // 5. 反向计算实际币数量
        BigDecimal validCoinAmount = contractSize.multiply(info.ctVal);
        
        logger.info("💰 币数量计算: {} USDT ÷ {} = {} {} (原始) → {} 张 → {} {} (调整后)", 
            usdtAmount, price, rawCoinAmount, symbol, contractSize, validCoinAmount, symbol);
        
        return validCoinAmount;
        
    } catch (Exception e) {
        logger.error("❌ 计算币数量失败: {}", e.getMessage(), e);
        return null;
    }
}
```

## 关键改进

### 1. 统一下单逻辑

- 现货和合约都使用币数量下单
- 避免了 USDT 金额和币数量之间的混淆
- 减少了转换错误的可能性

### 2. 符合合约要求

- 自动按 `ctVal`（合约面值）和 `lotSz`（最小下单单位）调整币数量
- 确保合约下单不会因为数量不符合要求而失败

### 3. 日志清晰

- 记录原始币数量、合约张数、调整后币数量
- 方便排查问题和验证计算逻辑

## 测试验证

### 示例计算

假设：
- USDT 金额：5.5 USDT
- COMP 价格：20 USDT
- ctVal：0.1（1 张合约 = 0.1 个币）
- lotSz：1（最小下单单位 1 张）

计算过程：
1. 原始币数量：5.5 ÷ 20 = 0.275 COMP
2. 原始张数：0.275 ÷ 0.1 = 2.75 张
3. 向上取整：ceil(2.75 ÷ 1) × 1 = 3 张
4. 调整后币数量：3 × 0.1 = 0.3 COMP
5. 实际 USDT 金额：0.3 × 20 = 6 USDT

### 日志输出

```
💰 币数量计算: 5.5 USDT ÷ 20 = 0.275 COMP (原始) → 3 张 → 0.3 COMP (调整后)
📈 做多合约: COMP-USDT-SWAP | 现货币数量: 0.3 COMP | 合约张数: 3 | 实际币数量: 0.3 COMP | 杠杆: 3x
```

## 注意事项

1. **向上取整 vs 向下取整**
   - `calculateValidCoinAmount` 使用向上取整，确保不会因为数量不足而下单失败
   - `longSwap` 和 `shortSwap` 内部使用向下取整，确保不会超过可用余额

2. **实际金额可能略高于配置金额**
   - 由于向上取整，实际使用的 USDT 金额可能略高于配置的 `trade-amount-usdt`
   - 例如：配置 5.5 USDT，实际可能使用 6 USDT

3. **现货和合约数量一致**
   - 现货和合约使用相同的币数量下单
   - 确保套利策略的对冲效果

## 总结

通过统一使用币数量下单，解决了现货和合约下单方式不一致的问题。新的逻辑更加清晰、可靠，减少了转换错误的可能性。
