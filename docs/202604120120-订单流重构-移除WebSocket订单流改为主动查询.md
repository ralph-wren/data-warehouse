# 订单流重构 - 移除 WebSocket 订单流改为主动查询

## 问题描述

下单成功后没有使用 order id 查询订单明细,原因是订单明细查询依赖 WebSocket 推送的订单成交消息,但 WebSocket 可能有延迟或连接问题。

## 解决方案

### 1. 改为主动查询模式

在 `openPosition` 方法中,订单提交成功后立即启动异步任务主动查询订单明细(延迟 2 秒),不再依赖 WebSocket 推送。

### 2. 移除订单流相关逻辑

由于不再需要 WebSocket 订单流,移除了以下内容:

#### FlinkADSArbitrageJob.java
- 移除 `OKXOrderWebSocketSource` 和 `OrderUpdateParser` 相关 import
- 移除创建订单流的代码
- 移除 Connect 订单流的代码
- 更新注释和日志

#### TradingDecisionProcessor.java
- 将类定义从 `KeyedCoProcessFunction` 改为 `KeyedProcessFunction`
- 将 `processElement1` 方法改为 `processElement`
- 删除 `processElement2` 方法(处理订单更新)
- 删除 `handleOrderFailed` 方法(处理订单失败)
- 删除 `confirmOpen` 方法(确认开仓)
- 删除 `confirmClose` 方法(确认平仓)
- 删除 `PendingOrder` 相关的 import
- 删除 `pendingOrders` MapState 状态
- 删除 `pendingOrders` 状态的初始化代码
- 移除 `openPosition` 和 `closePosition` 方法中保存 `PendingOrder` 的代码

## 优势

1. **简化架构**: 不再需要维护订单流和 Connect 操作
2. **减少依赖**: 不依赖 WebSocket 订单推送的稳定性
3. **主动控制**: 主动查询订单明细,更可控
4. **减少状态**: 不再需要 `pendingOrders` MapState
5. **代码简洁**: 删除了大量订单确认相关的代码

## 注意事项

1. 订单查询延迟设置为 2 秒,可根据实际情况调整
2. 订单查询使用独立的线程池,不会阻塞主流程
3. 如果订单查询失败,会记录错误日志,但不影响后续交易
