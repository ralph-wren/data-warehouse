# 持仓限制修复 - 使用 Redis 全局持仓计数器

## 问题描述

虽然配置了 `max-positions: 1`（最大持仓1个币种），但在 `order_detail` 表中发现同时持仓了 SUSHI 和 ACH 两个币种。

## 根本原因

1. **Flink 状态隔离**：由于使用 `keyBy(symbol)` 对数据流进行分区，每个币种有独立的 Flink 状态，导致每个币种独立判断是否可以开仓
2. **缺少全局计数器**：没有使用 Redis 的全局持仓计数器 `okx:arbitrage:position:count` 来限制全局持仓数量
3. **竞态条件**：多个币种的数据可能同时到达，在各自的状态中都判断为"未持仓"，导致同时开仓

## 解决方案

使用 `RedisConnectionManager` 管理 Redis 连接，并在开仓/平仓时操作全局持仓计数器 `okx:arbitrage:position:count`。

### 1. 替换 Redis 连接池为 RedisConnectionManager

**修改前**：
```java
private transient redis.clients.jedis.JedisPool jedisPool;  // Redis 连接池

// 在 open 方法中初始化
String redisHost = config.getString("redis.host", "localhost");
int redisPort = config.getInt("redis.port", 6379);
// ... 创建 JedisPool
```

**修改后**：
```java
private transient RedisConnectionManager redisManager;  // Redis 连接管理器

// 在 open 方法中初始化
redisManager = new RedisConnectionManager(config);
logger.info("✓ Redis 连接管理器初始化成功");
```

### 2. 开仓前检查全局持仓数量

在 `openPosition` 方法开始时，先检查全局持仓数量：

```java
// ⭐ 关键修复：在开仓前检查全局持仓数量
try {
    long currentPositionCount = redisManager.getCounter(REDIS_KEY_POSITION_COUNT);
    logger.info("📊 当前全局持仓数量: {} / {}", currentPositionCount, maxPositions);
    
    if (currentPositionCount >= maxPositions) {
        logger.warn("⚠️ 全局持仓数量已达上限 ({}/{}),拒绝开仓: {}", 
            currentPositionCount, maxPositions, opp.symbol);
        
        // 写入拒绝日志
        String timestamp = LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss"));
        String rejectLog = String.format("%s,%s,OPEN,REJECTED,N/A,N/A,N/A,N/A,REJECTED,全局持仓数量已达上限(%d/%d)",
            timestamp, opp.symbol, currentPositionCount, maxPositions);
        orderDetailWriter.write(rejectLog);
        return;
    }
    
    // 先增加全局持仓计数（乐观锁）
    long newCount = redisManager.incr(REDIS_KEY_POSITION_COUNT);
    logger.info("✅ 全局持仓计数已增加: {} -> {}", currentPositionCount, newCount);
    
} catch (Exception e) {
    logger.error("❌ 检查全局持仓数量失败: {}", e.getMessage(), e);
    // Redis 操作失败不影响主流程,继续执行
}
```

### 3. 开仓失败时回滚计数

在现货或合约下单失败时，回滚全局持仓计数：

```java
// 现货下单失败
if (spotOrderId == null) {
    logger.error("❌ 现货下单失败,取消合约下单");
    
    // ⭐ 回滚全局持仓计数
    try {
        long newCount = redisManager.decr(REDIS_KEY_POSITION_COUNT);
        logger.info("🔄 现货下单失败,已回滚全局持仓计数: {}", newCount);
    } catch (Exception e) {
        logger.error("❌ 回滚全局持仓计数失败: {}", e.getMessage());
    }
    
    recordFailure(opp.symbol, "现货下单失败");
    return;
}
```

**注意**：如果现货成功但合约失败，不回滚计数，因为现货已经持仓。

### 4. 平仓成功后减少计数

在 `closePosition` 方法中，平仓订单提交成功后减少全局持仓计数：

```java
if (spotOrderId != null && swapOrderId != null) {
    logger.info("📝 平仓订单已提交: spotOrderId={}, swapOrderId={}", spotOrderId, swapOrderId);
    
    // ⭐ 平仓成功,减少全局持仓计数
    try {
        long newCount = redisManager.decr(REDIS_KEY_POSITION_COUNT);
        logger.info("✅ 平仓成功,已减少全局持仓计数: {}", newCount);
    } catch (Exception e) {
        logger.error("❌ 减少全局持仓计数失败: {}", e.getMessage());
    }
    
    // 写入订单日志...
}
```

### 5. 关闭时清理资源

在 `close` 方法中关闭 Redis 连接管理器：

```java
@Override
public void close() throws Exception {
    // ... 其他资源关闭
    
    // 关闭 Redis 连接管理器
    if (redisManager != null) {
        redisManager.close();
    }
    
    logger.info("TradingDecisionProcessor 已关闭");
}
```

## 关键改进

1. **全局计数器**：使用 Redis 的 `okx:arbitrage:position:count` 作为全局持仓计数器，跨越所有 Flink 并行度
2. **乐观锁机制**：先增加计数再下单，失败时回滚，避免竞态条件
3. **异常处理**：Redis 操作失败不影响主流程，只记录错误日志
4. **精确回滚**：根据不同的失败场景决定是否回滚计数
5. **统一管理**：使用 `RedisConnectionManager` 统一管理 Redis 连接，简化代码

## 测试验证

1. 启动 Flink 作业
2. 观察日志输出：
   - 开仓前：`📊 当前全局持仓数量: 0 / 1`
   - 开仓成功：`✅ 全局持仓计数已增加: 0 -> 1`
   - 第二个币种尝试开仓：`⚠️ 全局持仓数量已达上限 (1/1),拒绝开仓`
   - 平仓成功：`✅ 平仓成功,已减少全局持仓计数: 0`
3. 检查 Redis：
   ```bash
   redis-cli
   > GET okx:arbitrage:position:count
   "1"  # 持仓1个币种时
   ```
4. 检查 `order_detail.csv`：
   - 应该只有1个币种的开仓记录
   - 其他币种应该有 `REJECTED` 记录

## 配置说明

在 `application-dev.yml` 中配置最大持仓数量：

```yaml
arbitrage:
  trading:
    max-positions: 1  # 最大持仓数量（币种数）
```

## 注意事项

1. **Redis 可用性**：Redis 操作失败不会阻止交易，但会导致持仓限制失效
2. **计数器初始化**：首次启动时，Redis 中的计数器应该为 0
3. **手动清理**：如果程序异常退出，可能需要手动重置计数器：
   ```bash
   redis-cli
   > SET okx:arbitrage:position:count 0
   ```
4. **并发安全**：Redis 的 INCR/DECR 操作是原子性的，天然支持并发
