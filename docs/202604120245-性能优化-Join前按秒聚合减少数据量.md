# 性能优化 - Join前按秒聚合减少数据量

## 优化目标

优化 `FlinkADSArbitrageJob` 套利作业的性能,在 Interval Join 之前先对现货和合约两条流按秒聚合,保留每秒最新的一条记录,减少 Join 的数据量,提升作业性能。

## 优化前的问题

### 数据流特点
- 现货价格流和合约价格流都是高频数据,每秒可能有多条记录
- Interval Join 时间窗口为 ±2 秒,会关联大量数据
- 大量重复的价格数据参与 Join,浪费计算资源

### 性能瓶颈
```
Kafka(现货) ──┐
              ├─→ Interval Join (±2秒) ──→ 计算套利 ──→ 交易决策
Kafka(合约) ──┘
```

问题:
1. 每秒多条价格数据都参与 Join
2. Join 窗口内数据量大,计算开销高
3. 实际上只需要每秒最新的价格即可

## 优化方案

### 核心思路
在 Join 之前,先对两条流按秒聚合,保留每秒最新的一条记录:

```
Kafka(现货) ──→ 按秒聚合(保留最新) ──┐
                                    ├─→ Interval Join (±2秒) ──→ 计算套利
Kafka(合约) ──→ 按秒聚合(保留最新) ──┘
```

### 实现方式

使用 Flink 的 Tumbling Window + Reduce 实现按秒聚合:

**现货流聚合**:
```java
DataStream<SpotPrice> spotStream = env.fromSource(...)
    .map(...)  // 解析 JSON
    .filter(...)  // 过滤无效数据
    // ⭐ 按秒聚合,保留每秒最新的一条记录
    .keyBy(spot -> spot.symbol)
    .window(TumblingEventTimeWindows.of(Time.seconds(1)))
    .reduce((spot1, spot2) -> spot2.timestamp > spot1.timestamp ? spot2 : spot1)
    .name("Aggregate Spot By Second");
```

**合约流聚合**:
```java
DataStream<FuturesPrice> futuresStream = env.fromSource(...)
    .map(...)  // 解析 JSON
    .filter(...)  // 过滤无效数据
    // ⭐ 按秒聚合,保留每秒最新的一条记录
    .keyBy(futures -> futures.symbol)
    .window(TumblingEventTimeWindows.of(Time.seconds(1)))
    .reduce((futures1, futures2) -> futures2.timestamp > futures1.timestamp ? futures2 : futures1)
    .name("Aggregate Futures By Second");
```

### 技术要点

1. **Tumbling Window**: 滚动窗口,每秒一个窗口,窗口之间不重叠
2. **Reduce 函数**: 比较时间戳,保留最新的记录
3. **KeyBy**: 按币种分组,确保每个币种独立聚合
4. **Event Time**: 使用事件时间,确保时间语义正确

## 优化效果

### 数据量减少

假设每秒有 N 条价格数据:

**优化前**:
- 现货流: N 条/秒
- 合约流: N 条/秒
- Join 窗口(±2秒): 5 秒 × N 条 = 5N 条
- Join 数据量: 5N × 5N = 25N² 条

**优化后**:
- 现货流: 1 条/秒 (聚合后)
- 合约流: 1 条/秒 (聚合后)
- Join 窗口(±2秒): 5 秒 × 1 条 = 5 条
- Join 数据量: 5 × 5 = 25 条

**数据量减少**: 25N² → 25,减少了 N² 倍!

### 性能提升

假设 N = 10 (每秒10条数据):
- 优化前: 25 × 10² = 2500 条 Join
- 优化后: 25 条 Join
- 性能提升: 100 倍!

### 业务影响

1. **价格精度**: 保留每秒最新价格,不影响套利计算精度
2. **延迟**: 增加最多 1 秒的聚合延迟,对套利业务影响可忽略
3. **资源消耗**: 大幅降低 CPU 和内存使用

   - 内存使用应该降低

## 后续优化建议

1. **调整窗口大小**: 如果 1 秒聚合延迟太高,可以改为 500ms 或 200ms
2. **使用 ProcessWindowFunction**: 如果需要更复杂的聚合逻辑,可以使用 ProcessWindowFunction
3. **添加监控**: 添加自定义 Metrics 监控聚合前后的数据量变化

## 注意事项

1. **时间语义**: 必须使用 Event Time,确保聚合窗口正确
2. **Watermark**: 确保 Watermark 配置正确,避免数据丢失
3. **延迟**: 聚合会增加 1 秒延迟,需要评估业务影响
4. **状态大小**: 聚合会增加状态大小,但相比 Join 性能提升,这个代价是值得的
