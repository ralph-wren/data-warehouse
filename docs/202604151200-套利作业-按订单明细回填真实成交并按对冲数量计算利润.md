# 套利作业 - 按订单明细回填真实成交并按对冲数量计算利润

## 背景与问题

`FlinkADSArbitrageJob` 的套利流程中，开仓后虽然会异步查询订单详情，但未实现“稳定回填到持仓主状态并用于实时利润计算”的闭环，导致：

1. 利润计算经常退化为估算口径（固定费率 + 目标下单量）
2. 现货/合约部分成交时，结果偏差明显
3. 合约成交量（张）被当作币数量使用，成本计算可能失真

## 本次修复目标

1. 开仓后根据订单ID主动查询订单详情
2. 将真实成交信息回填到 `PositionState`
3. 使用“已对冲数量 + 真实手续费”计算未实现利润
4. 显式输出未对冲敞口，避免误判为完整套利收益

## 代码修改点

## 1) 持仓状态新增真实成交字段

文件：`src/main/java/com/crypto/dw/trading/PositionState.java`

新增字段：

- `actualSpotFilledQuantity`：现货实际成交币数量
- `actualSwapFilledContracts`：合约实际成交张数
- `actualSwapFilledCoin`：合约成交折算币数量（张 × ctVal）

用途：

- 区分“目标下单量”和“真实成交量”
- 为部分成交场景提供精确计算基础

## 2) 主线程按订单ID主动回填真实成交

文件：`src/main/java/com/crypto/dw/processor/TradingDecisionProcessor.java`

新增方法：

- `syncOrderDetailsFromOrderIds(PositionState position)`

核心逻辑：

1. 用持仓里的 `spotOrderId/swapOrderId` 调用 OKX 查询
2. 解析 `avgPx/accFillSz/fee`
3. 合约侧通过 `ctVal` 将张数折算为币数量
4. 回填 `PositionState` 的实际成交价、实际成交量、成本、手续费、总成本、总费用

注意：

- 在 Flink 主线程调用该方法，避免异步线程直接操作 Flink state

## 3) 实时利润改为真实成交口径

文件：`src/main/java/com/crypto/dw/processor/TradingDecisionProcessor.java`

方法：

- `updateUnrealizedProfit(...)`
- `shouldClose(...)`（止损判断同步口径）

新口径：

1. 入场价格优先使用 `actualSpotPrice/actualSwapPrice`
2. 数量使用 `actualSpotFilledQuantity/actualSwapFilledCoin`
3. 已对冲数量：`min(spotFillCoin, swapFillCoin)`
4. 未对冲数量：`abs(spotFillCoin - swapFillCoin)`
5. 手续费优先用 `totalFee`，缺失时回退旧逻辑
6. 利润率优先用真实总成本 `totalCost`

## 4) 修复合约成本计算单位问题

文件：`src/main/java/com/crypto/dw/processor/TradingDecisionProcessor.java`

方法：

- `updateArbitrageOpportunityWithFuturesDetail(...)`

修复前：

- `futuresCost = avgPx * accFillSz`（把“张”当“币”）

修复后：

- `futuresCost = avgPx * accFillSz * ctVal`

## 验证建议

1. 运行套利作业，触发开仓
2. 检查日志中是否出现“按订单ID同步真实成交明细”相关输出
3. 对比 `status_log_*.csv` 与 OKX 实时订单详情：
   - `actualSpotFilledQuantity`
   - `actualSwapFilledContracts`
   - `actualSwapFilledCoin`
   - `totalFee/totalCost`
4. 制造部分成交场景，确认：
   - 未实现利润按已对冲数量计算
   - 日志展示未对冲敞口（`未对冲`）

## 效果总结

1. 利润计算从“估算口径”升级为“真实成交口径”
2. 部分成交场景不再被当成完全对冲仓位计算
3. 成本、投入、手续费、预估利润之间的口径保持一致
4. 为后续做“自动补单/自动减仓”处理未对冲敞口打好基础
