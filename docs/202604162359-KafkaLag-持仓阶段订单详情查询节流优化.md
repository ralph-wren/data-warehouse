# Kafka Lag 排查与优化：持仓阶段订单详情查询节流

## 问题现象

- Kafka lag 持续升高，消费速率追不上生产速率。
- 套利作业在持仓期间吞吐明显下降，实时性变差。

## 根因分析

在 `TradingDecisionProcessor#processElement` 的持仓分支中，会调用 `syncOrderDetailsFromOrderIds`。

该方法会同步调用两次 OKX 订单详情查询（现货 + 合约）。  
当行情 tick 频率较高时，这个外部 I/O 会在 Flink 主线程上被高频触发，导致：

- 单条记录处理时延被拉高；
- 上游 Kafka Source 出现反压，lag 累积；
- 整体链路吞吐受限于外部 API RT。

## 本次优化

### 1) 增加回填节流配置

新增配置项：

- `arbitrage.trading.order-detail-sync-interval-ms`

已在以下配置文件中补齐：

- `src/main/resources/config/application-dev.yml`
- `src/main/resources/config/application-docker.yml`
- `src/main/resources/config/application-prod.yml`

当前默认值：`5000` ms。

### 2) 主线程回填改为按间隔触发

在 `TradingDecisionProcessor` 增加 `lastOrderDetailsSyncTime` 状态与 `shouldSyncOrderDetails()` 判定。

持仓分支中仅在满足最小间隔后才执行 `syncOrderDetailsFromOrderIds`，避免每条 tick 都触发外部查询。

## 预期收益

- 显著降低持仓阶段外部 API 调用频率；
- 减少 Flink 主线程阻塞时间；
- 缓解上游反压与 Kafka lag 堆积；
- 保持订单详情回填能力，同时将其从“每条触发”变为“节流触发”。

## 建议验证

1. 观察 Flink Web UI 指标（busy/backpressured）是否下降。  
2. 观察同一时段 Kafka lag 曲线是否趋稳并回落。  
3. 如 lag 仍高，可进一步把 `order-detail-sync-interval-ms` 调大到 `8000~15000` 进行压测对比。
