### 4. HTTP 100-continue 错误 ✅

**时间**: 2026-03-18 23:01

**问题描述**: 
```json
{"status":"FAILED","msg":"There is no 100-continue header"}
```

**根本原因**: 
- Java `HttpURLConnection` 自动添加 `Expect: 100-continue` 头
- Doris Stream Load API 不支持此机制

**解决方案**: 显式禁用 Expect 头
```java
conn.setRequestProperty("Expect", "");
```

**详细文档**: [202603182301-HTTP请求-解决100-continue错误.md](./202603182301-HTTP请求-解决100-continue错误.md)

### 5. Connection Reset 连接重置错误 ⚠️ (已有更好方案)

**时间**: 2026-03-18 23:10

**问题描述**: 
```
java.net.SocketException: Connection reset
    at java.net.SocketInputStream.read(SocketInputStream.java:186)
    at java.net.HttpURLConnection.getResponseCode(HttpURLConnection.java:527)
```

**根本原因**: 
- Doris 4.0.1-rc02 版本的 Stream Load API 存在 bug
- HTTP 协议处理不稳定
- 自定义 HTTP 实现难以适配版本差异

**临时方案**: 
1. 增加超时时间 (连接 60s, 读取 120s)
2. 禁用连接复用 `Connection: close`
3. 添加重试机制 (最多重试 3 次,指数退避)

**最终方案**: 使用官方 Doris Flink Connector (见问题 6)

**详细文档**: [202603182310-连接重置-解决Doris写入Connection-reset错误.md](./docs/202603182310-连接重置-解决Doris写入Connection-reset错误.md)

### 6. Doris Stream Load Bug (官方 Connector 方案) ✅

**时间**: 2026-03-18 23:45

**问题描述**: 
- 自定义 HTTP Stream Load 持续失败
- curl 测试也出现相同问题
- 无论如何设置 HTTP 头都无法解决

**根本原因**: 
- Doris 4.0.1-rc02 是 RC 版本,不稳定
- Stream Load API 的 HTTP 协议处理有 bug
- 自定义实现难以处理版本兼容性

**解决方案**: 使用官方 Doris Flink Connector
```java
DorisSink<String> dorisSink = DorisSink.<String>builder()
    .setDorisOptions(dorisOptions)
    .setDorisExecutionOptions(executionOptions)
    .setSerializer(new SimpleStringSerializer())
    .build();
```

**优势**:
- 官方维护,兼容性好
- 自动处理 HTTP 协议问题
- 内置重试和错误处理
- 支持多版本 Doris

**详细文档**: [202603182345-Doris写入-使用官方Connector解决Stream-Load-Bug.md](./docs/202603182345-Doris写入-使用官方Connector解决Stream-Load-Bug.md)

## 核心解决方案

### 1. 自定义 DorisStreamLoadSink

**位置**: `src/main/java/com/crypto/dw/flink/sink/DorisStreamLoadSink.java`

**核心功能**:
- 直接使用 HTTP Stream Load API
- 批量写入优化 (支持批量大小和时间间隔触发)
- 完善的错误处理和日志输出
- 禁用 Expect: 100-continue 头
- 连接重置重试机制

**关键代码**:
```java
// 禁用 100-continue
conn.setRequestProperty("Expect", "");

// 禁用连接复用
conn.setRequestProperty("Connection", "close");

// 增加超时时间
conn.setConnectTimeout(60000);   // 60 秒
conn.setReadTimeout(120000);     // 120 秒

// 批量写入
if (buffer.size() >= batchSize || 
    (currentTime - lastFlushTime) >= batchIntervalMs) {
    flush();
}

// 重试机制
for (int retry = 0; retry < maxRetries; retry++) {
    try {
        doFlush(recordCount);
        return;
    } catch (java.net.SocketException e) {
        logger.warn("连接被重置,第 {} 次重试", retry + 1);
        Thread.sleep(1000 * retry);  // 指数退避
    }
}

// 错误流处理
try (BufferedReader br = new BufferedReader(
        new InputStreamReader(
            responseCode >= 400 ? conn.getErrorStream() : conn.getInputStream(), 
            StandardCharsets.UTF_8))) {
    // 读取响应
}
```

## 性能优化建议

### 1. 批量写入参数

```yaml
doris:
  stream-load:
    batch-size: 1000        # 批量大小,建议 500-2000
    batch-interval-ms: 5000 # 批量间隔,建议 3000-10000
```

**调优建议**:
- 数据量大: 增加 batch-size,减少网络请求
- 延迟敏感: 减少 batch-interval-ms,加快数据写入
- 资源有限: 减少 batch-size,降低内存占用

### 2. Flink 并行度

```yaml
flink:
  execution:
    parallelism: 4  # 根据 CPU 核心数调整
```

**调优建议**:
- 并行度 = CPU 核心数 × 1-2
- 数据倾斜时适当增加并行度
- 资源有限时适当减少并行度

### 3. Checkpoint 配置

```yaml
flink:
  checkpoint:
    interval: 60000  # Checkpoint 间隔 (ms)
```

**调优建议**:
- 数据一致性要求高: 减少间隔 (30s-60s)
- 性能优先: 增加间隔 (60s-300s)
- 状态较大: 适当增加间隔

### 8. Docker 网络 - Flink 连接 Doris BE 地址问题 ✅ ⭐⭐⭐ (最终成功)

**时间**: 2026-03-19 01:00

**问题描述**:
```
org.apache.http.NoHttpResponseException: 172.19.0.3:8040 failed to respond
    at org.apache.doris.flink.sink.writer.DorisStreamLoad.abortPreCommit
```

**问题现象**:
- Flink 使用官方 Doris Connector 时无法连接 BE
- 错误显示尝试连接 Docker 内部 IP `172.19.0.3:8040`
- 宿主机无法访问 Docker 桥接网络的内部 IP

**根本原因**:
1. Doris 使用 Docker 桥接网络启动,BE 注册地址为 Docker 内部 IP
2. 官方 Doris Flink Connector 会自动从 FE 查询 BE 节点列表
3. FE 返回的是 Docker 内部 IP,宿主机无法访问
4. Host 网络模式在 Windows/Mac 上支持不完整,配置复杂
5. BE 持久化数据中保存了旧的地址信息,导致冲突

**最终解决方案**: Docker 桥接网络 + 端口映射 + 配置 BE 节点 ⭐

**关键配置**:

1. **Docker Compose** (`docker-compose-doris.yml`):
```yaml
services:
  doris-fe:
    image: apache/doris:fe-3.0.7-rc01-1204
    environment:
      - FE_SERVERS=fe1:172.25.0.2:9010
      - FE_ID=1
      - PRIORITY_NETWORKS=172.25.0.0/16  # 关键参数
    networks:
      doris-net:
        ipv4_address: 172.25.0.2
    ports:
      - "8030:8030"
      - "9030:9030"
      - "9010:9010"

  doris-be:
    image: apache/doris:be-3.0.7-rc01-1204
    environment:
      - FE_SERVERS=fe1:172.25.0.2:9010
      - BE_ADDR=172.25.0.3:9050
      - PRIORITY_NETWORKS=172.25.0.0/16  # 关键参数
    networks:
      doris-net:
        ipv4_address: 172.25.0.3
    ports:
      - "8040:8040"
      - "9050:9050"
      - "8060:8060"
      - "9060:9060"
```

2. **Flink 配置** (`config/application.yml`):
```yaml
doris:
  fe:
    http-url: http://127.0.0.1:8030
    jdbc-url: jdbc:mysql://127.0.0.1:9030
  be:
    nodes: "127.0.0.1:8040"  # 通过端口映射访问
  database: crypto_dw
```

**部署步骤**:
```bash
# 1. 完全重置 (清理旧数据)
./reset-doris-clean.sh

# 2. 启动 Doris
./start-doris-for-flink.sh

# 3. 创建数据库和表
mysql -h 127.0.0.1 -P 9030 -u root -e "CREATE DATABASE IF NOT EXISTS crypto_dw;"
mysql -h 127.0.0.1 -P 9030 -u root < sql/create_tables.sql

# 4. 验证部署
./test/verify-doris-flink-connection.sh
```

**验证结果**:
```
✓ FE 连接成功 (127.0.0.1:9030)
✓ BE 节点在线 (172.25.0.3:9050, Alive: true)
✓ BE HTTP 端口可访问 (127.0.0.1:8040)
✓ BE 磁盘容量正常 (927.261 GB / 1006.854 GB)
✓ 数据库和表创建成功
```

**新增文件**:
- `start-doris-for-flink.sh`: Doris 启动脚本 (自动化部署)
- `reset-doris-clean.sh`: Doris 重置脚本 (清理所有数据)
- `fix-doris-be-address.sh`: BE 地址修复脚本
- `docker-compose-doris.yml`: 桥接网络模式配置
- `test/verify-doris-flink-connection.sh`: 连接验证脚本

**技术要点**:
- `PRIORITY_NETWORKS` 参数指定 Doris 使用的网络接口
- `FE_SERVERS` 必须使用 `name:ip:port` 格式
- 端口映射让 Flink 通过 127.0.0.1 访问 Docker 内部的 BE
- 完全重置数据卷避免持久化数据冲突
- BE 需要 30-60 秒完成磁盘扫描

**解决方案**:

1. **添加 flink-json 依赖**:

```java
// 旧参数 (不支持)

```

3. **添加类型转换**:
```java
// trade_hour: BIGINT -> INT
```

4. **修改 Kafka 消费模式**:

- Flink JSON 格式依赖提供 JSON 序列化/反序列化功能

- Flink SQL 类型推断规则:
  - `EXTRACT(HOUR ...)` 返回 `BIGINT`
  - 算术运算会提升精度 (如 DECIMAL(20,8) + DECIMAL(20,8) = DECIMAL(21,8))
  - 除法运算大幅提升精度 (如 DECIMAL(20,8) / DECIMAL(20,8) = DECIMAL(38,17))
- Kafka 消费模式:
  - `earliest-offset`: 从最早的数据开始 (适合处理历史数据)
  - `latest-offset`: 从最新的数据开始 (适合实时处理)

### 12. Checkpoint 卡住 - 分区不匹配和数据质量问题 ✅ ⭐⭐⭐

3. **Checkpoint 机制**:
   - Doris Connector 在 Checkpoint 时会 flush 所有缓冲的数据

### 16. 数据流修复 - Kafka 消费模式导致无新数据问题 ✅ ⭐⭐⭐

**时间**: 2026-03-20 01:40

**问题描述**:
- Doris 表数据在增长，但都是旧数据（最新时间停留在几小时前）
- Flink Web UI 显示 Records Received = 0
- Grafana 监控显示无数据流入
- Kafka 消费者组显示 "has no active members"
- Kafka LAG 持续增加

**问题现象**:
```
Kafka Offset: 70000+ (持续增长)
Flink Current Offset: 70000 (不变)
Kafka LAG: 98 → 247 (持续增加)
Doris 数据: 不增长或增长缓慢
Records Received: 0
```

**根本原因**: ⭐⭐⭐

**Kafka Source 配置使用了 `OffsetsInitializer.latest()` 消费模式**

```java
// 问题代码
KafkaSource.<String>builder()
    .setStartingOffsets(OffsetsInitializer.latest())  // ❌ 只消费新数据
    .build();
```

**问题说明**:
- `latest()` 模式：只消费 Flink 作业启动**之后**产生的新数据
- 数据采集器在 Flink 作业启动**之前**就已经运行
- Kafka 中已经有大量历史数据（Offset 0 → 70000+）
- Flink 启动后，从 latest offset 开始消费，但此时没有新数据产生
- 导致 Flink 一直等待新数据，Records Received = 0

**解决方案**:

**1. 修改消费模式为 earliest（推荐）✅**

```java
/**
 * 创建 Kafka Source
 * 
 * 消费模式说明：
 * - earliest: 从最早的数据开始消费（适合处理历史数据）
 * - latest: 从最新的数据开始消费（适合实时处理）
 * - committed: 从上次提交的 offset 开始消费
 */
private static KafkaSource<String> createKafkaSource(ConfigLoader config) {
    // 读取消费模式配置，默认使用 earliest（从头开始消费）
    String startupMode = config.getString("kafka.consumer.startup-mode", "earliest");
    
    // 根据配置选择消费模式
    OffsetsInitializer offsetsInitializer;
    switch (startupMode.toLowerCase()) {
        case "latest":
            offsetsInitializer = OffsetsInitializer.latest();
            break;
        case "committed":
            offsetsInitializer = OffsetsInitializer.committedOffsets();
            break;
        case "earliest":
        default:
            offsetsInitializer = OffsetsInitializer.earliest();
            break;
    }
    
    return KafkaSource.<String>builder()
        .setStartingOffsets(offsetsInitializer)  // ✅ 使用配置的消费模式
        .build();
}
```

**2. 添加配置** (`config/application-dev.yml`):

```yaml
kafka:
  consumer:
    startup-mode: earliest  # 从头开始消费（开发环境）
    # startup-mode: committed  # 从上次提交位置开始（生产环境推荐）
    # startup-mode: latest  # 从最新数据开始（实时监控场景）
```

**验证结果**:

修复前 ❌:
```

Doris 数据: 不增长
Records Received: 0
```

修复后 ✅:
```

检查 3: LAG=24 ✅, Doris=194,315 ✅, 时间=17:39:00 ✅
检查 8: LAG=66 ✅, Doris=194,686 ✅, 时间=17:40:00 ✅
```

**Kafka 消费模式对比**:

| 模式 | 说明 | 适用场景 | 优点 | 缺点 |
|-----|------|---------|------|------|
| **earliest** | 从最早的数据开始 | 开发环境、数据回溯 | 不丢数据 | 首次启动慢 |
| **latest** | 从最新的数据开始 | 实时处理 | 启动快 | 丢失历史数据 |
| **committed** | 从上次提交位置开始 | 生产环境 | 断点续传 | 首次启动需要 offset |

**技术要点**:
- Kafka 消费模式直接影响数据处理行为
- `earliest` 适合开发环境，确保不丢数据
- `committed` 适合生产环境，支持断点续传
- `latest` 适合实时监控，不关心历史数据
- 配置要可调整，不要硬编码

**1. 创建 WatermarkStrategyFactory 工厂类**

创建了 `src/main/java/com/crypto/dw/flink/watermark/WatermarkStrategyFactory.java`，提供以下功能：

**主要方法**:

1. **无 Watermark 策略**（当前 ODS 层使用）
   ```java
   WatermarkStrategy<String> strategy = WatermarkStrategyFactory.noWatermarks();
   ```
   - 适用场景：不需要基于事件时间的窗口计算
   - ODS 层数据采集使用此策略

2. **有界乱序 Watermark（字符串类型）**
   ```java
   WatermarkStrategy<String> strategy = 
       WatermarkStrategyFactory.forBoundedOutOfOrdernessString(5000); // 5秒乱序
   ```
   - 适用场景：数据可能乱序到达，但延迟在可控范围内
   - 自动从 JSON 字符串中提取 "ts" 字段作为时间戳
   - 推荐用于 DWD/DWS 层实时计算

3. **单调递增 Watermark（字符串类型）**
   ```java
   WatermarkStrategy<String> strategy = 
       WatermarkStrategyFactory.forMonotonousTimestampsString();
   ```
   - 适用场景：数据严格按时间顺序到达
   - 自动从 JSON 字符串中提取 "ts" 字段作为时间戳

4. **通用 Watermark 策略**
   ```java
   WatermarkStrategy<TickerData> strategy = 
       WatermarkStrategyFactory.forBoundedOutOfOrderness(
           5000,
           (element, recordTimestamp) -> element.getTs()
       );
   ```
   - 支持自定义数据类型
   - 需要提供时间戳提取器

5. **空闲检测 Watermark**
   ```java
   WatermarkStrategy<String> baseStrategy = 
       WatermarkStrategyFactory.forBoundedOutOfOrdernessString(5000);
   WatermarkStrategy<String> strategy = 
       WatermarkStrategyFactory.withIdleness(baseStrategy, 60000); // 60秒空闲超时
   ```
   - 适用场景：某些分区可能长时间没有数据
   - 推荐用于多分区 Kafka 场景

**2. 在 FlinkODSJobDataStream 中使用**

**当前代码**（第 99 行）:
```java

    kafkaSource,

    "Kafka Source"
);
```

**修改后**（使用工厂类）:
```java

// 方式1：无 Watermark（当前使用）
DataStream<String> rawStream = env.fromSource(
    kafkaSource,
    WatermarkStrategyFactory.noWatermarks(),
    "Kafka Source"
);

// 方式2：有界乱序 Watermark（推荐用于实时计算）

// 方式3：有界乱序 + 空闲检测（推荐用于多分区场景）
WatermarkStrategy<String> strategy = WatermarkStrategyFactory.withIdleness(
    WatermarkStrategyFactory.forBoundedOutOfOrdernessString(5000),
    60000  // 60秒空闲超时
);

    "Kafka Source"
```

**3. 配置化 Watermark 策略**

可以在 `application.yml` 中添加配置：

```yaml
flink:
  watermark:
    strategy: bounded-out-of-orderness  # 策略类型：none, bounded-out-of-orderness, monotonous
    max-out-of-orderness: 5000          # 最大乱序时间（毫秒）
    idle-timeout: 60000                 # 空闲超时时间（毫秒）
```

然后在代码中读取配置：

```java
String strategyType = config.getString("flink.watermark.strategy", "none");
WatermarkStrategy<String> watermarkStrategy;

switch (strategyType) {
    case "bounded-out-of-orderness":
        long maxOutOfOrderness = config.getLong("flink.watermark.max-out-of-orderness", 5000);
        watermarkStrategy = WatermarkStrategyFactory.forBoundedOutOfOrdernessString(maxOutOfOrderness);
        
        // 如果配置了空闲超时，添加空闲检测
        long idleTimeout = config.getLong("flink.watermark.idle-timeout", 0);
        if (idleTimeout > 0) {
            watermarkStrategy = WatermarkStrategyFactory.withIdleness(watermarkStrategy, idleTimeout);
        }
        break;
        
    case "monotonous":
        watermarkStrategy = WatermarkStrategyFactory.forMonotonousTimestampsString();
        break;
        
    case "none":
    default:
        watermarkStrategy = WatermarkStrategyFactory.noWatermarks();
        break;
}

DataStream<String> rawStream = env.fromSource(
    kafkaSource,
    watermarkStrategy,
    "Kafka Source"
);
```

### 22. 监控过滤 - Grafana 面板只显示活跃 Job ✅ ⭐

**时间**: 2026-03-20 21:50

**问题描述**:
当前的 Grafana Flink 监控面板显示所有 job 的指标，包括已经 kill 的 job。这会导致面板中显示大量无用的历史数据，难以区分当前运行的作业。

**问题分析**:

Flink 的 Prometheus 指标包含 `job_id` 标签：
- 运行中的 job 有 `job_id` 标签
- 已 kill 的 job 的指标会逐渐消失
- 通过过滤 `job_id!=""` 可以只显示活跃的 job

**解决方案**:

在所有 job 相关的 Prometheus 查询中添加 `{job_id!=""}` 过滤条件。

**修改的查询（8 个）**:

1. **数据流入速率**
   ```promql
   # 修改前
   rate(flink_taskmanager_job_task_numRecordsIn[1m])
   
   # 修改后
   rate(flink_taskmanager_job_task_numRecordsIn{job_id!=""}[1m])
   ```

2. **数据流出速率**
   ```promql
   rate(flink_taskmanager_job_task_numRecordsOut{job_id!=""}[1m])
   ```

3. **Checkpoint 持续时间**
   ```promql
   flink_jobmanager_job_lastCheckpointDuration{job_id!=""}
   ```

4. **Checkpoint 完成次数**
   ```promql
   flink_jobmanager_job_numberOfCompletedCheckpoints{job_id!=""}
   ```

5. **Checkpoint 失败次数**
   ```promql
   flink_jobmanager_job_numberOfFailedCheckpoints{job_id!=""}
   ```

6. **输出缓冲池使用率**
   ```promql
   flink_taskmanager_job_task_buffers_outPoolUsage{job_id!=""}
   ```

7. **网络字节流入**
   ```promql
   rate(flink_taskmanager_job_task_numBytesIn{job_id!=""}[1m])
   ```

8. **网络字节流出**
   ```promql
   rate(flink_taskmanager_job_task_numBytesOut{job_id!=""}[1m])
   ```

**不需要过滤的指标**（集群级别）:
- `flink_jobmanager_numRunningJobs`：运行中的作业数
- `flink_jobmanager_numRegisteredTaskManagers`：TaskManager 数量
- `flink_jobmanager_taskSlotsAvailable`：可用任务槽
- `flink_jobmanager_taskSlotsTotal`：总任务槽
- `flink_taskmanager_Status_JVM_*`：JVM 指标

**Prometheus 标签过滤语法**:

```promql
# 等于
metric{label="value"}

# 不等于
metric{label!="value"}

# 正则匹配
metric{label=~"regex"}

# 正则不匹配
metric{label!~"regex"}

# 多个条件（AND）
metric{label1="value1", label2="value2"}
```

```bash

# 查看 Grafana 面板（应该看到作业指标）

# Kill 作业
JOB_ID=$(curl -s http://localhost:8081/jobs | jq -r '.jobs[0].id')
curl -X PATCH http://localhost:8081/jobs/$JOB_ID?mode=cancel
sleep 120  # 等待指标消失

# 查看 Grafana 面板（作业指标应该消失）
```

**技术要点**:

1. **job_id 标签的生命周期**:
   ```
   作业启动 → job_id 出现 → 指标推送
   作业运行 → job_id 存在 → 指标持续
   作业停止 → job_id 消失 → 指标停止
   ```

2. **为什么使用 job_id!=""**:
   - 简单有效，只显示活跃 job
   - 自动更新，无需手动维护
   - 不需要硬编码作业名称

3. **Pushgateway 注意事项**:
   - 使用 Pushgateway 时，指标会保留
   - 需要手动清理：`curl -X DELETE http://localhost:9091/metrics/job/flink-ods-job`

**优势**:

1. **清晰的监控视图**：只显示当前运行的作业
2. **自动更新**：作业启动/停止时自动更新
3. **性能优化**：减少查询数据量
4. **易于维护**：不需要手动清理

### 24. 监控修复 - 解决 Kafka 和 Doris 监控数据显示问题 ✅ ⭐⭐⭐

**时间**: 2026-03-20 23:10

**问题描述**:
在配置了 Kafka 和 Doris 监控面板后,Grafana 中面板显示 "No data",需要排查并修复监控数据采集和显示问题。

**问题现象**:
1. Prometheus 能查询到 Kafka 和 Doris 指标
2. Grafana Explore 中能查询到数据
3. 但 Grafana 监控面板显示 "No data"

**根本原因**: ⭐⭐⭐

**1. Kafka Exporter 指标限制**:
- 使用的 `danielqsj/kafka-exporter` 只提供基础指标
- 不提供 `kafka_topic_partition_current_offset`、`kafka_consumergroup_lag` 等高级指标
- 面板查询使用了不存在的指标,导致无数据

**2. Topic 名称错误**:
- 面板中使用 `topic="crypto-ticker"` (连字符)
- 实际 topic 名称是 `crypto_ticker` (下划线)

**3. 查询语句错误**:
- 对 gauge 类型指标使用 `rate()` 函数
- `rate()` 只适用于 counter 类型指标

**解决方案**:

**方案 1: 使用简化版监控面板** ✅ (当前方案)

创建只使用实际可用指标的简化版面板:

```json
// monitoring/grafana/dashboards/kafka-monitoring-simple.json
{
  "panels": [
    {
      "title": "Kafka Brokers",
      "targets": [{"expr": "kafka_brokers"}]
    },
    {
      "title": "Topic 数量",
      "targets": [{"expr": "count(kafka_topic_partitions)"}]
    },
    {
      "title": "总分区数",
      "targets": [{"expr": "sum(kafka_topic_partitions)"}]
    },
    {
      "title": "crypto_ticker 分区数",
      "targets": [{"expr": "kafka_topic_partitions{topic=\"crypto_ticker\"}"}]
    },
    {
      "title": "分区同步副本数",
      "targets": [{"expr": "kafka_topic_partition_in_sync_replica{topic=\"crypto_ticker\"}"}]
    },
    {
      "title": "分区 Leader 分布",
      "targets": [{"expr": "kafka_topic_partition_leader{topic=\"crypto_ticker\"}"}]
    }
  ]
}
```

**可用的 Kafka 指标**:
- `kafka_brokers`: Broker 数量
- `kafka_topic_partitions`: Topic 分区数
- `kafka_topic_partition_in_sync_replica`: 同步副本数
- `kafka_topic_partition_leader`: 分区 Leader
- `kafka_topic_partition_replicas`: 副本数
- `kafka_topic_partition_under_replicated_partition`: 未充分复制的分区

**方案 2: 升级到 JMX Exporter** (未来优化)

要获得完整的 Kafka 监控(消息速率、LAG、吞吐量等),需要:

1. **启用 Kafka JMX**:
```yaml
# docker-compose.yml (Kafka 配置)
environment:
  - KAFKA_JMX_OPTS=-Dcom.sun.management.jmxremote -Dcom.sun.management.jmxremote.port=9999 -Dcom.sun.management.jmxremote.authenticate=false -Dcom.sun.management.jmxremote.ssl=false
  - JMX_PORT=9999
ports:
  - "9999:9999"
```

2. **添加 JMX Exporter**:
```yaml
# docker-compose-monitoring.yml
jmx-exporter:
  image: bitnami/jmx-exporter:latest
  ports:
    - "5556:5556"
  environment:
    - SERVICE_PORT=5556
  volumes:
    - ./monitoring/jmx-exporter/kafka.yml:/etc/jmx-exporter/config.yml
  command:
    - "5556"
    - "/etc/jmx-exporter/config.yml"
```

3. **配置 JMX Exporter**:
```yaml
# monitoring/jmx-exporter/kafka.yml
lowercaseOutputName: true
lowercaseOutputLabelNames: true
rules:
  - pattern: kafka.server<type=(.+), name=(.+)><>Value
    name: kafka_server_$1_$2
  - pattern: kafka.server<type=(.+), name=(.+), topic=(.+)><>Value
    name: kafka_server_$1_$2
    labels:
      topic: "$3"
```

4. **更新 Prometheus 配置**:
```yaml
scrape_configs:
  - job_name: 'kafka-jmx'
    static_configs:
      - targets: ['jmx-exporter:5556']
```

```
Prometheus: 有数据
Explore: 能查询到数据
```

### 17. 监控升级 - Kafka JMX 详细监控实现 ✅ ⭐⭐⭐

**时间**: 2026-03-20 23:20

**问题描述**:
- 简化版 Kafka 监控只有基础指标（Broker 数量、Topic 数量、分区数等）
- 缺少关键的详细监控指标：
  - ❌ 消息流入/流出速率
  - ❌ 字节吞吐量
  - ❌ 请求延迟和性能
  - ❌ 消费者 LAG
  - ❌ JVM 和系统资源

**根本原因**:
- 当前使用的 `danielqsj/kafka-exporter` 只提供基础指标
- 不提供高级指标如消息速率、LAG、吞吐量等
- 需要通过 Kafka 的 JMX 接口获取完整的监控指标

**解决方案**: 升级到 JMX Exporter 架构 ⭐⭐⭐

**架构设计**:
```
Kafka (JMX enabled on port 9999)
    ↓
JMX Exporter (port 5556) ← 转换 JMX 指标为 Prometheus 格式
    ↓
Prometheus (port 9090) ← 抓取指标
    ↓
Grafana (port 3000) ← 可视化展示
```

**双层监控策略**:

保留两个 Exporter 以获得最全面的监控:

1. **Kafka Exporter** (danielqsj/kafka-exporter)
   - 提供 Topic 级别的基础指标
   - 轻量级，无需配置
   - 端口: 9308

2. **JMX Exporter** (bitnami/jmx-exporter)
   - 提供详细的性能指标
   - 需要 Kafka 启用 JMX
   - 端口: 5556

**实施步骤**:

**1. 创建 JMX Exporter 配置文件**

文件: `monitoring/jmx-exporter/kafka.yml`

配置了以下指标类别:
- **Broker Topic 指标**: 消息流入/流出速率、字节吞吐量
- **Replica Manager 指标**: 分区数、Leader 数、未充分复制的分区
- **Request 指标**: 请求延迟、队列时间、处理时间
- **Log Flush 指标**: 日志刷新性能
- **Controller 指标**: 控制器状态、Leader 选举
- **JVM 指标**: CPU、内存、GC

**2. 更新 docker-compose-monitoring.yml**

添加 JMX Exporter 服务:
```yaml
jmx-exporter:
  image: bitnami/jmx-exporter:latest
  container_name: kafka-jmx-exporter
  ports:
    - "5556:5556"
  volumes:
    - ./monitoring/jmx-exporter/kafka.yml:/etc/jmx-exporter/config.yml:ro
  command:
    - "5556"
    - "/etc/jmx-exporter/config.yml"
    - "host.docker.internal:9999"
```

**3. 更新 Prometheus 配置**

添加 JMX Exporter 抓取任务:
```yaml
- job_name: 'kafka-jmx'
  static_configs:
    - targets: ['jmx-exporter:5556']
```

**4. 重启 Kafka 并启用 JMX**

关键环境变量:
```bash
KAFKA_JMX_OPTS="-Dcom.sun.management.jmxremote \
  -Dcom.sun.management.jmxremote.port=9999 \
  -Dcom.sun.management.jmxremote.authenticate=false \
  -Dcom.sun.management.jmxremote.ssl=false"
JMX_PORT=9999
```

**5. 创建完整版监控面板**

文件: `monitoring/grafana/dashboards/kafka-monitoring-advanced.json`

包含 20 个监控面板，分为 5 个类别:

**集群概览** (6 个面板):

### 18. Windows Git Bash 路径转换问题 ✅

**时间**: 2026-03-20 23:30

**问题描述**:
```
OCI runtime exec failed: exec failed: unable to start container process: 
exec: "C:/Program Files/Git/opt/kafka/bin/kafka-topics.sh": 
stat C:/Program Files/Git/opt/kafka/bin/kafka-topics.sh: no such file or directory
```

**问题现象**:
- 在 Windows Git Bash 中运行 Docker exec 命令失败
- 路径 `/opt/kafka/bin/kafka-topics.sh` 被转换为 `C:/Program Files/Git/opt/kafka/bin/kafka-topics.sh`
- 这个 Windows 路径在 Docker 容器内部不存在

**根本原因**:
- Git Bash (MSYS2) 会自动将 POSIX 风格的路径转换为 Windows 路径
- `/path/to/file` → `C:/Program Files/Git/path/to/file`
- 当路径是 Docker 容器内部的路径时，转换会导致错误

**解决方案**: 使用 `bash -c` + `env -i` 清除环境变量 ✅

```bash
# ❌ 错误方式 (路径和命令名都错误)
docker exec kafka /opt/kafka/bin/kafka-topics.sh --list --bootstrap-server localhost:9092

# ⚠️ 部分正确 (命令名正确，但有 JMX 端口冲突)
docker exec kafka bash -c "kafka-topics --list --bootstrap-server localhost:9092"
# 错误: JMX connector server communication error: Port already in use: 9999

# ✅ 完全正确 (使用 env -i 清除环境变量，避免 JMX 冲突)
docker exec kafka bash -c "env -i PATH=/usr/bin kafka-topics --list --bootstrap-server localhost:9092"
```

**原理**:
- `bash -c "command"` 将整个命令作为字符串传递，避免 Git Bash 路径转换
- `env -i` 清除所有环境变量，避免继承容器的 JMX 配置
- `PATH=/usr/bin` 只保留必要的 PATH 环境变量
- 命令在容器内的 bash 中执行，路径保持不变，且不会尝试启动 JMX

**重要发现**: ⭐⭐⭐
- Confluent Kafka Docker 镜像 (confluentinc/cp-kafka:7.6.0) 的命令结构：
  - 命令位置: `/usr/bin/` (不是 `/opt/kafka/bin/`)
  - 命令名称: `kafka-topics` (没有 `.sh` 后缀)
  - 已在 PATH 中，可以直接调用
  
- **JMX 端口冲突问题**:
  - 容器启动时已设置 `KAFKA_JMX_OPTS` 和 `JMX_PORT=9999`
  - Kafka 命令工具会读取这些环境变量并尝试启动 JMX
  - 导致端口 9999 冲突：`Port already in use: 9999`
  - 解决方案：使用 `env -i` 清除所有环境变量，只保留 `PATH`

**其他方案**:

**方案 2: 使用双斜杠**
```bash
docker exec kafka //opt/kafka/bin/kafka-topics.sh --list
```

**方案 3: 使用环境变量**
```bash
MSYS_NO_PATHCONV=1 docker exec kafka /opt/kafka/bin/kafka-topics.sh --list
```

**修复的文件**:
- `restart-kafka-with-jmx.sh` - Kafka 重启脚本
- `kafka-setup.sh` - Kafka 设置脚本

**跨平台兼容性**:
- `bash -c` 方法在 Linux/Mac 上也能正常工作
- 是跨平台脚本的最佳实践

**关键经验**:

| 路径格式 | Git Bash 转换 | 用途 |
|---------|--------------|------|
| `/path/to/file` | `C:/Program Files/Git/path/to/file` | Windows 程序参数 |
| `/c/path/to/file` | `C:/path/to/file` | Windows 绝对路径 |
| `//path/to/file` | 不转换 | 网络路径或禁用转换 |
| `"command"` | 字符串内部不转换 | 命令字符串 |

### 19. JMX 连接 - Docker 网络 RMI 回调问题 ✅ ⭐⭐⭐

**时间**: 2026-03-21 00:10

**问题描述**:
```
java.rmi.ConnectException: Connection refused to host: host.docker.internal; 
nested exception is: java.net.ConnectException: Connection refused
```

**问题现象**:
- JMX Exporter HTTP 服务器启动成功（端口 5556）
- 但无法连接到 Kafka JMX（端口 9999）
- JMX Exporter 指标端点只有自身的 JVM 指标，没有 Kafka 指标
- TCP 连接测试成功，但 RMI 连接失败

**根本原因**: ⭐⭐⭐

**Java RMI (Remote Method Invocation) 的两阶段连接机制**:

1. **第一阶段**: 客户端连接到 RMI Registry（端口 9999）
2. **第二阶段**: RMI Registry 返回 RMI Server 的地址，客户端连接到 RMI Server

**问题所在**:
- Kafka 的 `java.rmi.server.hostname` 设置不正确
- RMI Registry 返回的地址是 JMX Exporter 容器无法访问的地址
- 导致第二阶段连接失败

**尝试的方案**:

1. **使用 host.docker.internal** ❌
   - `host.docker.internal` 在 Docker 容器内部无法正确解析
   - RMI 回调时仍然失败

2. **使用 localhost** ❌
   - RMI Registry 返回 `localhost`
   - JMX Exporter 容器中的 `localhost` 指向容器自己，而不是 Kafka

**最终解决方案**: Docker 网络 + 容器名称 ✅ ⭐⭐⭐

**核心思路**: 让 Kafka 和 JMX Exporter 在同一个 Docker 网络中，使用容器名称进行通信

**1. 修改 Kafka JMX 配置** (`restart-kafka-with-jmx.sh`):
```bash
# 关键环境变量
-e KAFKA_JMX_HOSTNAME=kafka \  # Confluent 镜像特有的环境变量
-e KAFKA_JMX_PORT=9999 \
-e KAFKA_JMX_OPTS="-Dcom.sun.management.jmxremote \
    -Dcom.sun.management.jmxremote.port=9999 \
    -Dcom.sun.management.jmxremote.rmi.port=9999 \
    -Dcom.sun.management.jmxremote.authenticate=false \
    -Dcom.sun.management.jmxremote.ssl=false \
    -Djava.rmi.server.hostname=kafka \  # 使用容器名称
    -Djava.net.preferIPv4Stack=true"
```

**2. 修改 JMX Exporter 配置** (`monitoring/jmx-exporter/kafka.yml`):
```yaml
# JMX 连接配置
# 使用 Kafka 容器名称直接连接（需要在同一 Docker 网络中）
hostPort: kafka:9999
```

**3. 连接 Kafka 到监控网络**:
```bash
# 在启动脚本中添加
echo "步骤 2.5: 连接 Kafka 到监控网络..."
if docker network ls | grep -q "monitoring-net"; then
    docker network connect data-warehouse_monitoring-net kafka 2>/dev/null || \
        echo "✓ Kafka 已在监控网络中"
    echo "✓ Kafka 已连接到监控网络"
fi
```

**关键经验**:

1. **RMI 连接需要正确的 hostname**:
   - `java.rmi.server.hostname` 必须是客户端可以访问的地址
   - 在 Docker 环境中，使用容器名称最可靠
   - 不要使用 `localhost` 或 `127.0.0.1`

2. **Docker 网络是关键**:
   - 容器间通信应该使用 Docker 网络

### 22. 监控增强 - 添加 Doris 详细监控指标 ✅ ⭐⭐⭐

**时间**: 2026-03-21 02:16

**需求描述**:
用户要求为 Doris 监控添加更详细、更必要的监控指标，以便更全面地了解 Doris 的运行状态和性能表现。

**实施方案**:

**1. 创建增强版监控面板**

创建了 `monitoring/grafana/dashboards/doris-monitoring-enhanced.json`，包含 38 个监控面板（原有 13 个，新增 25 个）。

**2. 新增监控指标分类**

#### FE (Frontend) 指标

**查询性能** (面板 1-5):
- `doris_fe_qps`: 每秒查询数
- `doris_fe_rps`: 每秒请求数
- `doris_fe_query_latency_ms`: 查询延迟（P95/P99）
- `doris_fe_query_err_rate`: 查询错误率 ⭐ 新增
- `doris_fe_query_err`: 查询错误总数 ⭐ 新增
- `doris_fe_query_instance_num`: 查询实例数 ⭐ 新增

**连接管理** (面板 6-7):
- `doris_fe_connection_total`: 连接总数（按用户分组）⭐ 新增
- `sum(doris_fe_connection_total)`: 活跃连接总数 ⭐ 新增

**Edit Log** (面板 8-9):
- `doris_fe_edit_log`: Edit Log 大小 ⭐ 新增
- `doris_fe_edit_log_write_latency_ms`: Edit Log 写入延迟 ⭐ 新增

**RPC 性能** (面板 10-11):
- `doris_fe_thrift_rpc_latency_ms`: Thrift RPC 延迟 ⭐ 新增
- `rate(doris_fe_thrift_rpc_total[1m])`: RPC 请求速率 ⭐ 新增

**线程池** (面板 12-14):
- `doris_fe_thread_pool{type="task_num"}`: 线程池任务数 ⭐ 新增
- `doris_fe_thread_pool{type="queue_size"}`: 线程池队列大小 ⭐ 新增
- `doris_fe_thread_pool{type="active_thread_num"}`: 活跃线程数 ⭐ 新增

**Cache** (面板 15-16):
- `doris_fe_cache_hit / (doris_fe_cache_hit + doris_fe_cache_miss)`: Cache 命中率 ⭐ 新增
- `rate(doris_fe_cache_added[1m])`: Cache 添加速率 ⭐ 新增

**事务** (面板 17-18):
- `rate(doris_fe_txn_counter[1m])`: 事务速率（begin/success/failed）
- `doris_fe_txn_num`: 运行中事务数
- `doris_fe_publish_txn_num`: 发布中事务数

**JVM** (面板 19-20):
- `jvm_heap_size_bytes`: JVM 堆内存使用
- `rate(jvm_gc[1m])`: GC 频率

**集群状态** (面板 21-26):
- `node_info{type="is_master"}`: FE Master 状态
- `node_info{type="be_node_num",state="alive"}`: BE 节点数
- `doris_fe_internal_database_num`: 数据库数量
- `doris_fe_internal_table_num`: 表数量
- `doris_fe_tablet_status_count`: Tablet 状态统计

#### BE (Backend) 指标

**Stream Load** (面板 27-28):
- `rate(doris_be_stream_load_txn_request[1m])`: Stream Load 请求速率 ⭐ 新增
- `doris_be_stream_load_current_processing`: 当前处理中的 Stream Load ⭐ 新增

**Compaction** (面板 29-31):
- `doris_be_compaction_task_state_total`: Compaction 任务状态 ⭐ 新增
- `rate(doris_be_compaction_rows_total[1m])`: Compaction 处理行数 ⭐ 新增
- `doris_be_compaction_max_score`: 最大 Compaction Score ⭐ 新增

**Cache** (面板 32):
- `doris_be_cache_hit_ratio`: BE Cache 命中率 ⭐ 新增

**网络** (面板 33-34):
- `rate(doris_be_network_send_packets[1m])`: 网络发送包速率 ⭐ 新增
- `rate(doris_be_network_receive_packets[1m])`: 网络接收包速率 ⭐ 新增

**内存** (面板 35-36):
- `rate(doris_be_memory_pgpgin[1m])`: 内存页面换入速率 ⭐ 新增
- `doris_be_memory_memtable_consumption`: Memtable 内存消耗 ⭐ 新增

**磁盘** (面板 37-38):
- `doris_be_disks_total_capacity`: 磁盘总容量
- `doris_be_disks_avail_capacity`: 磁盘可用容量
- `(doris_be_disks_total_capacity - doris_be_disks_avail_capacity) / doris_be_disks_total_capacity * 100`: 磁盘使用率 ⭐ 新增

**4. 关键监控指标说明**

#### 需要重点关注的指标

| 指标 | 正常值 | 警告值 | 危险值 | 说明 |
|-----|--------|--------|--------|------|
| 查询错误率 | 接近 0 | > 0.01 | > 0.05 | 查询错误率应该接近 0 |
| 连接数 | < 50 | 50-100 | > 100 | 连接数过高可能导致资源耗尽 |
| Edit Log 大小 | < 1GB | 1-5GB | > 5GB | 过大会影响 FE 启动速度 |
| 线程池队列 | 0 | > 0 | > 100 | 队列积压表示线程池不足 |
| Cache 命中率 | > 80% | 60-80% | < 60% | 命中率低会影响查询性能 |
| Compaction Score | < 50 | 50-100 | > 100 | Score > 100 表示 Compaction 积压严重 |
| 磁盘使用率 | < 70% | 70-80% | > 80% | 超过 80% 需要扩容或清理数据 |

**5. 告警建议**

#### 建议配置的告警规则

**查询错误率告警**:
```yaml
- alert: DorisHighQueryErrorRate
  expr: doris_fe_query_err_rate > 0.01
  for: 5m
  annotations:
    summary: "Doris 查询错误率过高"
    description: "查询错误率 {{ $value }}% 超过 1%"
```

**连接数告警**:
```yaml
- alert: DorisHighConnectionCount
  expr: sum(doris_fe_connection_total) > 100
  for: 5m
  annotations:
    summary: "Doris 连接数过高"
    description: "当前连接数 {{ $value }} 超过 100"
```

**Compaction Score 告警**:
```yaml
- alert: DorisHighCompactionScore
  expr: doris_be_compaction_max_score > 100
  for: 10m
  annotations:
    summary: "Doris Compaction Score 过高"
    description: "Compaction Score {{ $value }} 超过 100"
```

**磁盘使用率告警**:
```yaml
- alert: DorisHighDiskUsage
  expr: (doris_be_disks_total_capacity - doris_be_disks_avail_capacity) / doris_be_disks_total_capacity * 100 > 80
  for: 5m
  annotations:
    summary: "Doris 磁盘使用率过高"
    description: "磁盘使用率 {{ $value }}% 超过 80%"
```

**7. 性能优化建议**

#### 基于监控指标的优化方向

**查询性能优化**:
- 现象: 查询延迟高、QPS 低
- 优化方向:
  - 优化 SQL 查询语句
  - 添加合适的索引
  - 调整分区和分桶策略
  - 增加 BE 节点

**连接管理优化**:
- 现象: 连接数过高
- 优化方向:
  - 使用连接池
  - 及时关闭不用的连接
  - 调整连接超时时间
  - 增加 FE 节点

**Cache 优化**:
- 现象: Cache 命中率低
- 优化方向:
  - 增加 Cache 大小
  - 优化查询模式
  - 预热热点数据

**Compaction 优化**:
- 现象: Compaction Score 高
- 优化方向:
  - 增加 Compaction 线程数
  - 调整 Compaction 策略
  - 优化数据导入频率
  - 增加磁盘 I/O 性能

**内存优化**:
- 现象: 内存使用率高、频繁 GC
- 优化方向:
  - 增加 JVM 堆内存
  - 优化查询，减少内存使用
  - 调整 GC 参数
  - 增加物理内存

**磁盘优化**:
- 现象: 磁盘使用率高
- 优化方向:
  - 清理过期数据
  - 调整数据保留策略
  - 增加磁盘容量
  - 使用数据压缩

**技术要点**:

1. **指标选择原则**:
   - ✅ 能反映系统健康状态
   - ✅ 能帮助定位性能瓶颈
   - ✅ 能预警潜在问题
   - ✅ 有明确的优化方向

2. **面板设计原则**:
   - 相关指标放在一起
   - 重要指标放在上方
   - 使用合适的可视化类型（时序图、仪表盘、统计值）

3. **时间聚合**:
   - 使用 `rate()` 函数计算每秒的变化率
   - 使用 `avg_over_time()` 函数计算时间窗口内的平均值

4. **标签过滤**:
   - 按类型过滤: `doris_fe_thread_pool{type="task_num"}`
   - 按状态过滤: `node_info{type="be_node_num",state="alive"}`
